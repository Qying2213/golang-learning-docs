## 1. Go 基础（Q1-Q20）

### Q1：Go 参数传递是值传递还是引用传递？
**通俗理解：**
- Go 像“复印一份再传给函数”。
- 有些复印件里写着“底层数据地址”，所以看起来像引用。
**详细回答：**
1. Go 语义上只有值传递，没有语言级“引用传递”。
2. `slice/map/channel/interface` 这些值内部含指针，拷贝后可共享底层数据。
3. 因此要区分“变量被拷贝”与“底层数据是否共享”。
4. 如果不希望共享，必须显式复制（如 `copy`）。
**面试加分点：**
- 一句话总结：永远值拷贝，区别在拷贝内容里是否有指针。
**常见追问：**
- 为什么切片传参后修改元素会影响外部？
**追问详细回答：**
1. 切片传参会拷贝切片头（`ptr/len/cap`）。
2. `ptr` 指向同一底层数组，改元素就是改同一块内存。
3. 但如果 `append` 触发扩容，后续修改就可能只影响新数组。

### Q2：`new` 和 `make` 有什么区别？
**通俗理解：**
- `new` 是给你一块空地。
- `make` 是直接把厨房和水电都装好，你能立刻做饭。
**详细回答：**
1. `new(T)`：分配零值内存，返回 `*T`。
2. `make(T, ...)`：只用于 `slice/map/channel`，返回初始化后的 `T`。
3. `map/slice/channel` 通常应使用 `make`，因为需要运行时结构初始化。
4. 用 `new(map[K]V)` 得到的是 `*map[K]V`，底层 map 仍可能是 nil。
**面试加分点：**
- 强调 `make` 是“初始化运行时对象”，不只是分配内存。
**常见追问：**
- `var m map[string]int` 和 `make(map[string]int)` 的差别？
**追问详细回答：**
1. `var m ...` 是 nil map，可读不可写。
2. 写 nil map 会 panic。
3. `make` 后可直接写入，适合实际业务使用。

### Q3：数组和切片的核心区别？
**通俗理解：**
- 数组像固定长度的柜子。
- 切片像柜子的一段“可滑动窗口”。
**详细回答：**
1. 数组长度是类型的一部分，`[3]int` 和 `[4]int` 是不同类型。
2. 切片是描述符（`ptr/len/cap`），不是实际数据本体。
3. 切片赋值/传参只拷贝描述符，可能共享底层数组。
4. 切片截断不一定释放底层大数组，可能造成内存长时间占用。
**面试加分点：**
- 知道“切小片引用大片”是常见内存坑。
**常见追问：**
- 如何复制切片避免共享底层数组？
**追问详细回答：**
1. `dst := make([]T, len(src)); copy(dst, src)`。
2. 或 `dst := append([]T(nil), src...)`。
3. 这样 `dst` 与 `src` 的底层数组完全解耦。

### Q4：`append` 扩容机制怎么回答？
**通俗理解：**
- 容量够就“原地加桌子”，不够就“搬新房子”。
**详细回答：**
1. 当 `len+新增 > cap` 时触发扩容并分配新底层数组。
2. 扩容后会把旧数据拷贝到新数组。
3. 扩容比例不是固定 2 倍，会受容量区间与元素大小影响。
4. 扩容后新旧切片可能不再共享底层数组。
**面试加分点：**
- 提前预估容量 `make([]T, 0, n)` 可减少扩容抖动。
**常见追问：**
- 为什么函数里 append 后外部有时变有时不变？
**追问详细回答：**
1. 不扩容时，底层数组共享，外部可能看到变化。
2. 扩容后，函数里拿到新数组，外部旧切片看不到后续变化。
3. 判断关键点是原切片 `cap` 是否足够。

### Q5：`defer` 的执行顺序和参数求值时机？
**通俗理解：**
- `defer` 是“离开房间前一定要做的收尾动作”。
**详细回答：**
1. defer 在函数返回前执行，顺序是 LIFO（后进先出）。
2. defer 的参数在“声明 defer 时”就求值。
3. 适合资源释放、解锁、计时埋点。
4. 循环中大量 defer 需要评估性能开销。
**面试加分点：**
- 能说出“参数提前求值”这个细节通常会加分。
**常见追问：**
- `defer` + 命名返回值会怎样？
**追问详细回答：**
1. 命名返回值是函数内变量。
2. defer 在 return 语句之后、真正返回之前执行。
3. 所以 defer 可以修改命名返回值并影响最终结果。

### Q6：`panic`、`recover`、`error` 如何分工？
**通俗理解：**
- `error` 是可预期问题，`panic` 是紧急事故，`recover` 是安全气囊。
**详细回答：**
1. 业务分支错误优先返回 `error`。
2. panic 用于不可恢复或严重编程错误。
3. recover 只能在 defer 中生效，常用于服务边界兜底。
4. panic 不该当作常规流程控制手段。
**面试加分点：**
- 结合中间件：recover 后打日志、trace、请求参数、统一响应。
**常见追问：**
- 什么时候 panic 是合理的？
**追问详细回答：**
1. 核心不变量被破坏时（逻辑错误）。
2. 启动关键依赖失败且程序无法继续时。
3. 普通业务校验失败不应 panic。

### Q7：接口 `nil` 陷阱是什么？
**通俗理解：**
- 接口是“类型 + 值”的盒子，只要类型不空，盒子就不算空。
**详细回答：**
1. 接口底层是 `(type, data)`。
2. 只有 `type=nil && data=nil` 才等于 nil。
3. `var p *T=nil; var i any=p` 时，`i != nil`。
4. 这个坑在返回 `error` 时非常常见。
**面试加分点：**
- 说明你知道“有类型 nil”会误导上层错误判断。
**常见追问：**
- 如何避免这个坑？
**追问详细回答：**
1. 返回 error 时确保真正返回 `nil`。
2. 不要把 `(*MyErr)(nil)` 直接装箱成 `error` 返回。
3. 使用 `errors.Is/As` 做语义判断。

### Q8：类型断言和类型转换有什么区别？
**通俗理解：**
- 断言是“猜盒子里真实类型”；转换是“把一种兼容类型转成另一种”。
**详细回答：**
1. 类型断言用于接口取动态类型：`v, ok := x.(T)`。
2. 类型转换用于兼容类型间显式转化：`T(v)`。
3. 断言失败会 panic，建议用 `ok` 防护。
4. 大量分支可用 type switch 增强可读性。
**面试加分点：**
- 断言是运行时行为，转换更多是编译期可检查语义。
**常见追问：**
- `[]byte` 和 `string` 转换会拷贝吗？
**追问详细回答：**
1. 常规转换会拷贝。
2. 这是为了保持 string 不可变语义。
3. `unsafe` 可绕过拷贝，但工程风险高，慎用。

### Q9：`iota` 常见用法？
**通俗理解：**
- `iota` 是常量的自动编号器。
**详细回答：**
1. 在 const 块内按行自增。
2. 常用来定义状态枚举、权限位（`1 << iota`）。
3. 每个 const 块独立计数。
4. 可配合 `String()` 增强日志可读性。
**面试加分点：**
- 讲到位运算枚举（权限系统）会更像实战。
**常见追问：**
- 如何避免魔法数字？
**追问详细回答：**
1. 用具名常量替代裸数字。
2. 给常量补语义注释。
3. 阈值类数字尽量走配置并标注单位。

### Q10：Go 为什么提倡组合而非继承？
**通俗理解：**
- 少层级、多组合，结构更清晰。
**详细回答：**
1. Go 语言设计强调简洁和可读性。
2. 通过组合和接口实现复用与多态。
3. 减少复杂继承树导致的高耦合。
4. 更利于测试替换与服务化演进。
**面试加分点：**
- 接口建议由“消费方”定义，而非生产方先定义大接口。
**常见追问：**
- 什么时候该用接口，什么时候不用？
**追问详细回答：**
1. 多实现/可替换/需解耦时用接口。
2. 单实现、变化小场景直接用具体类型更简单。
3. 不要为了“看起来高级”而滥用接口。

### Q11：值接收者和指针接收者如何选择？
**通俗理解：**
- 小对象不改值可“拷贝传”，大对象或要改值就“传地址”。
**详细回答：**
1. 需要修改对象状态时必须用指针接收者。
2. 对象大、拷贝成本高时优先指针接收者。
3. 方法集规则会影响接口实现。
4. 同一类型建议保持接收者风格一致。
**面试加分点：**
- 能讲清方法集和接口实现关系。
**常见追问：**
- 方法集与接口实现关系是？
**追问详细回答：**
1. `T` 的方法集只含值接收者方法。
2. `*T` 的方法集含值+指针接收者方法。
3. 若接口依赖指针方法，则只有 `*T` 实现它。

### Q12：map 并发安全吗？
**通俗理解：**
- 多人同时改一本账本，不加锁很容易写乱。
**详细回答：**
1. 原生 map 并发读写不安全。
2. 并发写会触发 runtime 错误。
3. 方案：map+锁、分片锁、sync.Map。
4. 选型要看读写比例和操作复杂度。
**面试加分点：**
- `sync.Map` 不等于“性能更高”，要看场景。
**常见追问：**
- `sync.Map` 的适用边界？
**追问详细回答：**
1. 读多写少、key 稳定场景更适合。
2. 写多删多时常不如 map+Mutex。
3. 复合逻辑仍需外层同步保证一致性。

### Q13：`init()` 的执行规则和注意事项？
**通俗理解：**
- `init` 是程序开机自检脚本，别在里面干太重的活。
**详细回答：**
1. 按依赖顺序初始化包。
2. 包内先初始化变量，再执行 init。
3. 不要在 init 做重 I/O、网络依赖、隐式副作用。
4. 关键初始化尽量显式化，便于测试。
**面试加分点：**
- 强调“可测试性优先”，别把逻辑藏在 init。
**常见追问：**
- 多个包 init 顺序如何确定？
**追问详细回答：**
1. 先被依赖包，后依赖它的包。
2. 顺序由 import 依赖图决定。
3. 不建议依赖文件名级别顺序细节。

### Q14：Go 错误处理的最佳实践？
**通俗理解：**
- 错误要“层层带上下文”传上去，而不是丢失细节。
**详细回答：**
1. 使用 `%w` 包装错误保留链路。
2. 用 `errors.Is/As` 判断语义，而非字符串包含。
3. 按层增加上下文（哪个模块、哪个参数）。
4. 业务层区分可重试与不可重试错误。
**面试加分点：**
- 错误处理要服务于“可观测性和可恢复性”。
**常见追问：**
- 什么时候要自定义错误类型？
**追问详细回答：**
1. 需要结构化字段（错误码、状态、对象ID）时。
2. 上层要按类型分支处理时。
3. 只是传字符串通常没必要。

### Q15：泛型适合哪些场景？
**通俗理解：**
- 代码重复多、逻辑同构时，泛型能省很多模板代码。
**详细回答：**
1. 适合容器、集合、通用算法、工具库。
2. 不适合把业务逻辑强行泛型化。
3. 约束要最小、清晰，避免类型体操。
4. 选择标准是可读性和复用收益。
**面试加分点：**
- 提到“先有重复再抽泛型”会很务实。
**常见追问：**
- 泛型会影响性能吗？
**追问详细回答：**
1. 大多数业务场景影响可忽略。
2. 是否有差异应以 benchmark 结果为准。
3. 不要为微小性能牺牲可读性。

### Q16：空结构体 `struct{}` 的用途？
**通俗理解：**
- 表示“只有信号，没有内容”。
**详细回答：**
1. 占用零字节（逻辑上最小表示）。
2. 常用于 `map[T]struct{}` 当集合。
3. 常用于 `chan struct{}` 表示通知信号。
4. 语义清晰：只关心“有/无”。
**面试加分点：**
- 与 `bool` 相比，struct{} 更明确“无载荷语义”。
**常见追问：**
- 为什么不用 `bool`？
**追问详细回答：**
1. `bool` 仍有额外数据语义。
2. `struct{}` 更准确表示“只要存在，不关心值”。
3. 在大集合里也更省空间。

### Q17：浅拷贝和深拷贝在 Go 里怎么理解？
**通俗理解：**
- 默认拷贝“外壳”，外壳里指向的内容可能还是共享。
**详细回答：**
1. Go 赋值是按值拷贝。
2. 若字段里有 slice/map/pointer，底层数据可能仍共享。
3. 深拷贝要手动拷贝每层引用字段。
4. 深拷贝缺失是并发 bug 常见根因。
**面试加分点：**
- 大对象建议提供显式 `Clone()` 方法。
**常见追问：**
- 为什么 map 赋值后两个变量互相影响？
**追问详细回答：**
1. map 变量拷贝的是运行时描述符。
2. 两个变量指向同一底层哈希表。
3. 任一方写入都会被另一方看到。

### Q18：Go 的 `==` 比较有哪些限制？
**通俗理解：**
- 不是所有类型都能直接比，像 map/slice 这种“动态容器”不能直接比内容。
**详细回答：**
1. 可比较类型才能用 `==`。
2. `slice/map/function` 不能直接比较（仅可与 nil 比）。
3. 结构体是否可比较取决于字段是否全可比较。
4. 内容比较需自定义逻辑或辅助库。
**面试加分点：**
- Go 1.21+ 可提 `slices.Equal`。
**常见追问：**
- 两个切片怎么比较内容是否相等？
**追问详细回答：**
1. 先比较长度。
2. 再逐元素比较。
3. 或用 `slices.Equal`（新版本标准库包）。

### Q19：`string` 不可变有何意义？
**通俗理解：**
- 不可变让共享更安全，省去很多并发修改风险。
**详细回答：**
1. string 不可变，避免被意外篡改。
2. 并发读取更安全，心智模型更简单。
3. 频繁拼接会产生新分配。
4. 大量拼接建议使用 `strings.Builder`。
**面试加分点：**
- 能说出 Builder 和 Buffer 适用差异。
**常见追问：**
- `strings.Builder` 和 `bytes.Buffer` 怎么选？
**追问详细回答：**
1. 纯文本拼接优先 Builder。
2. 二进制/字节流处理更适合 Buffer。
3. 两者都可结合池化减少分配。

### Q20：Go Modules 依赖治理怎么做？
**通俗理解：**
- 依赖版本要“有账本、有校验、可回滚”。
**详细回答：**
1. `go.mod` 管理版本，`go.sum` 做完整性校验。
2. `go mod tidy` 清理无用依赖并补齐缺失依赖。
3. 升级依赖要配回归测试和变更审查。
4. 生产建议配私有代理和漏洞扫描。
**面试加分点：**
- 强调供应链安全（依赖可信与可审计）。
**常见追问：**
- `go mod tidy` 的真实作用？
**追问详细回答：**
1. 与源码对齐依赖关系。
2. 删除无用依赖，补全必要依赖。
3. 让 `go.mod/go.sum` 保持干净一致。

## 2. 并发与 Runtime（Q21-Q40）

### Q21：GMP 调度模型是什么？
**通俗理解：**
- G 是任务，M 是工人，P 是工位；工人占到工位才能干活。
**详细回答：**
1. G（goroutine）是用户态协程。
2. M（machine）是 OS 线程。
3. P（processor）持有本地队列与调度上下文。
4. 调度器通过 work stealing 提升多核利用率。
**面试加分点：**
- 能提到 syscall 阻塞时 P 的让渡机制。
**常见追问：**
- `GOMAXPROCS` 控制什么？
**追问详细回答：**
1. 控制 P 的数量上限。
2. 常理解为同时并行执行 Go 代码的上限。
3. 不等于总线程数。

### Q22：如何识别和治理 goroutine 泄漏？
**通俗理解：**
- 协程像开出来的工单，没关闭就越积越多。
**详细回答：**
1. 观察 goroutine 数是否持续增长且不回落。
2. 常见根因：channel 永久阻塞、无退出条件、下游卡死。
3. 方案：context 取消、超时、关闭协议。
4. 线上结合 `pprof/goroutine` 分析阻塞栈。
**面试加分点：**
- 强调“每个 goroutine 必须有退出路径”。
**常见追问：**
- 你项目里如何预防泄漏？
**追问详细回答：**
1. 所有后台协程都接收 ctx。
2. 对 channel 操作加超时分支。
3. 服务退出时统一取消并等待回收。

### Q23：无缓冲和有缓冲 channel 区别？
**通俗理解：**
- 无缓冲像面对面交接；有缓冲像先放到临时货架。
**详细回答：**
1. 无缓冲 send/recv 必须同步配对。
2. 有缓冲在容量内发送可不阻塞。
3. 缓冲可平滑瞬时峰值，但不能替代治理。
4. 缓冲过大可能掩盖消费慢问题。
**面试加分点：**
- 用“背压”概念解释队列长度。
**常见追问：**
- 缓冲是不是越大越好？
**追问详细回答：**
1. 不是，过大可能导致延迟增大、内存占用高。
2. 故障时积压恢复更慢。
3. 容量应根据吞吐与 SLA 压测确定。

### Q24：channel 关闭原则怎么讲？
**通俗理解：**
- 谁生产消息，谁宣布“停止生产”。
**详细回答：**
1. 一般由发送方关闭 channel。
2. 多发送方需引入统一关闭协调者。
3. 对已关闭 channel 再 close 会 panic。
4. 接收方用 `v,ok := <-ch` 判断结束。
**面试加分点：**
- close 是“协议信号”，不是资源释放 API。
**常见追问：**
- 如何判断 channel 是否关闭？
**追问详细回答：**
1. 只有通过接收返回的 `ok` 判断。
2. 没有并发安全的通用 `isClosed`。
3. 设计上应靠协议而不是探测。

### Q25：`select` 的行为特点？
**通俗理解：**
- 像多个电话同时响，系统随机先接一个。
**详细回答：**
1. 多 case 同时就绪时随机选择。
2. default 可做非阻塞收发。
3. 常用于超时、取消、多路复用。
4. 滥用 default 会导致忙等。
**面试加分点：**
- 建议配 `time.Ticker` 或 sleep 防空转。
**常见追问：**
- select 能实现优先级吗？
**追问详细回答：**
1. 原生 select 无优先级语义。
2. 可分层 select 人工实现优先级。
3. 或改用优先队列模型。

### Q26：`Mutex` 与 `RWMutex` 怎么选？
**通俗理解：**
- 读多写少像图书馆，可多人读；写多则单锁更简单高效。
**详细回答：**
1. 读多写少、读临界区长时 RWMutex 可能更优。
2. 写多或临界区短时 Mutex 常更快。
3. 锁选择应基于 benchmark。
4. 正确性优先于“看起来高性能”。
**面试加分点：**
- 能说“先 Mutex，再按瓶颈升级”。
**常见追问：**
- 锁竞争如何定位？
**追问详细回答：**
1. 开启 mutex/block profile。
2. 找等待最长热点锁。
3. 缩小临界区、减少共享状态。

### Q27：`sync.Once` 的适用边界？
**通俗理解：**
- 只执行一次的开关，适合初始化，不适合失败重试逻辑。
**详细回答：**
1. Once 保证函数只执行一次且线程安全。
2. 适合单例初始化、配置加载。
3. 对“可能失败且需重试”的场景不友好。
4. 失败重试要单独设计状态机。
**面试加分点：**
- 可提 `sync.OnceValue`（新版本简化只读初始化）。
**常见追问：**
- 初始化失败怎么办？
**追问详细回答：**
1. 返回 `(obj, err)` 并显式缓存状态。
2. 允许重试或降级方案。
3. 不把可失败流程硬套 Once。

### Q28：`WaitGroup` 常见坑有哪些？
**通俗理解：**
- 记账器加减不平就会“永远等不到归零”。
**详细回答：**
1. `Add` 与 `Done` 不匹配会死等或 panic。
2. 必须在启动 goroutine 前 Add。
3. WaitGroup 不可复制。
4. 复杂任务组合推荐 errgroup。
**面试加分点：**
- 提到“先 Add 后启动”是核心细节。
**常见追问：**
- 如何优雅等待多任务并返回错误？
**追问详细回答：**
1. 用 `errgroup.WithContext`。
2. 任一任务失败可取消整体。
3. Wait 返回统一错误，便于上层处理。

### Q29：context 正确使用原则？
**通俗理解：**
- context 是“请求生命周期遥控器”。
**详细回答：**
1. `ctx` 作为函数第一个参数传递。
2. 用于取消、超时、请求级元数据。
3. 不存业务大对象、不放进结构体长期持有。
4. 所有外部调用建议绑定超时。
**面试加分点：**
- 强调“取消信号必须被下游真正消费”。
**常见追问：**
- `Background()` 和 `TODO()` 区别？
**追问详细回答：**
1. 功能上都可做根 context。
2. Background 表示明确根上下文。
3. TODO 表示暂未确定，后续应替换。

### Q30：`WithCancel`、`WithTimeout`、`WithDeadline` 区别？
**通俗理解：**
- 一个手动停，一个倒计时停，一个定点时间停。
**详细回答：**
1. WithCancel 手动取消。
2. WithTimeout 相对时间到期取消。
3. WithDeadline 绝对时间到期取消。
4. 三者都需要调用 cancel 释放资源。
**面试加分点：**
- 错误判断用 `errors.Is(err, context.DeadlineExceeded)`。
**常见追问：**
- 为什么超时了 goroutine 还没退出？
**追问详细回答：**
1. 代码未监听 `ctx.Done()`。
2. 阻塞在不支持 context 的调用。
3. 锁或通道阻塞缺少退出分支。

### Q31：Go 内存模型中 happens-before 有什么价值？
**通俗理解：**
- 它是并发世界里的“先后秩序法律”。
**详细回答：**
1. 定义写入对读取的可见性。
2. 无同步关系就不能假设可见。
3. `unlock->lock`、`send->receive` 都建立 happens-before。
4. 数据竞争意味着行为不可预测。
**面试加分点：**
- 强调“能跑不等于正确”。
**常见追问：**
- `time.Sleep` 能替代同步吗？
**追问详细回答：**
1. 不能，Sleep 只延时。
2. 不提供可见性和有序性保证。
3. 需使用锁/channel/atomic。

### Q32：什么是逃逸分析？
**通俗理解：**
- 编译器决定变量住“栈”还是“堆”。
**详细回答：**
1. 变量生命周期不超栈帧可放栈。
2. 超出生命周期会逃逸到堆。
3. 堆对象增多会提升 GC 负担。
4. 常见触发：返回指针、闭包捕获、接口装箱。
**面试加分点：**
- 会用 `go build -gcflags="-m"` 看逃逸。
**常见追问：**
- 逃逸一定是坏事吗？
**追问详细回答：**
1. 不是，正确性优先。
2. 只是会带来分配和 GC 成本。
3. 要基于热点路径优化，不盲目追零逃逸。

### Q33：Go GC 三色标记法怎么讲？
**通俗理解：**
- 白色未访问、灰色待处理、黑色已确认存活。
**详细回答：**
1. 从根对象开始标记可达对象。
2. 灰色对象扫描完成后变黑。
3. 最终白色对象作为垃圾回收。
4. Go 使用并发标记清扫减少长停顿。
**面试加分点：**
- 提到写屏障用于并发标记正确性。
**常见追问：**
- 为什么还会有 STW？
**追问详细回答：**
1. 某些关键阶段需要短暂停顿保证一致性。
2. 如初始标记、终止标记等。
3. 目标是“短且可控”，不是绝对零停顿。

### Q34：`GOGC` 和 `GOMEMLIMIT` 如何调优？
**通俗理解：**
- GOGC 像“垃圾车出动频率”，GOMEMLIMIT 像“仓库总容量红线”。
**详细回答：**
1. GOGC 控制堆增长比例触发 GC。
2. GOMEMLIMIT 设内存软上限，runtime 会更积极回收。
3. 调优目标是吞吐、延迟、内存占用平衡。
4. 必须结合业务负载与监控数据。
**面试加分点：**
- 线上调优要看 GC pause、heap、GC CPU 占比。
**常见追问：**
- 你重点看哪些 GC 指标？
**追问详细回答：**
1. `heap_alloc/heap_inuse`。
2. GC pause 的 P95/P99。
3. GC 次数与 GC CPU 比例。

### Q35：`sync.Pool` 适合什么场景？
**通俗理解：**
- 临时对象回收箱，减少重复创建成本。
**详细回答：**
1. 适合短生命周期且高频分配对象。
2. GC 时 pool 可能被清空，不保证命中。
3. 不适合连接池等强生命周期资源。
4. 典型用途是 bytes.Buffer 复用。
**面试加分点：**
- 说明“池化也有成本，不是越多越好”。
**常见追问：**
- 什么时候用 Pool 反而变慢？
**追问详细回答：**
1. 命中率低场景收益小。
2. 对象过大导致内存压力上升。
3. 低并发时池管理开销可能超过收益。

### Q36：`go test -race` 有什么价值？
**通俗理解：**
- 像并发代码的“夜视仪”，能照出看不见的竞态问题。
**详细回答：**
1. 在运行时检测共享内存并发读写冲突。
2. 可提前发现 data race 风险。
3. 特别适合并发库和核心逻辑测试。
4. race 模式有性能开销，主要用于测试环境。
**面试加分点：**
- 说明 race 不是万能，需要覆盖足够测试路径。
**常见追问：**
- race 通过是否代表绝对安全？
**追问详细回答：**
1. 不代表绝对安全。
2. 未覆盖路径仍可能有竞态。
3. 需结合代码审查与压力测试。

### Q37：atomic 和 mutex 怎么选？
**通俗理解：**
- 原子像“单步操作”，锁像“临界区围栏”。
**详细回答：**
1. 原子适合简单计数、状态位更新。
2. 多字段一致性和复杂逻辑优先锁。
3. 原子代码可读性差，维护成本可能更高。
4. 先用锁保证正确，再看是否原子化优化。
**面试加分点：**
- 提到 CAS 在高竞争下会自旋重试。
**常见追问：**
- CAS 常见问题？
**追问详细回答：**
1. 高冲突下自旋耗 CPU。
2. 可能出现 ABA 问题。
3. 需退避策略和版本戳等补偿机制。

### Q38：如何用 pprof 定位性能问题？
**通俗理解：**
- 先找最耗资源的“主犯”，再看调用链根因。
**详细回答：**
1. CPU 问题看 profile。
2. 内存问题看 heap/allocs。
3. 锁等待看 mutex/block。
4. 固定窗口前后对比优化结果。
**面试加分点：**
- 强调“先测量后优化”的工程方法。
**常见追问：**
- 线上如何安全开 pprof？
**追问详细回答：**
1. 仅内网开放或加鉴权。
2. 限时开启，排查后关闭。
3. 避免公网暴露调试端点。

### Q39：Worker Pool 怎么设计更稳？
**通俗理解：**
- 固定工人 + 排队任务，避免系统被瞬时流量冲垮。
**详细回答：**
1. 固定 worker 数控制并发上限。
2. 有界队列做背压。
3. 任务执行必须有超时和取消。
4. 退出时先停接新任务，再清空在途任务。
**面试加分点：**
- 配监控：队列长度、处理耗时、失败率。
**常见追问：**
- worker 数量怎么确定？
**追问详细回答：**
1. CPU 密集按核数附近配置。
2. I/O 密集可放大但要压测。
3. 以 SLA 和资源指标最终定型。

### Q40：服务优雅退出如何实现？
**通俗理解：**
- 不是“立刻断电”，而是“停止接客，把在店顾客服务完再关门”。
**详细回答：**
1. 捕获 `SIGTERM/SIGINT`。
2. 先摘流量，停止新请求进入。
3. 等待在途请求完成，设置最大等待时间。
4. 关闭数据库、MQ、后台协程等资源。
**面试加分点：**
- K8s 中要配合 readiness、preStop、terminationGracePeriod。
**常见追问：**
- 为什么优雅退出仍会丢请求？
**追问详细回答：**
1. 摘流太晚或探针配置不合理。
2. 终止宽限期太短被强杀。
3. 下游调用无超时导致退出阻塞。

## 3. MySQL（Q41-Q55）

### Q41：InnoDB 为什么是 MySQL 默认引擎？
**通俗理解：**
- 既能保证事务安全，又能扛高并发，是“综合能力最均衡”的引擎。
**详细回答：**
1. 支持事务、行级锁、崩溃恢复。
2. MVCC 提升并发读性能。
3. 适合互联网常见 OLTP 场景。
4. 配套生态和运维经验成熟。
**面试加分点：**
- 可补 `Buffer Pool + Redo/Undo` 架构理解。
**常见追问：**
- MyISAM 为什么逐渐被淘汰？
**追问详细回答：**
1. 不支持事务与行级锁。
2. 崩溃恢复能力弱。
3. 高并发写场景稳定性较差。

### Q42：聚簇索引和二级索引区别？
**通俗理解：**
- 聚簇索引像“正文书页”；二级索引像“目录”。
**详细回答：**
1. 聚簇索引叶子节点存整行数据。
2. 二级索引叶子节点存主键值。
3. 二级索引查整行通常需要回表。
4. 覆盖索引可避免回表。
**面试加分点：**
- 主键建议短、稳定、递增，减少页分裂。
**常见追问：**
- 最左前缀原则是什么？
**追问详细回答：**
1. 联合索引按从左到右连续匹配。
2. `(a,b,c)` 可利用 `a`、`a,b`、`a,b,c`。
3. 跳过 `a` 直接查 `b` 常难有效命中。

### Q43：ACID 应该怎么讲？
**通俗理解：**
- 事务要么全成要么全败，且结果可靠不乱。
**详细回答：**
1. A 原子性：事务内操作不可分割。
2. C 一致性：事务前后数据满足约束。
3. I 隔离性：并发事务互不干扰到规定程度。
4. D 持久性：提交后数据不丢。
**面试加分点：**
- 强调“一致性是目标，A/I/D 是手段”。
**常见追问：**
- 强一致和最终一致如何选？
**追问详细回答：**
1. 资金、库存等强约束链路优先强一致。
2. 通知、推荐可采用最终一致。
3. 依据业务风险成本做权衡。

### Q44：隔离级别和幻读怎么回答？
**通俗理解：**
- 隔离级别是“并发干扰容忍度”的分档标准。
**详细回答：**
1. 级别有 RU、RC、RR、Serializable。
2. InnoDB 默认 RR。
3. RR 下结合 MVCC 与 Next-Key Lock 抑制幻读。
4. 当前读与快照读需要区分。
**面试加分点：**
- 能讲清 `for update` 是当前读。
**常见追问：**
- `SELECT ... FOR UPDATE` 属于哪类读？
**追问详细回答：**
1. 属于当前读。
2. 会读取最新版本并加锁。
3. 与普通快照读语义不同。

### Q45：Redo Log 和 Undo Log 作用？
**通俗理解：**
- Redo 像“重做清单”，Undo 像“撤销清单”。
**详细回答：**
1. Redo 保证崩溃恢复与持久性。
2. Undo 支持事务回滚和 MVCC 历史版本。
3. 两者共同支撑事务正确性和并发性能。
4. Checkpoint 协调日志与数据页落盘进度。
**面试加分点：**
- 可对比 binlog 所在层次和用途。
**常见追问：**
- Binlog 与 Redo Log 区别？
**追问详细回答：**
1. Redo 是 InnoDB 物理日志。
2. Binlog 是 Server 层逻辑日志。
3. 前者偏恢复，后者偏复制与审计。

### Q46：Binlog 格式怎么选？
**通俗理解：**
- 记录“SQL 语句”还是“每行变更”，是准确性与体积权衡。
**详细回答：**
1. STATEMENT 体积小但可能不确定。
2. ROW 最准确，适合复制和 CDC。
3. MIXED 按语句特征自动选择。
4. 生产常优先 ROW 保证一致性。
**面试加分点：**
- 指出非确定性函数在 STATEMENT 下风险高。
**常见追问：**
- 为什么某些 SQL 不适合 STATEMENT？
**追问详细回答：**
1. `now()`、`rand()` 等会导致重放不一致。
2. 依赖环境状态语句可重复性差。
3. 用 ROW 更稳妥。

### Q47：慢 SQL 排查流程是什么？
**通俗理解：**
- 先找最慢路口，再看堵车原因，最后改路和信号灯。
**详细回答：**
1. 慢日志定位高频慢 SQL。
2. EXPLAIN 分析扫描行数与索引命中。
3. 优化 SQL、索引、表结构。
4. 压测验证并灰度上线。
**面试加分点：**
- 强调“先定位后优化，不盲改”。
**常见追问：**
- `Using filesort` 一定不好吗？
**追问详细回答：**
1. 不一定，小数据量可接受。
2. 关键看是否落盘和总体耗时。
3. 要结合实际 workload 判断。

### Q48：索引失效常见原因有哪些？
**通俗理解：**
- 你建了目录，但查询写法让目录用不上。
**详细回答：**
1. 对索引列做函数或隐式类型转换。
2. 不满足最左前缀。
3. 低选择性字段优化器可能放弃索引。
4. 统计信息过旧会误导优化器。
**面试加分点：**
- 能主动提“ANALYZE 更新统计信息”。
**常见追问：**
- 为什么加了索引反而更慢？
**追问详细回答：**
1. 写入维护索引成本上升。
2. 错误索引让执行计划更差。
3. 索引太多影响缓存和存储效率。

### Q49：回表与覆盖索引如何平衡？
**通俗理解：**
- 目录够用就别翻正文，翻得越少越快。
**详细回答：**
1. 二级索引查整行通常要回表。
2. 覆盖索引可直接返回结果，减少随机 I/O。
3. 但索引太宽会增加写成本。
4. 覆盖应针对高频核心查询做。
**面试加分点：**
- 能说“读写平衡”而不是一味追求覆盖。
**常见追问：**
- 你如何做读写权衡？
**追问详细回答：**
1. 先保障高频慢查询。
2. 控制索引数量，定期淘汰低价值索引。
3. 结合写入 TPS 和慢查询数据持续调整。

### Q50：MySQL 锁机制怎么讲？
**通俗理解：**
- 锁就是并发场景下的交通规则，防止互相撞车。
**详细回答：**
1. 常见有行锁、间隙锁、临键锁、意向锁。
2. InnoDB 行锁加在索引上，不是物理行号。
3. 不走索引会导致锁范围扩大。
4. 死锁不可完全避免，需重试机制。
**面试加分点：**
- 强调“事务尽量短，锁粒度尽量小”。
**常见追问：**
- 如何降低死锁概率？
**追问详细回答：**
1. 统一加锁顺序。
2. 缩短事务、避免长事务。
3. 让 SQL 命中索引减少锁范围。

### Q51：主从延迟怎么治理？
**通俗理解：**
- 主库写完后，从库“抄作业”有延迟。
**详细回答：**
1. 建立复制延迟监控和告警。
2. 关键读场景走主库（读己之写）。
3. 拆大事务、优化慢 SQL、提升从库资源。
4. 一致性敏感场景可使用位点等待机制。
**面试加分点：**
- 能讲“按业务分级一致性策略”。
**常见追问：**
- 业务怎么感知并容忍延迟？
**追问详细回答：**
1. 关键路径临时读主。
2. 非关键路径允许最终一致并做提示。
3. 延迟超阈值自动切换读策略。

### Q52：分库分表什么时候做？
**通俗理解：**
- 单机扛不住时才拆，不是越早拆越好。
**详细回答：**
1. 单库容量、QPS、写入瓶颈逼近上限时考虑。
2. 优先垂直拆分，再水平分片。
3. 分片键设计决定后续扩容成本。
4. 要预留迁移、跨分片查询和事务策略。
**面试加分点：**
- 能主动说“分片不是银弹，复杂度很高”。
**常见追问：**
- 分片键选错怎么迁移？
**追问详细回答：**
1. 新旧分片并行双写。
2. 回填数据并做一致性校验。
3. 灰度切流后再下线旧分片。

### Q53：连接池参数怎么调？
**通俗理解：**
- 连接像服务窗口，太少排队，太多把数据库压垮。
**详细回答：**
1. `MaxOpenConns` 控制最大连接。
2. `MaxIdleConns` 提升复用、减少握手开销。
3. `ConnMaxLifetime` 避免长连接老化问题。
4. 参数需结合 DB 规格与压测结果调优。
**面试加分点：**
- 强调要看等待时间和连接利用率。
**常见追问：**
- 连接池过大或过小会怎样？
**追问详细回答：**
1. 过小导致等待和超时。
2. 过大增加数据库切换成本和资源压力。
3. 目标是稳定利用率与低等待。

### Q54：为什么要避免长事务？
**通俗理解：**
- 长事务像长期占道施工，别人都得绕路或堵住。
**详细回答：**
1. 持锁时间长，冲突加剧。
2. 日志和版本清理被拖慢。
3. 会引发主从延迟放大。
4. 应拆为短事务+异步补偿。
**面试加分点：**
- 能结合具体业务讲拆分策略。
**常见追问：**
- 如何发现长事务？
**追问详细回答：**
1. 监控事务持续时间。
2. 查锁等待与阻塞链。
3. 结合慢日志定位具体 SQL。

### Q55：MySQL 性能优化经验如何高分表达？
**通俗理解：**
- 不是“我加了索引”，而是“问题-动作-结果-验证”闭环。
**详细回答：**
1. 给出优化前指标（QPS、P99、慢 SQL 数）。
2. 描述关键动作（索引、SQL、缓存、连接池）。
3. 给优化后量化结果。
4. 说明如何防回归（监控+压测+灰度）。
**面试加分点：**
- 主动讲副作用和权衡会更成熟。
**常见追问：**
- 如何确认优化有效且可持续？
**追问详细回答：**
1. 固定基准压测前后对比。
2. 上线灰度观察指标回归。
3. 设告警阈值并保留回滚方案。

## 4. Redis（Q56-Q67）

### Q56：Redis 为什么快？
**通俗理解：**
- 内存存取 + 简洁执行路径，所以非常快。
**详细回答：**
1. 基于内存，避免磁盘随机 I/O。
2. 单线程命令执行避免复杂锁竞争。
3. I/O 多路复用提升并发连接处理效率。
4. 数据结构针对典型场景优化。
**面试加分点：**
- Redis 6+ 网络 I/O 可多线程，但命令执行仍主要单线程。
**常见追问：**
- 单线程为何还能高并发？
**追问详细回答：**
1. 命令执行快，单次阻塞短。
2. 内存操作远快于磁盘。
3. 多路复用减少线程切换成本。

### Q57：Redis 数据结构与场景匹配？
**通俗理解：**
- 选结构就像选工具，不同工具解决不同问题。
**详细回答：**
1. String：缓存、计数、分布式锁。
2. Hash：对象字段存储。
3. List：消息队列、时间线。
4. Set/ZSet：去重、标签、排行榜。
**面试加分点：**
- 能补充 HLL、Bitmap、Geo 更加分。
**常见追问：**
- 排行榜为什么常用 ZSet？
**追问详细回答：**
1. 天然按 score 排序。
2. 支持查排名和分段查询。
3. 更新分数效率高，业务适配好。

### Q58：穿透、击穿、雪崩怎么治理？
**通俗理解：**
- 三种问题本质都是“缓存没兜住，压力打到数据库”。
**详细回答：**
1. 穿透：查不存在数据，布隆过滤/空值缓存。
2. 击穿：热点 key 失效，互斥重建/逻辑过期。
3. 雪崩：大量 key 同时失效，过期打散+限流降级。
4. 目标是保护 DB 和核心链路可用。
**面试加分点：**
- 强调治理是“组合拳”，不是单点技巧。
**常见追问：**
- 空值缓存副作用是什么？
**追问详细回答：**
1. 占用额外内存。
2. 可能造成新数据短暂不可见。
3. 需短 TTL + 业务校正策略。

### Q59：RDB 和 AOF 如何选择？
**通俗理解：**
- RDB 像定期快照，AOF 像连续流水账。
**详细回答：**
1. RDB 恢复快、文件小，但丢失窗口较大。
2. AOF 数据更完整，但文件大且恢复慢。
3. 常见生产策略是两者结合。
4. fsync 策略直接影响性能与可靠性。
**面试加分点：**
- 能说 `appendfsync everysec` 是常见折中策略。
**常见追问：**
- AOF 重写何时触发？
**追问详细回答：**
1. 手动 `BGREWRITEAOF`。
2. 或达到自动重写阈值触发。
3. 目的是压缩历史命令、降低恢复成本。

### Q60：过期删除和淘汰策略怎么答？
**通俗理解：**
- 到期不代表立刻删，Redis 会“按策略清理”。
**详细回答：**
1. 惰性删除：访问时发现过期再删。
2. 定期删除：后台随机抽样清理。
3. 内存紧张时触发淘汰策略。
4. 常见淘汰策略有 LRU/LFU 家族。
**面试加分点：**
- 清晰区分“过期删除”与“内存淘汰”。
**常见追问：**
- `allkeys-lru` 和 `volatile-lru` 区别？
**追问详细回答：**
1. allkeys-lru：所有 key 都可淘汰。
2. volatile-lru：仅淘汰设置 TTL 的 key。
3. 若 TTL key 太少，volatile 可能不够用。

### Q61：Redis 分布式锁正确姿势？
**通俗理解：**
- 上锁要有身份证，解锁必须核验身份证。
**详细回答：**
1. 加锁：`SET key val NX PX ttl`。
2. 解锁：Lua 脚本原子校验 val 后删除。
3. 要考虑超时、续期、客户端崩溃。
4. 锁不能替代业务幂等。
**面试加分点：**
- 说明“锁是兜底手段，不是首选”。
**常见追问：**
- Redlock 争议点是什么？
**追问详细回答：**
1. 网络分区和时钟漂移下正确性边界有争议。
2. 极端场景可能出现双持锁风险。
3. 应配业务级幂等与补偿机制。

### Q62：缓存与数据库一致性怎么设计？
**通俗理解：**
- 数据库是真账本，缓存是加速副本，副本错了要能修。
**详细回答：**
1. 常用“先写 DB，再删缓存”。
2. 删缓存失败要重试或异步补偿。
3. 强一致代价高，很多场景采用最终一致。
4. 热点数据可加版本号降低脏读风险。
**面试加分点：**
- 能说“分场景一致性等级”。
**常见追问：**
- 延迟双删是否必须？
**追问详细回答：**
1. 不是必须。
2. 仅在并发竞争下脏读概率高时考虑。
3. 要评估复杂度与收益。

### Q63：Redis 集群模式有哪些？
**通俗理解：**
- 主从做复制，哨兵做切换，集群做分片扩容。
**详细回答：**
1. 主从：读扩展和容灾基础。
2. Sentinel：主从监控与故障转移。
3. Cluster：slot 分片 + 高可用。
4. 需要关注跨 slot 操作限制。
**面试加分点：**
- 说到 hash tag 定位同 slot 会加分。
**常见追问：**
- 如何让相关 key 落同一 slot？
**追问详细回答：**
1. 使用 `{tag}` 哈希标签。
2. `{same}:a` 与 `{same}:b` 会落同一 slot。
3. 便于多 key 操作。

### Q64：大 key 和热 key 怎么治理？
**通俗理解：**
- 大 key 像超大包裹，热 key 像单窗口万人排队。
**详细回答：**
1. 大 key：拆分、压缩、渐进删除。
2. 热 key：本地缓存、读扩散、分片。
3. 先监控识别，再灰度治理。
4. 治理后要有容量和告警策略。
**面试加分点：**
- 主动提 `UNLINK` 删除大 key 的优势。
**常见追问：**
- 如何在线识别热 key？
**追问详细回答：**
1. 统计 key 访问频次。
2. 结合慢日志、CPU、网络指标。
3. 通过代理层或监控工具定位。

### Q65：为什么 Redis 不能直接替代数据库？
**通俗理解：**
- Redis 是快，但不擅长复杂关系和强事务约束。
**详细回答：**
1. 关系查询和约束能力弱于 RDBMS。
2. 事务/一致性语义不适合主账场景。
3. 更适合作为缓存层和实时状态层。
4. 关键主数据仍应落关系型数据库。
**面试加分点：**
- “数据库为真，Redis 为快”这句话好记且专业。
**常见追问：**
- 哪些业务可 Redis 当主存？
**追问详细回答：**
1. 会话、排行榜、计数等弱一致场景。
2. 可容忍短暂丢失或已有补偿链路。
3. 资金、订单主链路不建议仅用 Redis。

### Q66：Pipeline 和 Lua 脚本如何选择？
**通俗理解：**
- Pipeline 省往返，Lua 保原子。
**详细回答：**
1. Pipeline 减少 RTT，提升吞吐。
2. Lua 在服务端原子执行，适合复合逻辑。
3. 需要强一致操作时优先 Lua。
4. 纯批量请求优化网络时优先 Pipeline。
**面试加分点：**
- 能提“Pipeline 不是事务”。
**常见追问：**
- Pipeline 会带来什么风险？
**追问详细回答：**
1. 单批过大可能阻塞。
2. 错误定位复杂度上升。
3. 需控制批次大小并监控延迟。

### Q67：Redis 故障演练怎么做？
**通俗理解：**
- 平时演练断电断网，出事才不会慌。
**详细回答：**
1. 演练主从切换、节点重启、缓存雪崩。
2. 验证告警触发、降级策略、恢复流程。
3. 记录 MTTR 和恢复步骤。
4. 形成 runbook 与自动化脚本。
**面试加分点：**
- 演练目标不是“走流程”，而是减少真实事故恢复时间。
**常见追问：**
- Redis 你会重点监控哪些指标？
**追问详细回答：**
1. 内存使用率与淘汰数。
2. 命令延迟、慢查询。
3. 复制延迟、连接数、命中率。

## 5. PostgreSQL（Q68-Q76）

### Q68：PostgreSQL 的核心优势是什么？
**通俗理解：**
- 功能更全、扩展更强，适合复杂数据场景。
**详细回答：**
1. 标准 SQL 支持度高。
2. 复杂查询能力强（窗口函数、CTE 等）。
3. JSONB 和扩展生态成熟。
4. MVCC 与事务能力稳定。
**面试加分点：**
- 可提 PostGIS、全文检索等扩展能力。
**常见追问：**
- PG 和 MySQL 怎么选？
**追问详细回答：**
1. 复杂查询、扩展需求强偏 PG。
2. 常规 OLTP 与团队经验成熟可选 MySQL。
3. 选型看业务场景与团队能力，不绝对化。

### Q69：PG 的 MVCC 核心机制？
**通俗理解：**
- 每条记录保留版本，读事务按快照看“当时世界”。
**详细回答：**
1. 行版本由 `xmin/xmax` 控制可见性。
2. 事务读取依赖快照，不与写强冲突。
3. 更新产生新版本，旧版本待清理。
4. 读写并发性能通常较好。
**面试加分点：**
- 能连接到 VACUUM 机制说明更完整。
**常见追问：**
- 表膨胀为什么会发生？
**追问详细回答：**
1. 更新/删除会产生死元组。
2. 清理不及时会堆积。
3. 长事务会阻碍可回收进度。

### Q70：VACUUM/AUTOVACUUM 有什么作用？
**通俗理解：**
- 像定期清理仓库垃圾，保证系统长期健康。
**详细回答：**
1. 回收死元组空间。
2. 维护统计信息，帮助优化器。
3. 防止事务 ID 回卷风险。
4. 对高写入表要重点调优参数。
**面试加分点：**
- 提到长事务会拖慢 autovacuum 很加分。
**常见追问：**
- 怎么判断 autovacuum 跟不上？
**追问详细回答：**
1. 死元组持续上升。
2. 表膨胀明显、查询变慢。
3. autovacuum 活动不足或被阻塞。

### Q71：PG 索引类型怎么选？
**通俗理解：**
- 不同查询模式要选不同索引工具。
**详细回答：**
1. B-Tree：等值和范围查询。
2. GIN：JSONB、数组、全文检索。
3. BRIN：超大表且数据有顺序相关。
4. 索引应与查询算子匹配。
**面试加分点：**
- 能区分 `jsonb_ops` 与 `jsonb_path_ops` 更专业。
**常见追问：**
- JSONB 常用什么索引？
**追问详细回答：**
1. 多数场景用 GIN。
2. 常规用 `jsonb_ops`。
3. 包含查询可考虑 `jsonb_path_ops`。

### Q72：`EXPLAIN ANALYZE` 怎么看？
**通俗理解：**
- 看地图不仅看规划路线，还要看真实通行时间。
**详细回答：**
1. 对比估算行数与实际行数偏差。
2. 找耗时最高的执行节点。
3. 看是否出现 seq scan、排序溢出等高成本操作。
4. 结合索引与 SQL 写法迭代优化。
**面试加分点：**
- 强调“优化前后计划对比与量化验证”。
**常见追问：**
- 为什么不走你建的索引？
**追问详细回答：**
1. 统计信息不准。
2. 选择性不足，顺序扫更便宜。
3. SQL 写法导致索引不可用。

### Q73：PG 锁机制如何回答？
**通俗理解：**
- 锁是并发秩序，不同操作申请不同级别通行证。
**详细回答：**
1. 有行级锁、表级锁、咨询锁等。
2. DDL/DML 会触发不同锁冲突。
3. 排障看 `pg_locks` 和阻塞链。
4. 缩短事务是降低锁冲突核心策略。
**面试加分点：**
- 讲到死锁日志与阻塞链定位路径。
**常见追问：**
- 如何定位死锁 SQL？
**追问详细回答：**
1. 查看数据库死锁日志。
2. 结合 `pg_locks` 找等待关系。
3. 对照应用日志还原事务顺序。

### Q74：PG 主从复制基础怎么讲？
**通俗理解：**
- 主库写 WAL，从库按日志追平。
**详细回答：**
1. 复制基于 WAL 日志流。
2. 支持同步与异步模式。
3. 异步延迟低但容灾时可能有少量丢失风险。
4. 复制延迟需要持续监控。
**面试加分点：**
- 能说清一致性与可用性的取舍。
**常见追问：**
- 复制延迟怎么看？
**追问详细回答：**
1. 主库看 `pg_stat_replication`。
2. 对比接收与重放 LSN 差距。
3. 结合业务 SLO 设阈值告警。

### Q75：分区表什么时候用？
**通俗理解：**
- 大表分区像把大仓库分区管理，查找和维护更高效。
**详细回答：**
1. 超大表按时间或范围分区。
2. 可利用分区裁剪减少扫描。
3. 有利于冷热数据管理和归档。
4. 分区键设计错误会带来额外复杂度。
**面试加分点：**
- 提醒分区不是性能万能药。
**常见追问：**
- 分区后一定更快吗？
**追问详细回答：**
1. 不一定，需命中分区裁剪才明显。
2. 跨分区查询可能仍贵。
3. 设计不当会增加维护成本。

### Q76：如何讲 PG 调优经验？
**通俗理解：**
- 先找瓶颈，再对症下药，不盲调参数。
**详细回答：**
1. 明确问题指标（慢查询、I/O、CPU）。
2. 优先优化 SQL 和索引。
3. 再做参数调优与存储策略优化。
4. 结果要量化，并说明风险控制。
**面试加分点：**
- 说出“SQL 优化优先于参数玄学”。
**常见追问：**
- 先改 SQL 还是先改参数？
**追问详细回答：**
1. 通常先改 SQL/索引。
2. 参数调优作为第二层。
3. 所有改动都要压测验证。

## 6. RabbitMQ（Q77-Q84）

### Q77：为什么要用 MQ？
**通俗理解：**
- 像中转站：生产者和消费者不用实时硬绑定。
**详细回答：**
1. 解耦上下游系统。
2. 削峰填谷应对突发流量。
3. 异步化降低主链路响应时间。
4. 可提升系统弹性与容错能力。
**面试加分点：**
- 同时提到“引入 MQ 带来一致性复杂度”。
**常见追问：**
- 哪些场景不适合上 MQ？
**追问详细回答：**
1. 低流量且强同步业务。
2. 团队无运维能力时风险大。
3. 复杂度增加可能大于收益。

### Q78：RabbitMQ 核心组件怎么讲？
**通俗理解：**
- Producer 发消息到 Exchange，Exchange 按规则转到 Queue，Consumer 消费。
**详细回答：**
1. Exchange 是路由中心。
2. Queue 是存储与消费队列。
3. Binding 绑定路由规则。
4. RoutingKey 决定消息走向。
**面试加分点：**
- 明确 direct/topic/fanout/headers 区别。
**常见追问：**
- 四种 Exchange 区别？
**追问详细回答：**
1. direct：精确匹配 routing key。
2. topic：通配符匹配。
3. fanout：广播。
4. headers：按 header 匹配（较少用）。

### Q79：消息可靠投递怎么保证？
**通俗理解：**
- 发件有回执，收件处理完再签收。
**详细回答：**
1. 生产端开启 confirm。
2. 消费端手动 ACK，失败重试。
3. 队列和消息设置持久化。
4. 异常场景要设计重投与补偿。
**面试加分点：**
- 指出“可靠投递 + 幂等消费”必须同时存在。
**常见追问：**
- confirm 和事务怎么选？
**追问详细回答：**
1. confirm 吞吐更高，生产常用。
2. 事务语义强但性能开销大。
3. 大多数场景选择 confirm + 幂等重试。

### Q80：重复消费如何处理？
**通俗理解：**
- 快递可能重复派送，收件人要能识别“已签收”。
**详细回答：**
1. MQ 常见语义是至少一次。
2. 消费端必须做幂等。
3. 幂等可基于业务唯一键或状态机。
4. 不把“恰好一次”完全寄托中间件。
**面试加分点：**
- 幂等记录应有 TTL/归档，防止无限增长。
**常见追问：**
- 幂等键如何设计？
**追问详细回答：**
1. 选业务天然唯一标识。
2. 要可重放、稳定、不可重复。
3. 设计过期策略与清理机制。

### Q81：消息积压怎么排查？
**通俗理解：**
- 入队比出队快，就像仓库进货快于出货，迟早爆仓。
**详细回答：**
1. 先看入队/出队速率差。
2. 查消费者处理能力与下游瓶颈。
3. 临时扩容消费者并限制重试风暴。
4. 后续做容量评估与限流策略。
**面试加分点：**
- 能给出“估算消费者数量”的方法。
**常见追问：**
- 消费者数量怎么估算？
**追问详细回答：**
1. 测单消费者吞吐。
2. `需要消费者数 >= 峰值入队速率 / 单消费者吞吐`。
3. 预留冗余应对抖动。

### Q82：死信队列（DLQ）有什么用？
**通俗理解：**
- 主路堵住的“问题包裹”转去旁路处理，不拖垮主线。
**详细回答：**
1. 承接重试失败消息。
2. 隔离毒消息避免主队列阻塞。
3. 便于人工/自动补偿。
4. 需配套治理流程。
**面试加分点：**
- DLQ 的消息必须可观测、可追踪、可处理。
**常见追问：**
- 如何区分可重试和不可重试错误？
**追问详细回答：**
1. 网络/超时类多为可重试。
2. 参数错误/业务冲突多为不可重试。
3. 用错误码分层分类更稳定。

### Q83：延迟消息如何实现？
**通俗理解：**
- 先把任务放“定时柜”，到点再取出来执行。
**详细回答：**
1. RabbitMQ 可用 TTL+DLX 组合。
2. 或使用延迟交换机插件。
3. 要考虑延迟精度与堆积风险。
4. 大规模定时任务可考虑专用调度系统。
**面试加分点：**
- 结合订单超时取消场景讲会更实战。
**常见追问：**
- 订单超时取消怎么设计？
**追问详细回答：**
1. 下单后投延迟消息。
2. 消费时校验状态，未支付则取消并回补库存。
3. 全流程幂等，避免重复处理。

### Q84：RabbitMQ 和 Kafka 怎么选？
**通俗理解：**
- RabbitMQ 擅长业务路由，Kafka 擅长海量日志流。
**详细回答：**
1. RabbitMQ 路由灵活、低延迟、业务队列友好。
2. Kafka 吞吐高、可回放、流处理生态强。
3. 二者不是替代关系。
4. 选型看消息语义、吞吐、顺序、运维成本。
**面试加分点：**
- 给出团队现状与迁移代价评估。
**常见追问：**
- 什么情况下迁移 Kafka？
**追问详细回答：**
1. 日志/埋点流量巨量增长。
2. 需要长期保留与回放。
3. 需要流处理生态能力。

## 7. HTTP / gRPC / 服务治理（Q85-Q90）

### Q85：REST 与 gRPC 如何选型？
**通俗理解：**
- 内部系统追性能和契约，用 gRPC；对外开放兼容性优先，用 REST。
**详细回答：**
1. gRPC 基于 HTTP/2，序列化高效，适合服务间调用。
2. REST 对浏览器和生态友好，调试门槛低。
3. 很多团队采用混合架构（内 gRPC、外 REST）。
4. 选型看调用方、工具链和组织能力。
**面试加分点：**
- 可提 gRPC-Gateway 作为桥接方案。
**常见追问：**
- 浏览器如何调用 gRPC？
**追问详细回答：**
1. 常见用 grpc-web + Envoy。
2. 或服务端提供 REST 转换层。
3. 浏览器不直接等价支持标准 gRPC 语义。

### Q86：HTTP 超时该如何配置？
**通俗理解：**
- 超时像安全阀，不配会把线程和连接拖死。
**详细回答：**
1. 服务端要配置读/写/空闲超时。
2. 客户端要配置连接、握手、响应头、整体超时。
3. 超时预算应按调用链分配。
4. 重试策略必须与超时策略协同。
**面试加分点：**
- 一定要讲“分层超时预算”。
**常见追问：**
- 为什么不能只配一个总超时？
**追问详细回答：**
1. 无法定位具体慢在哪段。
2. 重试容易放大流量。
3. 分段超时更利于排障和控制。

### Q87：幂等与重试如何设计？
**通俗理解：**
- 重试之前先保证“重放不会造成副作用”。
**详细回答：**
1. 先定义幂等键和状态机。
2. 仅对可恢复错误做重试。
3. 采用指数退避+抖动防重试风暴。
4. 服务端保证幂等比客户端保证更可靠。
**面试加分点：**
- 区分 4xx/5xx/429 处理策略。
**常见追问：**
- 429、5xx、4xx 分别怎么处理？
**追问详细回答：**
1. 429：退避后重试。
2. 5xx：幂等前提下有限重试。
3. 4xx：通常不重试，除非明确可恢复。

### Q88：限流、熔断、降级关系是什么？
**通俗理解：**
- 限流防进水过猛，熔断防故障扩散，降级保核心服务不断。
**详细回答：**
1. 限流控制入口负载。
2. 熔断在下游不稳时快速失败保护系统。
3. 降级在资源紧张时保住核心功能。
4. 三者应组合使用。
**面试加分点：**
- 能讲熔断状态机（关闭/打开/半开）。
**常见追问：**
- 半开状态如何判定恢复？
**追问详细回答：**
1. 放少量探测请求。
2. 连续成功达到阈值才关闭熔断。
3. 若失败则重新打开熔断。

### Q89：JWT 方案优缺点怎么讲？
**通俗理解：**
- JWT 像“自带说明书的门票”，验证方便，但回收难。
**详细回答：**
1. 优点：无状态、跨服务校验方便。
2. 缺点：主动失效难、泄露风险高。
3. 常见方案：短 token + refresh token + 黑名单。
4. 高安全场景要有设备管理和审计。
**面试加分点：**
- 主动提 token 轮换与吊销策略。
**常见追问：**
- JWT 放哪里更安全？
**追问详细回答：**
1. Web 常放 HttpOnly+Secure Cookie。
2. 降低 XSS 读取风险。
3. 结合 CSRF 防护与短过期策略。

### Q90：如何防止 API 契约漂移？
**通俗理解：**
- 接口文档和代码要“同一真相源”，否则前后端各说各话。
**详细回答：**
1. 采用契约优先（OpenAPI/Proto）。
2. 从契约生成代码和文档。
3. CI 增加契约校验与集成测试。
4. 版本管理要有兼容性规则。
**面试加分点：**
- 提到“破坏性变更门禁”会很加分。
**常见追问：**
- 你如何推动团队执行？
**追问详细回答：**
1. 先试点证明收益。
2. 把校验接入流水线，减少人为失误。
3. 用模板和脚手架降低上手成本。

## 8. Kubernetes / DevOps（Q91-Q98）

### Q91：Deployment 和 StatefulSet 如何选？
**通俗理解：**
- 无状态服务随时可替换，有状态服务需要“固定身份+固定存储”。
**详细回答：**
1. Deployment 适合无状态副本管理。
2. StatefulSet 提供稳定网络标识与有序部署。
3. 数据库、队列等有状态服务更适合 StatefulSet。
4. 存储一般配 PVC/PV。
**面试加分点：**
- 能说清 stateful 的运维风险与备份要求。
**常见追问：**
- 为什么 MySQL 不建议用 Deployment？
**追问详细回答：**
1. DB 需要稳定身份和卷绑定。
2. Deployment 的随机替换不利于状态一致性。
3. StatefulSet 更符合数据库运行语义。

### Q92：Service 和 Ingress 的关系？
**通俗理解：**
- Service 管内部转发，Ingress 管外部入口路由。
**详细回答：**
1. Service 提供服务发现和四层负载均衡。
2. Ingress 提供七层路由、TLS、域名管理。
3. 常见链路是 Ingress -> Service -> Pod。
4. 两者不是替代关系。
**面试加分点：**
- 可补云 LB 与 Ingress Controller 关系。
**常见追问：**
- Ingress 能替代 Service 吗？
**追问详细回答：**
1. 不能。
2. Ingress 的后端目标通常仍是 Service。
3. Service 是集群服务访问基础能力。

### Q93：ConfigMap 与 Secret 区别？
**通俗理解：**
- ConfigMap 放普通配置，Secret 放敏感信息。
**详细回答：**
1. 两者都可通过 env 或卷注入。
2. Secret 需要额外权限控制与审计。
3. 配置管理要做版本化和变更记录。
4. 敏感配置建议配合 KMS/密钥管理。
**面试加分点：**
- 提到“Secret 不是绝对安全，需配 RBAC 和加密”。
**常见追问：**
- 配置热更新怎么做？
**追问详细回答：**
1. 挂载文件并监听变更热加载。
2. 或配置中心推送触发刷新。
3. 不支持热更新的应用走滚动重启。

### Q94：liveness/readiness/startup 探针区别？
**通俗理解：**
- readiness 是能不能接单，liveness 是还活着没，startup 是启动保护。
**详细回答：**
1. liveness 失败会重启容器。
2. readiness 失败会摘流量，不一定重启。
3. startup 用于慢启动，避免误判。
4. 探针参数应根据真实启动/响应特性调整。
**面试加分点：**
- 提到“探针检查不能只测端口存活”。
**常见追问：**
- 探针参数怎么调优？
**追问详细回答：**
1. 参考应用启动时间和 P99。
2. 避免过严误杀，过松发现慢。
3. 结合线上指标迭代调整。

### Q95：HPA + requests/limits 如何协同？
**通俗理解：**
- HPA 决定“要几台”，requests/limits 决定“每台吃多少资源”。
**详细回答：**
1. request 决定调度保底资源。
2. limit 限制上限，防止抢占。
3. HPA 按指标自动扩缩容。
4. 需防止频繁震荡与指标延迟影响。
**面试加分点：**
- 可提 CPU 可超分、内存不可超分。
**常见追问：**
- 指标延迟会导致什么问题？
**追问详细回答：**
1. 扩容滞后导致峰值顶不住。
2. 缩容滞后造成资源浪费。
3. 需配冗余与防抖策略。

### Q96：K8s 故障排查顺序怎么讲？
**通俗理解：**
- 先看“状态和事件”，再看“日志和依赖”，最后看“节点和控制面”。
**详细回答：**
1. 检查 Pod 状态、事件、重启原因。
2. 查看容器日志与探针配置。
3. 检查资源限制、镜像拉取、DNS/网络。
4. 最后排查节点压力和控制面组件。
**面试加分点：**
- 给出固定 runbook 流程很加分。
**常见追问：**
- CrashLoopBackOff 常见根因？
**追问详细回答：**
1. 启动命令错或配置缺失。
2. 依赖不可达、探针误判。
3. OOMKilled 后反复重启。

### Q97：CI/CD 流程如何设计更稳？
**通俗理解：**
- 像工厂流水线，每道质检不过就不放行。
**详细回答：**
1. 基础阶段：构建、单测、集成测试。
2. 安全阶段：镜像扫描、依赖漏洞扫描。
3. 发布阶段：灰度、验收、自动回滚。
4. 全流程版本可追溯与审计。
**面试加分点：**
- 数据库迁移纳入发布门禁是关键点。
**常见追问：**
- 发布前你会加哪些检查？
**追问详细回答：**
1. 契约兼容检查。
2. 迁移脚本可回滚验证。
3. 关键指标基线对比。

### Q98：可观测性三件套如何落地？
**通俗理解：**
- 日志看细节、指标看趋势、追踪看全链路。
**详细回答：**
1. 日志结构化并统一字段规范。
2. 指标覆盖四类黄金信号（延迟、流量、错误、饱和）。
3. Trace 贯穿网关到下游服务。
4. 三者通过 trace_id 关联。
**面试加分点：**
- 强调“告警要可执行，不要噪音”。
**常见追问：**
- 你最关注哪些 SLI？
**追问详细回答：**
1. 成功率/错误率。
2. P95/P99 延迟。
3. 饱和度（CPU、内存、队列积压）。

## 9. 综合项目表达（Q99-Q100）

### Q99：项目经历怎么讲才更优秀？
**通俗理解：**
- 不只讲“做了啥”，要讲“为什么这么做、结果如何”。
**详细回答：**
1. 背景：业务目标与规模。
2. 问题：当时最大瓶颈与风险。
3. 行动：你的决策、实现、权衡。
4. 结果：量化指标变化（P99、错误率、成本）。
**面试加分点：**
- 用 STAR 结构，带真实数字。
**常见追问：**
- 如果重做一次你会怎么优化？
**追问详细回答：**
1. 先说会保留的正确决策。
2. 再说会前置的能力（测试、监控、容量）。
3. 最后给可落地的分阶段演进计划。

### Q100：面试结尾反问什么最加分？
**通俗理解：**
- 反问是展示你“关注长期交付能力”的机会。
**详细回答：**
1. 问团队核心技术挑战和业务阶段。
2. 问发布质量门禁、值班与事故机制。
3. 问岗位 3 个月目标与考核方式。
4. 问技术成长路径与协作方式。
**面试加分点：**
- 反问不要只问福利，要体现你能创造价值。
