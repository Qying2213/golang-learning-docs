# 输入输出系统 ⭐⭐

> 本章目标：理解 I/O 的工作方式，为学习操作系统的 I/O 模型打基础

---

## 1. I/O 设备概述

### 1.1 常见 I/O 设备

| 类型 | 设备 | 特点 |
|------|------|------|
| 输入设备 | 键盘、鼠标、网卡（接收） | 向计算机输入数据 |
| 输出设备 | 显示器、打印机、网卡（发送） | 从计算机输出数据 |
| 存储设备 | 硬盘、SSD、U盘 | 既输入又输出 |

### 1.2 I/O 的挑战

**问题：** I/O 设备比 CPU 慢太多

```
CPU 执行指令：0.3 ns
内存访问：100 ns
SSD 读取：100,000 ns（100 μs）
网络请求：1,000,000 ns（1 ms）起步
```

如果 CPU 等待 I/O 完成，大部分时间在空转。

## 2. I/O 控制方式

### 2.1 程序查询方式（轮询）

CPU 不断查询 I/O 设备是否完成：

```
while (设备未就绪) {
    // CPU 空转等待
}
读取数据
```

**Go 中的类比：**
```go
// 轮询方式（不推荐）
for {
    data, err := tryRead()
    if err == nil {
        process(data)
        break
    }
    // CPU 空转
}
```

**缺点：** CPU 利用率低，一直在等待

### 2.2 中断方式

I/O 完成后，设备发送中断信号通知 CPU：

```
1. CPU 发起 I/O 请求
2. CPU 去做其他事情
3. I/O 完成，设备发送中断
4. CPU 暂停当前工作，处理 I/O 数据
5. CPU 继续之前的工作
```

```
时间线：
CPU:  [工作A] [发起I/O] [工作B] [中断处理] [工作B继续]
I/O:           [读取数据........] [完成]
                                    ↑
                                  发送中断
```

**优点：** CPU 不用空转，可以做其他事

### 2.3 DMA（直接内存访问）

DMA 控制器直接在设备和内存之间传输数据，不需要 CPU 参与：

```
传统方式：
设备 → CPU → 内存（CPU 搬运数据）

DMA 方式：
设备 → DMA 控制器 → 内存（CPU 不参与）
         ↓
      完成后中断 CPU
```

**优点：** 大量数据传输时，CPU 完全解放

**Go 中的体现：**
```go
// 读取大文件时，操作系统使用 DMA
file, _ := os.Open("bigfile.dat")
data := make([]byte, 1024*1024)
file.Read(data)  // 底层使用 DMA，CPU 不用逐字节搬运
```

## 3. I/O 与操作系统

### 3.1 系统调用

用户程序通过系统调用请求 I/O：

```
用户程序
    ↓ 系统调用（read/write）
操作系统内核
    ↓ 驱动程序
硬件设备
```

**Go 中的 I/O 系统调用：**
```go
// 这些操作最终都会调用操作系统的系统调用
file.Read()   // → syscall.Read()
file.Write()  // → syscall.Write()
conn.Read()   // → syscall.Read()
```

### 3.2 缓冲区

操作系统使用缓冲区减少 I/O 次数：

```
用户程序写入：
write("a")  → 缓冲区 [a]
write("b")  → 缓冲区 [ab]
write("c")  → 缓冲区 [abc]
flush()     → 一次性写入磁盘 [abc]
```

**Go 中的缓冲：**
```go
// 无缓冲：每次 Write 都是系统调用
file.Write([]byte("a"))
file.Write([]byte("b"))
file.Write([]byte("c"))
// 3 次系统调用

// 有缓冲：积累后一次写入
writer := bufio.NewWriter(file)
writer.Write([]byte("a"))
writer.Write([]byte("b"))
writer.Write([]byte("c"))
writer.Flush()
// 1 次系统调用
```

## 4. 磁盘 I/O

### 4.1 机械硬盘（HDD）

```
┌─────────────────────────────────┐
│           磁盘结构               │
│                                  │
│    ┌─────────────────────┐      │
│    │    ○ ○ ○ ○ ○ ○      │ ← 磁道│
│    │   ○ ○ ○ ○ ○ ○ ○     │      │
│    │  ○ ○ ○ ○ ○ ○ ○ ○    │      │
│    │ ○ ○ ○ ○ ● ○ ○ ○ ○   │ ← 磁头│
│    │  ○ ○ ○ ○ ○ ○ ○ ○    │      │
│    │   ○ ○ ○ ○ ○ ○ ○     │      │
│    │    ○ ○ ○ ○ ○ ○      │      │
│    └─────────────────────┘      │
└─────────────────────────────────┘

访问时间 = 寻道时间 + 旋转延迟 + 传输时间
         ≈ 5ms + 4ms + 0.1ms
         ≈ 10ms
```

**顺序读写 vs 随机读写：**
- 顺序读写：100-200 MB/s
- 随机读写：0.5-1 MB/s（差 100 倍！）

### 4.2 固态硬盘（SSD）

```
SSD 没有机械部件，用闪存芯片存储

访问时间 ≈ 0.1ms（比 HDD 快 100 倍）

顺序读写：500-3000 MB/s
随机读写：50-500 MB/s
```

### 4.3 对 Go 程序的影响

```go
// 顺序写入（快）
for i := 0; i < 1000000; i++ {
    file.Write(data)  // 追加写入
}

// 随机写入（慢）
for i := 0; i < 1000000; i++ {
    file.Seek(rand.Int63(), 0)  // 随机位置
    file.Write(data)
}
```

## 5. 网络 I/O

### 5.1 网络 I/O 的特点

```
网络 I/O 延迟：
本地网络：0.1-1 ms
互联网：10-100 ms
跨国：100-300 ms

带宽：
千兆网卡：125 MB/s
万兆网卡：1.25 GB/s
```

### 5.2 网络 I/O 模型

**阻塞 I/O：**
```go
// 阻塞等待数据
conn.Read(buf)  // 没有数据就一直等
```

**非阻塞 I/O：**
```go
// 立即返回，没数据返回错误
conn.SetNonblock(true)
n, err := conn.Read(buf)
if err == EAGAIN {
    // 没有数据，稍后再试
}
```

**I/O 多路复用：**
```go
// 同时监听多个连接
// Go 的 netpoller 底层使用 epoll/kqueue
for {
    conn, _ := listener.Accept()
    go handleConn(conn)  // 每个连接一个 goroutine
}
```

### 5.3 Go 的网络模型

Go 使用 goroutine + netpoller 实现高并发网络：

```
┌─────────────────────────────────────────┐
│              Go 程序                     │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐   │
│  │goroutine│ │goroutine│ │goroutine│   │
│  │ conn1   │ │ conn2   │ │ conn3   │   │
│  └────┬────┘ └────┬────┘ └────┬────┘   │
│       └──────────┬┴───────────┘        │
│                  ↓                      │
│           ┌──────────────┐             │
│           │  netpoller   │             │
│           │ (epoll/kqueue)│             │
│           └──────────────┘             │
└─────────────────────────────────────────┘
                   ↓
              操作系统内核
```

**优势：**
- 每个连接一个 goroutine，代码简单
- 底层使用 epoll/kqueue，性能高
- 自动调度，不需要手动管理

## 6. I/O 性能优化

### 6.1 减少系统调用

```go
// 差：多次系统调用
for _, line := range lines {
    file.Write([]byte(line))
}

// 好：合并后一次系统调用
var buf bytes.Buffer
for _, line := range lines {
    buf.WriteString(line)
}
file.Write(buf.Bytes())
```

### 6.2 使用缓冲

```go
// 使用 bufio 减少系统调用
reader := bufio.NewReader(file)
writer := bufio.NewWriter(file)
```

### 6.3 并发 I/O

```go
// 并发读取多个文件
var wg sync.WaitGroup
for _, filename := range files {
    wg.Add(1)
    go func(f string) {
        defer wg.Done()
        data, _ := os.ReadFile(f)
        process(data)
    }(filename)
}
wg.Wait()
```

### 6.4 使用 mmap

```go
// 内存映射文件，避免数据复制
import "golang.org/x/exp/mmap"

reader, _ := mmap.Open("bigfile.dat")
data := reader.Data()  // 直接访问文件内容
```

## 7. 练习题

### 练习1：I/O 方式对比

**Q：中断方式相比轮询方式的优点是什么？**

<details>
<summary>查看答案</summary>

1. **CPU 利用率高**：轮询时 CPU 空转等待，中断时 CPU 可以做其他事
2. **响应及时**：设备完成后立即通知 CPU
3. **适合慢速设备**：不用一直占用 CPU

缺点：中断处理有开销，频繁中断反而降低性能

</details>

### 练习2：缓冲的作用

**Q：为什么使用 bufio 可以提升性能？**

<details>
<summary>查看答案</summary>

1. **减少系统调用**：系统调用有开销（用户态→内核态切换）
2. **批量处理**：积累数据后一次性读写
3. **减少磁盘 I/O**：磁盘按块读写，小数据合并后效率更高

例子：
- 无缓冲：写 1000 次，1000 次系统调用
- 有缓冲：写 1000 次，可能只有 10 次系统调用

</details>

### 练习3：Go 网络模型

**Q：Go 如何实现高并发网络？**

<details>
<summary>查看答案</summary>

Go 使用 **goroutine + netpoller** 模型：

1. 每个连接一个 goroutine（代码简单）
2. 底层使用 epoll/kqueue（I/O 多路复用）
3. goroutine 阻塞时自动让出 CPU
4. 数据就绪时自动唤醒 goroutine

这样既有同步代码的简洁，又有异步 I/O 的性能。

</details>

## 8. 本章小结

```
I/O 控制方式：
├── 轮询：CPU 空转等待（效率低）
├── 中断：设备完成后通知 CPU（常用）
└── DMA：设备直接访问内存（大数据传输）

磁盘 I/O：
├── HDD：机械硬盘，随机访问慢
├── SSD：固态硬盘，随机访问快
└── 顺序访问比随机访问快很多

网络 I/O：
├── 阻塞 I/O：等待数据
├── 非阻塞 I/O：立即返回
└── I/O 多路复用：同时监听多个连接

Go 的 I/O：
├── goroutine + netpoller
├── 使用 bufio 减少系统调用
└── 并发 I/O 提升吞吐量
```

**Go 开发者要点：**
1. 使用 bufio 减少系统调用
2. 顺序读写比随机读写快
3. Go 的网络模型已经很高效，直接用就行
4. 大文件考虑 mmap
