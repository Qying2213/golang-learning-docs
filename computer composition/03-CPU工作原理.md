## 2. 寄存器

### 2.1 什么是寄存器？

寄存器是 CPU 内部的存储单元，速度最快，数量很少。

**类比：** 寄存器就像你手里拿着的东西，内存就像抽屉里的东西。手里的东西用起来最快，但只能拿几个。

### 2.2 常见寄存器（x86-64）

| 寄存器 | 用途 | 说明 |
|--------|------|------|
| RAX | 累加器 | 存放运算结果、函数返回值 |
| RBX | 基址寄存器 | 通用 |
| RCX | 计数器 | 循环计数 |
| RDX | 数据寄存器 | 存放数据 |
| RSP | 栈指针 | 指向栈顶 |
| RBP | 基址指针 | 指向栈帧底部 |
| RIP | 指令指针 | 指向下一条要执行的指令 |
| RFLAGS | 标志寄存器 | 存放运算状态（进位、零、负等） |

### 2.3 Go 程序中的寄存器

```go
func add(a, b int) int {
    return a + b
}
```

编译后（简化）：
```assembly
; a 在 AX 寄存器，b 在 BX 寄存器
ADD AX, BX    ; AX = AX + BX
RET           ; 返回，结果在 AX 中
```

**查看 Go 汇编：**
```bash
go tool compile -S main.go
```

## 3. 指令系统

### 3.1 什么是指令？

指令是 CPU 能理解的命令，由操作码和操作数组成。

```
指令格式：
┌──────────────┬──────────────┬──────────────┐
│   操作码     │   操作数1    │   操作数2    │
│  (做什么)    │  (对谁做)    │  (对谁做)    │
└──────────────┴──────────────┴──────────────┘

例子：
ADD  AX, BX
 ↓    ↓   ↓
操作码 目标 源
(加法) (AX) (BX)

含义：AX = AX + BX
```

### 3.2 常见指令类型

| 类型 | 指令 | 说明 | 例子 |
|------|------|------|------|
| 数据传送 | MOV | 复制数据 | `MOV AX, 10` |
| 算术运算 | ADD, SUB, MUL, DIV | 加减乘除 | `ADD AX, BX` |
| 逻辑运算 | AND, OR, XOR, NOT | 位运算 | `AND AX, 0xFF` |
| 比较 | CMP | 比较两个值 | `CMP AX, BX` |
| 跳转 | JMP, JE, JNE | 改变执行流程 | `JMP label` |
| 调用 | CALL, RET | 函数调用和返回 | `CALL func` |
| 栈操作 | PUSH, POP | 压栈和出栈 | `PUSH AX` |

### 3.3 Go 代码对应的指令

```go
// Go 代码
a := 10
b := 20
c := a + b
if c > 25 {
    c = c - 5
}
```

```assembly
; 对应的汇编（简化）
MOV  AX, 10      ; a := 10
MOV  BX, 20      ; b := 20
ADD  AX, BX      ; c := a + b (AX = 30)
CMP  AX, 25      ; 比较 c 和 25
JLE  skip        ; 如果 c <= 25，跳过
SUB  AX, 5       ; c = c - 5
skip:
```

## 4. 指令执行周期

### 4.1 五个阶段

```
┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌─────────┐
│  取指   │ → │  译码   │ → │  执行   │ → │  访存   │ → │  写回   │
│ Fetch   │   │ Decode  │   │ Execute │   │ Memory  │   │ Write   │
└─────────┘   └─────────┘   └─────────┘   └─────────┘   └─────────┘
```

**详细说明：**

| 阶段 | 做什么 | 例子（ADD AX, [100]） |
|------|--------|----------------------|
| 取指 | 从内存读取指令 | 读取 ADD 指令 |
| 译码 | 解析指令，知道要做什么 | 知道是加法，操作数是 AX 和内存地址 100 |
| 执行 | 执行运算 | ALU 执行加法 |
| 访存 | 读写内存（如果需要） | 从地址 100 读取数据 |
| 写回 | 把结果写回寄存器 | 结果写入 AX |

### 4.2 时钟周期

CPU 的工作由时钟信号驱动：

```
时钟信号：
    ┌──┐  ┌──┐  ┌──┐  ┌──┐
    │  │  │  │  │  │  │  │
────┘  └──┘  └──┘  └──┘  └──

每个上升沿，CPU 执行一个操作
```

**主频 3 GHz = 每秒 30 亿个时钟周期**

## 5. 流水线

### 5.1 为什么需要流水线？

**没有流水线：**
```
指令1: 取指 → 译码 → 执行 → 访存 → 写回
                                        指令2: 取指 → 译码 → 执行 → 访存 → 写回

执行 2 条指令需要 10 个周期
```

**有流水线：**
```
时钟周期:  1     2     3     4     5     6
指令1:   取指  译码  执行  访存  写回
指令2:         取指  译码  执行  访存  写回
指令3:               取指  译码  执行  访存  写回

执行 3 条指令只需要 7 个周期！
```

### 5.2 流水线的问题

**1. 数据冒险（Data Hazard）**
```assembly
ADD AX, BX    ; AX = AX + BX
SUB CX, AX    ; CX = CX - AX（需要等 ADD 的结果）
```
第二条指令需要第一条的结果，必须等待。

**2. 控制冒险（Control Hazard）**
```assembly
CMP AX, 10
JE  label     ; 如果相等就跳转
ADD BX, 1     ; 这条指令要不要执行？不知道！
```
跳转指令导致不知道下一条该执行什么。

### 5.3 分支预测

CPU 会**猜测**跳转的结果，提前执行：

```
if (x > 0) {
    // 分支 A
} else {
    // 分支 B
}
```

- CPU 猜测走分支 A，提前执行 A 的指令
- 如果猜对了，继续执行
- 如果猜错了，丢弃已执行的结果，重新执行（浪费 10-20 个周期）

**Go 性能优化：让分支更可预测**
```go
// 不好：随机分支，预测困难
for _, v := range data {
    if v > threshold {
        sum += v
    }
}

// 好：先排序，分支更可预测
sort.Ints(data)
for _, v := range data {
    if v > threshold {
        sum += v
    }
}
```

## 6. 多核与并行

### 6.1 多核 CPU

```
┌─────────────────────────────────────────────────────┐
│                      CPU                             │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  ┌─────────┐│
│  │ Core 0  │  │ Core 1  │  │ Core 2  │  │ Core 3  ││
│  │  L1/L2  │  │  L1/L2  │  │  L1/L2  │  │  L1/L2  ││
│  └────┬────┘  └────┬────┘  └────┬────┘  └────┬────┘│
│       └────────────┴────────────┴────────────┘     │
│                        │                            │
│               ┌────────┴────────┐                  │
│               │    L3 Cache     │                  │
│               └─────────────────┘                  │
└─────────────────────────────────────────────────────┘
```

### 6.2 Go 的并行

```go
// 设置使用的核心数
runtime.GOMAXPROCS(4)

// goroutine 可以在不同核心上并行执行
go func() { /* 可能在 Core 0 */ }()
go func() { /* 可能在 Core 1 */ }()
go func() { /* 可能在 Core 2 */ }()
go func() { /* 可能在 Core 3 */ }()
```

### 6.3 缓存一致性问题

多核 CPU 每个核心有自己的 Cache，可能导致数据不一致：

```
Core 0 的 Cache: x = 10
Core 1 的 Cache: x = 10

Core 0 修改: x = 20（只在 Core 0 的 Cache 中）
Core 1 读取: x = 10（还是旧值！）
```

**解决：缓存一致性协议（MESI）**

CPU 硬件自动保证缓存一致性，但有性能开销。

**Go 中的影响：**
```go
// 多个 goroutine 访问同一个变量需要同步
var counter int64

// 错误：数据竞争
go func() { counter++ }()
go func() { counter++ }()

// 正确：使用原子操作
go func() { atomic.AddInt64(&counter, 1) }()
go func() { atomic.AddInt64(&counter, 1) }()
```

## 7. CPU 性能指标

### 7.1 主要指标

| 指标 | 含义 | 例子 |
|------|------|------|
| 主频 | 每秒时钟周期数 | 3.5 GHz |
| 核心数 | CPU 核心数量 | 8 核 |
| IPC | 每周期执行指令数 | IPC = 4 |
| 缓存大小 | L1/L2/L3 容量 | L3 = 16 MB |

### 7.2 性能计算

```
执行时间 = 指令数 × CPI ÷ 主频

CPI = 1 / IPC

例子：
- 程序有 10 亿条指令
- IPC = 2（CPI = 0.5）
- 主频 = 4 GHz

执行时间 = 1,000,000,000 × 0.5 ÷ 4,000,000,000
        = 0.125 秒
```

## 8. 练习题

### 练习1：指令理解

**Q：以下汇编代码做了什么？**
```assembly
MOV AX, 5
MOV BX, 3
ADD AX, BX
```

<details>
<summary>查看答案</summary>

1. 把 5 放入 AX 寄存器
2. 把 3 放入 BX 寄存器
3. AX = AX + BX = 5 + 3 = 8

最终 AX = 8

</details>

### 练习2：流水线

**Q：5 级流水线执行 100 条指令，理想情况下需要多少个时钟周期？**

<details>
<summary>查看答案</summary>

```
第 1 条指令：5 个周期
第 2-100 条指令：每条 1 个周期（流水线满载）

总周期 = 5 + (100 - 1) = 104 个周期
```

如果没有流水线：100 × 5 = 500 个周期

流水线加速比：500 / 104 ≈ 4.8 倍

</details>

### 练习3：分支预测

**Q：为什么排序后的数组遍历比未排序的快？**

```go
// 未排序
data := []int{3, 1, 4, 1, 5, 9, 2, 6, ...}  // 随机

// 排序后
data := []int{1, 1, 2, 3, 4, 5, 6, 9, ...}  // 有序

for _, v := range data {
    if v > 5 {
        sum += v
    }
}
```

<details>
<summary>查看答案</summary>

**未排序时：**
- 数据随机，分支结果不可预测
- CPU 分支预测经常猜错
- 每次猜错浪费 10-20 个周期

**排序后：**
- 前面都是 ≤5 的数，分支都不走
- 后面都是 >5 的数，分支都走
- CPU 分支预测很容易猜对

所以排序后的数组遍历更快，即使排序本身需要时间。

</details>

## 9. 本章小结

```
CPU 组成：
├── 控制单元 (CU)：取指、译码、控制
├── 运算单元 (ALU)：算术和逻辑运算
└── 寄存器：高速存储

指令执行：
取指 → 译码 → 执行 → 访存 → 写回

流水线：
├── 多条指令重叠执行
├── 数据冒险：需要等待前面指令的结果
└── 控制冒险：分支导致不确定

分支预测：
├── CPU 猜测分支结果
├── 猜对：继续执行
└── 猜错：浪费周期

多核：
├── 多个核心并行执行
├── 缓存一致性问题
└── Go 的 goroutine 可以利用多核
```

**Go 开发者要点：**
1. 原子操作比锁更快（直接用 CPU 指令）
2. 让分支更可预测可以提升性能
3. 多核并行需要注意数据竞争
