# 存储系统 ⭐⭐⭐

> 本章目标：理解内存层次结构、Cache 原理、虚拟内存，为学习 Go 内存管理打基础

## 1. 存储层次结构

### 1.1 为什么需要层次结构？

**问题：** CPU 太快，内存太慢

```
CPU 执行一条指令：0.3 ns
访问内存：100 ns

如果每次都访问内存，CPU 99% 的时间在等待！
```

**解决：** 用多级存储，越靠近 CPU 越快越小

### 1.2 存储金字塔

```
                    ┌───────┐
                    │寄存器 │  ← 最快，几十个
                    │ <1ns  │
                   ┌┴───────┴┐
                   │L1 Cache │  ← 很快，32-64 KB
                   │  1-2ns  │
                  ┌┴─────────┴┐
                  │ L2 Cache  │  ← 快，256KB-1MB
                  │   4-7ns   │
                 ┌┴───────────┴┐
                 │  L3 Cache   │  ← 较快，8-64MB
                 │   12-20ns   │
                ┌┴─────────────┴┐
                │     内存      │  ← 中等，8-128GB
                │   80-100ns   │
               ┌┴───────────────┴┐
               │      SSD        │  ← 较慢，256GB-4TB
               │   10-100μs     │
              ┌┴─────────────────┴┐
              │       HDD         │  ← 慢，1-10TB
              │    5-10ms        │
              └───────────────────┘
```

### 1.3 访问延迟对比

| 存储 | 延迟 | 相对时间（如果 1 周期 = 1 秒） |
|------|------|------------------------------|
| 寄存器 | 0.3 ns | 1 秒 |
| L1 Cache | 1 ns | 3 秒 |
| L2 Cache | 4 ns | 13 秒 |
| L3 Cache | 12 ns | 40 秒 |
| 内存 | 100 ns | 5 分钟 |
| SSD | 100 μs | 4 天 |
| HDD | 10 ms | 1 年 |

## 2. Cache（高速缓存）

### 2.1 Cache 的作用

Cache 存储最近使用的数据，下次访问时直接从 Cache 读取。

```
CPU 需要数据 X
    ↓
先查 L1 Cache
    ↓ 没有
查 L2 Cache
    ↓ 没有
查 L3 Cache
    ↓ 没有
从内存读取 X，同时存入 Cache
    ↓
下次访问 X 时，直接从 Cache 读取（快！）
```

### 2.2 局部性原理

Cache 有效的原因：**程序访问数据有规律**

**时间局部性：** 刚访问的数据很可能再次访问
```go
for i := 0; i < 1000; i++ {
    sum += arr[0]  // arr[0] 被访问 1000 次
}
```

**空间局部性：** 访问一个地址后，很可能访问附近的地址
```go
for i := 0; i < len(arr); i++ {
    sum += arr[i]  // 顺序访问，空间局部性好
}
```

### 2.3 Cache 命中与缺失

```
Cache 命中（Hit）：数据在 Cache 中，直接读取
Cache 缺失（Miss）：数据不在 Cache 中，需要从内存读取

命中率 = 命中次数 / 总访问次数

命中率 95% 意味着：
- 95% 的访问只需要几纳秒
- 5% 的访问需要 100 纳秒
```

### 2.4 Cache 行（Cache Line）

Cache 不是一个字节一个字节存的，而是按**行**存储：

```
Cache Line 大小：通常 64 字节

访问 arr[0] 时，会把 arr[0] 到 arr[7] 都加载到 Cache
（假设 int64，每个 8 字节，64 / 8 = 8 个元素）
```

**Go 性能优化：利用 Cache Line**
```go
// 好：顺序访问 64 个元素，Cache 友好
// 第一次访问触发 1 次 cache miss，后续 63 次都是 cache hit
for i := 0; i < 64; i++ {
    sum += arr[i]
}

// 差：跳跃访问 64 个元素，Cache 不友好
// 每次访问都跳过一个 Cache Line，导致每次都可能 cache miss
for i := 0; i < 64; i++ {
    sum += arr[i * 64]
}
```

**关键点：** 相同的访问次数下，顺序访问能复用 Cache Line 中预加载的数据，跳跃访问则浪费了这些数据。

### 2.5 伪共享（False Sharing）

多核 CPU 中，不同核心修改同一 Cache Line 的不同变量会导致性能下降：

```go
// 伪共享问题
type Data struct {
    a int64  // Core 0 修改
    b int64  // Core 1 修改
}
// a 和 b 在同一个 Cache Line，互相影响

// 解决：填充，让它们在不同 Cache Line
type Data struct {
    a int64
    _ [56]byte  // 填充 56 字节，凑够 64 字节
    b int64
}
```

## 3. 内存

### 3.1 内存的组织

```
内存地址空间（64 位系统）：
0x0000000000000000 ~ 0xFFFFFFFFFFFFFFFF

实际可用：通常 48 位地址，256 TB 空间
```

### 3.2 内存访问

```go
var x int64 = 100

// x 存储在内存的某个地址，比如 0x7fff5fbff8a0
// 访问 x 时：
// 1. CPU 发出地址 0x7fff5fbff8a0
// 2. 内存控制器找到对应位置
// 3. 读取 8 字节数据返回给 CPU
```

### 3.3 内存对齐

CPU 访问对齐的数据更快：

```go
// 对齐的结构体
type Good struct {
    a int64  // 8 字节，偏移 0
    b int64  // 8 字节，偏移 8
}
// 大小：16 字节

// 未对齐的结构体
type Bad struct {
    a int8   // 1 字节，偏移 0
    b int64  // 8 字节，偏移 8（编译器自动填充 7 字节）
    c int8   // 1 字节，偏移 16
}
// 大小：24 字节（有 8 字节浪费）

// 优化后
type Better struct {
    b int64  // 8 字节，偏移 0
    a int8   // 1 字节，偏移 8
    c int8   // 1 字节，偏移 9
}
// 大小：16 字节
```

**Go 中查看结构体大小：**
```go
fmt.Println(unsafe.Sizeof(Good{}))   // 16
fmt.Println(unsafe.Sizeof(Bad{}))    // 24
fmt.Println(unsafe.Sizeof(Better{})) // 16
```

## 4. 虚拟内存

### 4.1 为什么需要虚拟内存？

**问题：**
1. 多个程序同时运行，如何分配内存？
2. 程序需要的内存比物理内存大怎么办？
3. 如何保护程序之间不互相干扰？

**虚拟内存如何解决这些问题：**

| 问题 | 虚拟内存的解决方案 |
|------|-------------------|
| 内存分配 | 每个程序有独立的虚拟地址空间，操作系统负责映射到物理内存 |
| 内存不够 | 把不常用的页换出到磁盘（Swap），需要时再换回来 |
| 程序隔离 | 程序只能访问自己的虚拟地址，无法访问其他程序的内存 |

### 4.2 虚拟地址 vs 物理地址

```
程序 A 看到的地址（虚拟地址）：
0x0000 ~ 0xFFFF

程序 B 看到的地址（虚拟地址）：
0x0000 ~ 0xFFFF

实际物理内存：
┌─────────────────────────────────┐
│  程序 A 的数据  │  程序 B 的数据  │
│  物理地址 0x1000│  物理地址 0x2000│
└─────────────────────────────────┘

虚拟地址 → MMU 翻译 → 物理地址
```

### 4.3 页（Page）

内存按**页**管理，通常 4 KB 一页：

```
虚拟地址空间：
┌────────┬────────┬────────┬────────┐
│ Page 0 │ Page 1 │ Page 2 │ Page 3 │
└────────┴────────┴────────┴────────┘
    ↓         ↓         ↓         ↓
   映射      映射      映射      映射
    ↓         ↓         ↓         ↓
┌────────┬────────┬────────┬────────┐
│Frame 5 │Frame 2 │ 磁盘   │Frame 8 │  ← 物理内存/磁盘
└────────┴────────┴────────┴────────┘
```

### 4.4 页表

页表记录虚拟页到物理页的映射：

```
页表：
┌──────────┬──────────┬───────┐
│ 虚拟页号 │ 物理页号 │ 有效位│
├──────────┼──────────┼───────┤
│    0     │    5     │   1   │  ← 在内存中
│    1     │    2     │   1   │  ← 在内存中
│    2     │    -     │   0   │  ← 在磁盘上
│    3     │    8     │   1   │  ← 在内存中
└──────────┴──────────┴───────┘
```

### 4.5 缺页中断

访问不在内存中的页时，触发缺页中断：

```
1. 程序访问虚拟页 2
2. 查页表，发现有效位 = 0（不在内存）
3. 触发缺页中断
4. 操作系统从磁盘读取页 2 到内存
5. 更新页表
6. 重新执行访问
```

### 4.6 Go 程序的内存布局

```
高地址
┌─────────────────────┐
│       栈 (Stack)    │  ← goroutine 的栈
│         ↓           │
├─────────────────────┤
│                     │
│      空闲区域       │
│                     │
├─────────────────────┤
│         ↑           │
│       堆 (Heap)     │  ← Go 的内存分配器管理
├─────────────────────┤
│       数据段        │  ← 全局变量
├─────────────────────┤
│       代码段        │  ← 程序代码
└─────────────────────┘
低地址
```

## 5. Go 内存管理预览

### 5.1 栈 vs 堆

```go
func foo() {
    x := 10      // 可能在栈上
    y := new(int) // 可能在堆上
    *y = 20
}
```

**栈：**
- 分配快（只需移动栈指针）
- 自动回收（函数返回时）
- 大小有限（Go 默认 1 MB，可增长）

**堆：**
- 分配慢（需要内存分配器）
- 需要 GC 回收
- 大小几乎无限

### 5.2 逃逸分析

Go 编译器决定变量放栈还是堆：

```go
func foo() *int {
    x := 10
    return &x  // x 逃逸到堆上，因为函数返回后还要用
}

func bar() int {
    x := 10
    return x   // x 在栈上，函数返回后不需要了
}
```

**查看逃逸分析：**
```bash
go build -gcflags="-m" main.go
```

## 6. 练习题

### 练习1：Cache 命中率

**Q：以下代码哪个 Cache 命中率更高？**

```go
// 代码 A
for i := 0; i < 1000; i++ {
    for j := 0; j < 1000; j++ {
        sum += arr[i][j]
    }
}

// 代码 B
for j := 0; j < 1000; j++ {
    for i := 0; i < 1000; i++ {
        sum += arr[i][j]
    }
}
```

<details>
<summary>查看答案</summary>

**代码 A 命中率更高**

原因：
- Go 的二维数组按行存储：arr[0][0], arr[0][1], ..., arr[0][999], arr[1][0], ...
- 代码 A 按行访问，空间局部性好
- 代码 B 按列访问，每次跳跃 1000 个元素，Cache 不友好

</details>

### 练习2：结构体优化

**Q：优化以下结构体的内存布局**

```go
type User struct {
    Age     int8
    ID      int64
    Active  bool
    Balance int64
}
```

<details>
<summary>查看答案</summary>

```go
// 优化前：32 字节
type User struct {
    Age     int8   // 1 + 7 填充
    ID      int64  // 8
    Active  bool   // 1 + 7 填充
    Balance int64  // 8
}

// 优化后：24 字节
type User struct {
    ID      int64  // 8
    Balance int64  // 8
    Age     int8   // 1
    Active  bool   // 1 + 6 填充
}
```

把大的字段放前面，小的放后面，减少填充。

</details>

### 练习3：虚拟内存

**Q：为什么每个程序都能使用地址 0x400000？**

<details>
<summary>查看答案</summary>

因为这是**虚拟地址**，不是物理地址。

每个程序有自己的虚拟地址空间，虚拟地址 0x400000 会被 MMU 翻译成不同的物理地址：
- 程序 A 的 0x400000 → 物理地址 0x1000000
- 程序 B 的 0x400000 → 物理地址 0x2000000

这样程序之间互不干扰。

</details>

## 7. 本章小结

```
存储层次：
寄存器 → L1 → L2 → L3 → 内存 → SSD → HDD
（越靠近 CPU 越快越小）

Cache：
├── 利用局部性原理
├── Cache Line：64 字节
├── 命中率决定性能
└── 伪共享问题

内存：
├── 内存对齐提升性能
└── 结构体字段顺序影响大小

虚拟内存：
├── 虚拟地址 → MMU → 物理地址
├── 页：4 KB
├── 页表：记录映射关系
└── 缺页中断：从磁盘加载

Go 内存：
├── 栈：快，自动回收
├── 堆：慢，需要 GC
└── 逃逸分析决定分配位置
```

**Go 开发者要点：**
1. 顺序访问数组比随机访问快很多
2. 结构体字段顺序影响内存占用
3. 减少堆分配可以降低 GC 压力
