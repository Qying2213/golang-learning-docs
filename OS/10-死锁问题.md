## 1. 什么是死锁？⭐⭐⭐

### 1.1 死锁定义

```
死锁（Deadlock）：两个或多个进程/goroutine 互相等待对方持有的资源，
导致都无法继续执行的状态。

┌─────────────────────────────────────────────────────┐
│                                                      │
│   Goroutine A          Goroutine B                  │
│   ┌─────────┐          ┌─────────┐                  │
│   │ 持有锁1  │          │ 持有锁2  │                  │
│   │ 等待锁2  │←────────→│ 等待锁1  │                  │
│   └─────────┘          └─────────┘                  │
│                                                      │
│   双方都在等待对方释放锁，永远无法继续                │
│   这就是死锁！                                       │
│                                                      │
└─────────────────────────────────────────────────────┘
```

### 1.2 死锁示例

```go
package main

import (
    "fmt"
    "sync"
    "time"
)

var (
    lockA sync.Mutex
    lockB sync.Mutex
)

func main() {
    // Goroutine 1: 先获取 A，再获取 B
    go func() {
        lockA.Lock()
        fmt.Println("Goroutine 1: 获得锁 A")
        time.Sleep(100 * time.Millisecond)  // 模拟一些操作
        
        fmt.Println("Goroutine 1: 尝试获取锁 B...")
        lockB.Lock()  // 死锁！等待 Goroutine 2 释放 B
        fmt.Println("Goroutine 1: 获得锁 B")
        lockB.Unlock()
        lockA.Unlock()
    }()
    
    // Goroutine 2: 先获取 B，再获取 A
    go func() {
        lockB.Lock()
        fmt.Println("Goroutine 2: 获得锁 B")
        time.Sleep(100 * time.Millisecond)
        
        fmt.Println("Goroutine 2: 尝试获取锁 A...")
        lockA.Lock()  // 死锁！等待 Goroutine 1 释放 A
        fmt.Println("Goroutine 2: 获得锁 A")
        lockA.Unlock()
        lockB.Unlock()
    }()
    
    // 等待（程序会卡住）
    time.Sleep(5 * time.Second)
    fmt.Println("程序结束")  // 永远不会执行到这里
}
```

**运行结果**：
```
Goroutine 1: 获得锁 A
Goroutine 2: 获得锁 B
Goroutine 1: 尝试获取锁 B...
Goroutine 2: 尝试获取锁 A...
（程序卡住，永远不会结束）
```

### 📝 练习题 1.1

**问题**：以下代码会死锁吗？为什么？

```go
func main() {
    ch := make(chan int)
    ch <- 1
    fmt.Println(<-ch)
}
```

<details>
<summary>点击查看答案</summary>

**答案**：会死锁！

**原因**：
- `ch` 是无缓冲 channel
- `ch <- 1` 会阻塞，等待接收者
- 但接收者 `<-ch` 在发送之后，永远不会执行
- 主 goroutine 自己阻塞自己

**Go 运行时会检测到这种死锁**：
```
fatal error: all goroutines are asleep - deadlock!
```

**修复方法**：
```go
// 方法1：使用缓冲 channel
ch := make(chan int, 1)
ch <- 1
fmt.Println(<-ch)

// 方法2：使用 goroutine
ch := make(chan int)
go func() { ch <- 1 }()
fmt.Println(<-ch)
```

</details>

## 2. 死锁的四个必要条件 ⭐⭐⭐

### 2.1 四个条件（缺一不可）

```
死锁发生必须同时满足以下四个条件：

┌─────────────────────────────────────────────────────┐
│  1. 互斥条件（Mutual Exclusion）                    │
│     ─────────────────────────────────────────────   │
│     资源只能被一个进程独占使用                       │
│     例：锁只能被一个 goroutine 持有                 │
├─────────────────────────────────────────────────────┤
│  2. 持有并等待（Hold and Wait）                     │
│     ─────────────────────────────────────────────   │
│     持有资源的同时等待获取其他资源                   │
│     例：持有锁 A，等待锁 B                          │
├─────────────────────────────────────────────────────┤
│  3. 不可剥夺（No Preemption）                       │
│     ─────────────────────────────────────────────   │
│     资源只能由持有者主动释放，不能被强制剥夺         │
│     例：锁不能被其他 goroutine 强制释放             │
├─────────────────────────────────────────────────────┤
│  4. 循环等待（Circular Wait）                       │
│     ─────────────────────────────────────────────   │
│     存在等待资源的环形链                            │
│     例：A 等 B，B 等 A                              │
└─────────────────────────────────────────────────────┘

打破任意一个条件，就可以预防死锁！
```

### 2.2 图示理解

```
循环等待的形成：

      ┌───────────┐
      │ Goroutine │
      │     A     │
      └─────┬─────┘
            │ 持有
            ↓
      ┌───────────┐         ┌───────────┐
      │   锁 A    │←────────│ Goroutine │
      └───────────┘  等待   │     B     │
            ↑               └─────┬─────┘
            │ 等待                │ 持有
      ┌───────────┐               ↓
      │ Goroutine │         ┌───────────┐
      │     A     │←────────│   锁 B    │
      └───────────┘  持有   └───────────┘

形成了环：A 持有锁A，等待锁B；B 持有锁B，等待锁A
```

### 📝 练习题 2.1

**问题**：分析以下场景，哪些条件满足了？

场景：两个人过独木桥，桥只能容纳一人，双方都走到桥中间不肯退让。

<details>
<summary>点击查看答案</summary>

**答案**：

1. **互斥条件** ✅：桥只能容纳一人（资源独占）
2. **持有并等待** ✅：各自占据半边桥，等待对方让路
3. **不可剥夺** ✅：不能强制把对方推下桥
4. **循环等待** ✅：A 等 B 让路，B 等 A 让路

**解决方法**：
- 破坏互斥：加宽桥（增加资源）
- 破坏持有等待：规定必须一次性通过
- 破坏不可剥夺：允许强制退让
- 破坏循环等待：规定优先级（如年长者先过）

## 3. 死锁预防 ⭐⭐⭐

### 3.1 破坏互斥条件

```go
// 使用无锁数据结构（原子操作）
package main

import (
    "fmt"
    "sync"
    "sync/atomic"
)

func main() {
    var counter int64
    var wg sync.WaitGroup
    
    // 使用原子操作，不需要锁，不会死锁
    for i := 0; i < 1000; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            atomic.AddInt64(&counter, 1)
        }()
    }
    
    wg.Wait()
    fmt.Println("Counter:", counter)
}
```

### 3.2 破坏持有并等待

```go
// 方法：一次性获取所有需要的锁
package main

import (
    "fmt"
    "sync"
)

var (
    lockA      sync.Mutex
    lockB      sync.Mutex
    globalLock sync.Mutex  // 全局锁
)

func safeOperation() {
    // 先获取全局锁，确保原子性地获取所有锁
    globalLock.Lock()
    
    lockA.Lock()
    lockB.Lock()
    
    globalLock.Unlock()  // 释放全局锁
    
    // 执行操作...
    fmt.Println("操作完成")
    
    lockB.Unlock()
    lockA.Unlock()
}

func main() {
    var wg sync.WaitGroup
    
    for i := 0; i < 10; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            safeOperation()
        }()
    }
    
    wg.Wait()
}
```

### 3.3 破坏不可剥夺（使用 TryLock）

```go
// Go 1.18+ 支持 TryLock
package main

import (
    "fmt"
    "sync"
    "time"
)

var (
    lockA sync.Mutex
    lockB sync.Mutex
)

func tryOperation() bool {
    // 尝试获取锁 A
    if !lockA.TryLock() {
        return false  // 获取失败，放弃
    }
    
    // 尝试获取锁 B
    if !lockB.TryLock() {
        lockA.Unlock()  // 释放已获取的锁 A
        return false
    }
    
    // 成功获取两把锁
    defer lockA.Unlock()
    defer lockB.Unlock()
    
    fmt.Println("操作成功")
    return true
}

func main() {
    var wg sync.WaitGroup
    
    for i := 0; i < 10; i++ {
        wg.Add(1)
        go func(id int) {
            defer wg.Done()
            
            // 重试直到成功
            for !tryOperation() {
                time.Sleep(time.Millisecond)
            }
            fmt.Printf("Goroutine %d 完成\n", id)
        }(i)
    }
    
    wg.Wait()
}
```

### 3.4 破坏循环等待（最常用）⭐⭐⭐

```go
// 方法：按固定顺序获取锁
package main

import (
    "fmt"
    "sync"
    "time"
)

var (
    lockA sync.Mutex  // 顺序 1
    lockB sync.Mutex  // 顺序 2
)

// 所有 goroutine 都按 A → B 的顺序获取锁
func operation1() {
    lockA.Lock()  // 先 A
    defer lockA.Unlock()
    
    lockB.Lock()  // 后 B
    defer lockB.Unlock()
    
    fmt.Println("operation1 完成")
}

func operation2() {
    lockA.Lock()  // 先 A（不是 B！）
    defer lockA.Unlock()
    
    lockB.Lock()  // 后 B
    defer lockB.Unlock()
    
    fmt.Println("operation2 完成")
}

func main() {
    go operation1()
    go operation2()
    
    time.Sleep(time.Second)
    fmt.Println("程序正常结束")
}
```

### 📝 练习题 3.1

**问题**：银行转账场景，如何避免死锁？

```go
type Account struct {
    id      int
    mu      sync.Mutex
    balance int
}

func transfer(from, to *Account, amount int) {
    from.mu.Lock()
    defer from.mu.Unlock()
    
    to.mu.Lock()
    defer to.mu.Unlock()
    
    from.balance -= amount
    to.balance += amount
}
```

<details>
<summary>点击查看答案</summary>

**问题分析**：
如果同时执行 `transfer(A, B, 100)` 和 `transfer(B, A, 50)`，可能死锁。

**解决方案：按账户 ID 顺序加锁**

```go
type Account struct {
    id      int
    mu      sync.Mutex
    balance int
}

func transfer(from, to *Account, amount int) {
    // 按 ID 顺序确定加锁顺序
    first, second := from, to
    if from.id > to.id {
        first, second = to, from
    }
    
    first.mu.Lock()
    defer first.mu.Unlock()
    
    second.mu.Lock()
    defer second.mu.Unlock()
    
    // 执行转账
    if from.balance >= amount {
        from.balance -= amount
        to.balance += amount
    }
}

func main() {
    accountA := &Account{id: 1, balance: 1000}
    accountB := &Account{id: 2, balance: 1000}
    
    var wg sync.WaitGroup
    
    // 并发转账
    for i := 0; i < 100; i++ {
        wg.Add(2)
        go func() {
            defer wg.Done()
            transfer(accountA, accountB, 10)
        }()
        go func() {
            defer wg.Done()
            transfer(accountB, accountA, 10)
        }()
    }
    
    wg.Wait()
    fmt.Printf("A: %d, B: %d\n", accountA.balance, accountB.balance)
}
```

## 4. Go 中常见的死锁场景 ⭐⭐⭐

### 4.1 Channel 死锁

```go
// 场景1：无缓冲 channel，没有接收者
func deadlock1() {
    ch := make(chan int)
    ch <- 1  // 永远阻塞，没有接收者
}

// 场景2：无缓冲 channel，没有发送者
func deadlock2() {
    ch := make(chan int)
    <-ch  // 永远阻塞，没有发送者
}

// 场景3：缓冲 channel 满了
func deadlock3() {
    ch := make(chan int, 2)
    ch <- 1
    ch <- 2
    ch <- 3  // 阻塞，缓冲区满了
}

// 场景4：互相等待的 channel
func deadlock4() {
    ch1 := make(chan int)
    ch2 := make(chan int)
    
    go func() {
        <-ch1     // 等待 ch1
        ch2 <- 1  // 发送到 ch2
    }()
    
    go func() {
        <-ch2     // 等待 ch2
        ch1 <- 1  // 发送到 ch1
    }()
    
    // 两个 goroutine 互相等待
    select {}
}
```

### 4.2 锁的重入死锁

```go
package main

import "sync"

var mu sync.Mutex

func funcA() {
    mu.Lock()
    defer mu.Unlock()
    
    funcB()  // 调用 funcB，但 funcB 也需要锁
}

func funcB() {
    mu.Lock()  // 死锁！同一个 goroutine 再次获取同一把锁
    defer mu.Unlock()
    
    // 操作...
}

func main() {
    funcA()  // 死锁
}
```

**解决方案**：

```go
// 方案1：使用不加锁的内部函数
func funcA() {
    mu.Lock()
    defer mu.Unlock()
    
    funcBInternal()  // 调用不加锁的版本
}

func funcB() {
    mu.Lock()
    defer mu.Unlock()
    
    funcBInternal()
}

func funcBInternal() {
    // 实际逻辑，不加锁
}

// 方案2：使用可重入锁（需要自己实现）
```

### 4.3 WaitGroup 使用错误

```go
// 错误：Add 在 goroutine 内部
func deadlock() {
    var wg sync.WaitGroup
    
    go func() {
        wg.Add(1)  // 可能在 Wait 之后执行
        defer wg.Done()
        fmt.Println("工作")
    }()
    
    wg.Wait()  // 可能在 Add 之前执行
}

// 正确：Add 在 goroutine 外部
func correct() {
    var wg sync.WaitGroup
    
    wg.Add(1)  // 先 Add
    go func() {
        defer wg.Done()
        fmt.Println("工作")
    }()
    
    wg.Wait()
}
```

## 5. 死锁检测 ⭐⭐

### 5.1 Go 运行时检测

```go
package main

func main() {
    ch := make(chan int)
    <-ch  // 死锁
}

// 运行时会检测到：
// fatal error: all goroutines are asleep - deadlock!
//
// goroutine 1 [chan receive]:
// main.main()
//     /path/main.go:5 +0x4d

**注意**：Go 只能检测所有 goroutine 都阻塞的情况，不能检测部分死锁。

    "time"

```

## 6. 避免死锁的最佳实践 ⭐⭐⭐

### 6.1 使用 Channel 代替锁

```go
package main

import "fmt"

// 使用 channel 实现线程安全的计数器
type Counter struct {
    inc   chan struct{}
    dec   chan struct{}
    get   chan int
    value int
}

func NewCounter() *Counter {
    c := &Counter{
        inc: make(chan struct{}),
        dec: make(chan struct{}),
        get: make(chan int),
    }
    
    go func() {
        for {
            select {
            case <-c.inc:
                c.value++
            case <-c.dec:
                c.value--
            case c.get <- c.value:
            }
        }
    }()
    
    return c
}

func (c *Counter) Inc()       { c.inc <- struct{}{} }
func (c *Counter) Dec()       { c.dec <- struct{}{} }
func (c *Counter) Value() int { return <-c.get }

func main() {
    counter := NewCounter()
    counter.Inc()
    counter.Inc()
    fmt.Println(counter.Value())  // 2
}
```

### 6.2 使用 Context 超时

```go
package main

import (
    "context"
    "fmt"
    "sync"
    "time"
)

func operationWithTimeout(ctx context.Context, mu *sync.Mutex) error {
    done := make(chan struct{})
    
    go func() {
        mu.Lock()
        defer mu.Unlock()
        // 执行操作...
        time.Sleep(100 * time.Millisecond)
        close(done)
    }()
    
    select {
    case <-done:
        return nil
    case <-ctx.Done():
        return ctx.Err()
    }
}

func main() {
    var mu sync.Mutex
    
    // 先锁住，模拟死锁场景
    mu.Lock()
    
    // 带超时的操作
    ctx, cancel := context.WithTimeout(context.Background(), time.Second)
    defer cancel()
    
    err := operationWithTimeout(ctx, &mu)
    if err != nil {
        fmt.Println("操作超时:", err)
    }
    
    mu.Unlock()
}
```

### 6.3 使用 select 避免阻塞

```go
package main

import (
    "fmt"
    "time"
)

func main() {
    ch1 := make(chan int)
    ch2 := make(chan int)
    
    go func() {
        time.Sleep(2 * time.Second)
        ch1 <- 1
    }()
    
    // 使用 select 避免死锁
    select {
    case v := <-ch1:
        fmt.Println("从 ch1 收到:", v)
    case v := <-ch2:
        fmt.Println("从 ch2 收到:", v)
    case <-time.After(time.Second):
        fmt.Println("超时")
    default:
        fmt.Println("没有数据")
    }
}
```

## 7. 经典问题：哲学家就餐 ⭐⭐

### 7.1 问题描述

```
5 个哲学家围坐在圆桌旁，每人面前有一盘意面。
每两个哲学家之间有一把叉子（共 5 把）。
哲学家需要同时拿起左右两把叉子才能吃面。

┌─────────────────────────────────────────────────────┐
│                                                      │
│              哲学家 0                                │
│            叉子0   叉子1                            │
│         哲学家4       哲学家1                        │
│        叉子4           叉子2                        │
│           哲学家3   哲学家2                          │
│              叉子3                                   │
│                                                      │
│  如果每个哲学家都拿起左边的叉子，等待右边的叉子      │
│  → 死锁！                                           │
│                                                      │
└─────────────────────────────────────────────────────┘
```

### 7.2 解决方案

```go
package main

import (
    "fmt"
    "sync"
    "time"
)

type Fork struct {
    id int
    mu sync.Mutex
}

type Philosopher struct {
    id        int
    leftFork  *Fork
    rightFork *Fork
}

func (p *Philosopher) dine(wg *sync.WaitGroup) {
    defer wg.Done()
    
    for i := 0; i < 3; i++ {
        p.think()
        p.eat()
    }
}

func (p *Philosopher) think() {
    fmt.Printf("哲学家 %d 正在思考\n", p.id)
    time.Sleep(100 * time.Millisecond)
}

func (p *Philosopher) eat() {
    // 关键：按叉子 ID 顺序获取，避免死锁
    first, second := p.leftFork, p.rightFork
    if p.leftFork.id > p.rightFork.id {
        first, second = p.rightFork, p.leftFork
    }
    
    first.mu.Lock()
    second.mu.Lock()
    
    fmt.Printf("哲学家 %d 正在吃饭（叉子 %d 和 %d）\n", 
        p.id, p.leftFork.id, p.rightFork.id)
    time.Sleep(100 * time.Millisecond)
    
    second.mu.Unlock()
    first.mu.Unlock()
}

func main() {
    n := 5
    forks := make([]*Fork, n)
    for i := 0; i < n; i++ {
        forks[i] = &Fork{id: i}
    }
    
    philosophers := make([]*Philosopher, n)
    for i := 0; i < n; i++ {
        philosophers[i] = &Philosopher{
            id:        i,
            leftFork:  forks[i],
            rightFork: forks[(i+1)%n],
        }
    }
    
    var wg sync.WaitGroup
    for _, p := range philosophers {
        wg.Add(1)
        go p.dine(&wg)
    }
    
    wg.Wait()
    fmt.Println("所有哲学家都吃完了")
}
```

## 8. 面试常见问题 ⭐⭐⭐

**Q1: 死锁的四个必要条件是什么？**

<details>
<summary>点击查看答案</summary>

1. **互斥条件**：资源只能被一个进程独占
2. **持有并等待**：持有资源的同时等待其他资源
3. **不可剥夺**：资源只能由持有者主动释放
4. **循环等待**：存在等待资源的环形链

打破任意一个条件即可预防死锁。

</details>

**Q2: 如何预防死锁？**

<details>
<summary>点击查看答案</summary>

1. **破坏互斥**：使用无锁数据结构（原子操作）
2. **破坏持有等待**：一次性获取所有锁
3. **破坏不可剥夺**：使用 TryLock，获取失败则释放已有锁
4. **破坏循环等待**：按固定顺序获取锁（最常用）

</details>

**Q3: Go 中如何检测死锁？**

<details>
<summary>点击查看答案</summary>

1. **运行时检测**：Go 会检测所有 goroutine 都阻塞的情况
2. **pprof**：查看 goroutine 状态
3. **go-deadlock 库**：检测潜在死锁
4. **-race 标志**：检测数据竞争（不是死锁，但相关）

</details>

## 9. 本章小结

### 死锁四个条件

| 条件 | 说明 | 破坏方法 |
|------|------|---------|
| 互斥 | 资源独占 | 无锁数据结构 |
| 持有等待 | 持有并等待 | 一次性获取 |
| 不可剥夺 | 不能强制释放 | TryLock |
| 循环等待 | 环形等待链 | 按顺序加锁 |

### 最佳实践

1. 优先使用 channel
2. 按固定顺序获取锁
3. 使用 context 超时
4. 使用 select 避免阻塞
5. 使用 pprof 监控
