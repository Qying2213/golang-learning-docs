## 1. 并发问题 ⭐⭐⭐

### 1.1 竞态条件（Race Condition）

```go
package main

import (
    "fmt"
    "sync"
)

var counter = 0

func main() {
    var wg sync.WaitGroup
    
    // 启动 1000 个 goroutine 同时增加 counter
    for i := 0; i < 1000; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            counter++  // 竞态条件！
        }()
    }
    
    wg.Wait()
    fmt.Println("Counter:", counter)  // 结果不确定，可能小于 1000
}
```

**为什么会出问题？**

```
counter++ 实际上是三步操作：

goroutine A              goroutine B              counter 值
─────────────────────────────────────────────────────────────
1. 读取 counter (0)                                   0
                         1. 读取 counter (0)          0
2. 加 1 得到 1                                        0
                         2. 加 1 得到 1               0
3. 写回 counter                                       1
                         3. 写回 counter              1

结果：两次 ++ 只增加了 1！

这就是"丢失更新"问题。
```

### 📝 练习题 1.1

**问题**：以下代码有竞态条件吗？如果有，如何修复？

```go
var config map[string]string

func init() {
    config = make(map[string]string)
    config["key"] = "value"
}

func GetConfig(key string) string {
    return config[key]
}

func SetConfig(key, value string) {
    config[key] = value
}
```

<details>
<summary>点击查看答案</summary>

**答案**：有竞态条件！

**问题分析**：
- map 不是并发安全的
- 多个 goroutine 同时读写 map 会导致 panic
- 即使只是读取，如果有其他 goroutine 在写入，也会出问题

**修复方法**：

```go
var (
    config   map[string]string
    configMu sync.RWMutex
)

func init() {
    config = make(map[string]string)
    config["key"] = "value"
}

func GetConfig(key string) string {
    configMu.RLock()
    defer configMu.RUnlock()
    return config[key]
}

func SetConfig(key, value string) {
    configMu.Lock()
    defer configMu.Unlock()
    config[key] = value
}
```

或者使用 `sync.Map`：

```go
var config sync.Map

func init() {
    config.Store("key", "value")
}

func GetConfig(key string) string {
    if v, ok := config.Load(key); ok {
        return v.(string)
    }
    return ""
}

func SetConfig(key, value string) {
    config.Store(key, value)
}
```

</details>

## 2. 互斥锁（Mutex）⭐⭐⭐

### 2.1 基本使用

```go
package main

import (
    "fmt"
    "sync"
)

var (
    counter = 0
    mu      sync.Mutex
)

func main() {
    var wg sync.WaitGroup
    
    for i := 0; i < 1000; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            
            mu.Lock()         // 获取锁
            counter++         // 临界区
            mu.Unlock()       // 释放锁
        }()
    }
    
    wg.Wait()
    fmt.Println("Counter:", counter)  // 一定是 1000
}
```

### 2.2 使用 defer 确保解锁

```go
func safeIncrement() {
    mu.Lock()
    defer mu.Unlock()  // 确保一定会解锁，即使 panic
    
    counter++
    
    // 即使这里 panic，也会解锁
    if counter > 100 {
        panic("too large")
    }
}
```

### 2.3 封装到结构体（推荐做法）

```go
package main

import (
    "fmt"
    "sync"
)

// SafeCounter 线程安全的计数器
type SafeCounter struct {
    mu    sync.Mutex
    value int
}

// Inc 增加计数
func (c *SafeCounter) Inc() {
    c.mu.Lock()
    defer c.mu.Unlock()
    c.value++
}

// Dec 减少计数
func (c *SafeCounter) Dec() {
    c.mu.Lock()
    defer c.mu.Unlock()
    c.value--
}

// Value 获取当前值
func (c *SafeCounter) Value() int {
    c.mu.Lock()
    defer c.mu.Unlock()
    return c.value
}

// Add 增加指定值
func (c *SafeCounter) Add(delta int) {
    c.mu.Lock()
    defer c.mu.Unlock()
    c.value += delta
}

func main() {
    counter := &SafeCounter{}
    
    var wg sync.WaitGroup
    for i := 0; i < 1000; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            counter.Inc()
        }()
    }
    
    wg.Wait()
    fmt.Println("Counter:", counter.Value())  // 1000
}
```

### 2.4 Mutex 的注意事项

```go
// ⚠️ 注意事项

// 1. 不要复制 Mutex
type Bad struct {
    mu sync.Mutex
}

func (b Bad) Method() {  // ❌ 值接收者会复制 Mutex
    b.mu.Lock()
    defer b.mu.Unlock()
}

func (b *Bad) Method() {  // ✅ 指针接收者
    b.mu.Lock()
    defer b.mu.Unlock()
}

// 2. 不要重复加锁（会死锁）
func bad() {
    mu.Lock()
    mu.Lock()  // ❌ 死锁！
    mu.Unlock()
    mu.Unlock()
}

// 3. 不要在持有锁时进行耗时操作
func bad2() {
    mu.Lock()
    time.Sleep(time.Second)  // ❌ 阻塞其他 goroutine
    mu.Unlock()
}

// 4. 锁的粒度要适当
// 太粗：并发度低
// 太细：锁开销大，容易死锁
```

### 📝 练习题 2.1

**问题**：实现一个线程安全的栈（Stack）。

<details>
<summary>点击查看答案</summary>

```go
package main

import (
    "errors"
    "fmt"
    "sync"
)

type Stack struct {
    mu    sync.Mutex
    items []interface{}
}

func NewStack() *Stack {
    return &Stack{
        items: make([]interface{}, 0),
    }
}

func (s *Stack) Push(item interface{}) {
    s.mu.Lock()
    defer s.mu.Unlock()
    s.items = append(s.items, item)
}

func (s *Stack) Pop() (interface{}, error) {
    s.mu.Lock()
    defer s.mu.Unlock()
    
    if len(s.items) == 0 {
        return nil, errors.New("stack is empty")
    }
    
    item := s.items[len(s.items)-1]
    s.items = s.items[:len(s.items)-1]
    return item, nil
}

func (s *Stack) Peek() (interface{}, error) {
    s.mu.Lock()
    defer s.mu.Unlock()
    
    if len(s.items) == 0 {
        return nil, errors.New("stack is empty")
    }
    
    return s.items[len(s.items)-1], nil
}

func (s *Stack) Len() int {
    s.mu.Lock()
    defer s.mu.Unlock()
    return len(s.items)
}

func (s *Stack) IsEmpty() bool {
    return s.Len() == 0
}

func main() {
    stack := NewStack()
    
    var wg sync.WaitGroup
    
    // 并发 Push
    for i := 0; i < 100; i++ {
        wg.Add(1)
        go func(n int) {
            defer wg.Done()
            stack.Push(n)
        }(i)
    }
    
    wg.Wait()
    fmt.Println("Stack length:", stack.Len())  // 100
    
    // 并发 Pop
    for i := 0; i < 50; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            stack.Pop()
        }()
    }
    
    wg.Wait()
    fmt.Println("Stack length after pop:", stack.Len())  // 50
}
```

## 3. 读写锁（RWMutex）⭐⭐⭐

### 3.1 为什么需要读写锁？

```
场景：读多写少的数据结构（如配置、缓存）

使用 Mutex 的问题：
┌─────────────────────────────────────────────────────┐
│  Reader 1: Lock() → 读取 → Unlock()                 │
│  Reader 2: Lock() → 等待...                         │
│  Reader 3: Lock() → 等待...                         │
│                                                      │
│  问题：读操作也要互斥，多个读不能并发               │
└─────────────────────────────────────────────────────┘

使用 RWMutex：
┌─────────────────────────────────────────────────────┐
│  Reader 1: RLock() → 读取 → RUnlock()               │
│  Reader 2: RLock() → 读取 → RUnlock()  ← 可以并发！ │
│  Reader 3: RLock() → 读取 → RUnlock()  ← 可以并发！ │
│                                                      │
│  Writer:   Lock() → 等待所有读完成 → 写入 → Unlock()│
└─────────────────────────────────────────────────────┘
```

### 3.2 读写锁规则

```
┌─────────────────────────────────────────────────────┐
│                   读写锁规则                         │
├─────────────────────────────────────────────────────┤
│  当前状态        │  读锁请求  │  写锁请求           │
├─────────────────────────────────────────────────────┤
│  无锁            │  ✅ 允许   │  ✅ 允许            │
│  有读锁          │  ✅ 允许   │  ❌ 阻塞            │
│  有写锁          │  ❌ 阻塞   │  ❌ 阻塞            │
└─────────────────────────────────────────────────────┘

总结：
- 读读不互斥（多个读可以并发）
- 读写互斥
- 写写互斥
- 写锁优先（防止写饥饿）
```

### 3.3 基本使用

```go
package main

import (
    "fmt"
    "sync"
    "time"
)

type Cache struct {
    mu   sync.RWMutex
    data map[string]string
}

func NewCache() *Cache {
    return &Cache{
        data: make(map[string]string),
    }
}

// Get 读操作：使用读锁
func (c *Cache) Get(key string) (string, bool) {
    c.mu.RLock()
    defer c.mu.RUnlock()
    
    value, ok := c.data[key]
    return value, ok
}

// Set 写操作：使用写锁
func (c *Cache) Set(key, value string) {
    c.mu.Lock()
    defer c.mu.Unlock()
    
    c.data[key] = value
}

// Delete 写操作：使用写锁
func (c *Cache) Delete(key string) {
    c.mu.Lock()
    defer c.mu.Unlock()
    
    delete(c.data, key)
}

// Len 读操作：使用读锁
func (c *Cache) Len() int {
    c.mu.RLock()
    defer c.mu.RUnlock()
    
    return len(c.data)
}

// GetAll 读操作：使用读锁
func (c *Cache) GetAll() map[string]string {
    c.mu.RLock()
    defer c.mu.RUnlock()
    
    // 返回副本，避免外部修改
    result := make(map[string]string, len(c.data))
    for k, v := range c.data {
        result[k] = v
    }
    return result
}

func main() {
    cache := NewCache()
    
    // 写入一些数据
    cache.Set("name", "Go")
    cache.Set("version", "1.21")
    
    var wg sync.WaitGroup
    
    // 启动多个读 goroutine
    for i := 0; i < 10; i++ {
        wg.Add(1)
        go func(id int) {
            defer wg.Done()
            for j := 0; j < 100; j++ {
                if value, ok := cache.Get("name"); ok {
                    _ = value
                }
            }
            fmt.Printf("Reader %d done\n", id)
        }(i)
    }
    
    // 启动写 goroutine
    wg.Add(1)
    go func() {
        defer wg.Done()
        for i := 0; i < 10; i++ {
            cache.Set("counter", fmt.Sprintf("%d", i))
            time.Sleep(10 * time.Millisecond)
        }
        fmt.Println("Writer done")
    }()
    
    wg.Wait()
    fmt.Println("Cache size:", cache.Len())
}
```

### 📝 练习题 3.1

**问题**：什么时候应该用 Mutex，什么时候应该用 RWMutex？

<details>
<summary>点击查看答案</summary>

**使用 Mutex 的场景**：
1. 读写比例接近（如 1:1）
2. 临界区很短（锁开销占比大）
3. 写操作频繁
4. 代码简单，不想引入复杂性

**使用 RWMutex 的场景**：
1. 读多写少（如 10:1 或更高）
2. 临界区较长（读操作耗时）
3. 需要高并发读取
4. 典型场景：配置读取、缓存查询

**注意**：
- RWMutex 比 Mutex 开销大
- 如果读写比例不够高，RWMutex 可能更慢
- 建议通过 benchmark 测试决定

**经验法则**：
- 读写比 > 10:1 → 考虑 RWMutex
- 读写比 < 10:1 → 使用 Mutex

</details>

## 4. 原子操作 ⭐⭐⭐

### 4.1 什么是原子操作？

```
原子操作：不可分割的操作，要么完全执行，要么完全不执行

普通操作（非原子）：
counter++  →  读取 → 加1 → 写回  （三步，可能被打断）

原子操作：
atomic.AddInt64(&counter, 1)  →  一步完成，不会被打断

优点：
- 比锁更快（无锁）
- 不会死锁
- 适合简单操作

缺点：
- 只能操作简单类型
- 不能保护复杂的临界区
```

### 4.2 常用原子操作

```go
package main

import (
    "fmt"
    "sync/atomic"
)

func main() {
    var value int64 = 0
    
    // 1. 加法
    atomic.AddInt64(&value, 10)
    fmt.Println("Add 10:", value)  // 10
    
    // 2. 减法（加负数）
    atomic.AddInt64(&value, -5)
    fmt.Println("Sub 5:", value)  // 5
    
    // 3. 读取
    v := atomic.LoadInt64(&value)
    fmt.Println("Load:", v)  // 5
    
    // 4. 存储
    atomic.StoreInt64(&value, 100)
    fmt.Println("Store 100:", value)  // 100
    
    // 5. 交换（返回旧值）
    old := atomic.SwapInt64(&value, 200)
    fmt.Println("Swap: old=", old, "new=", value)  // old=100 new=200
    
    // 6. 比较并交换（CAS）
    // 如果 value == 200，则设置为 300
    swapped := atomic.CompareAndSwapInt64(&value, 200, 300)
    fmt.Println("CAS success:", swapped, "value:", value)  // true, 300
    
    // CAS 失败的情况
    swapped = atomic.CompareAndSwapInt64(&value, 200, 400)
    fmt.Println("CAS success:", swapped, "value:", value)  // false, 300
}
```

### 4.3 atomic.Value

```go
package main

import (
    "fmt"
    "sync/atomic"
    "time"
)

type Config struct {
    Server   string
    Port     int
    Timeout  time.Duration
}

func main() {
    var config atomic.Value
    
    // 存储配置
    config.Store(&Config{
        Server:  "localhost",
        Port:    8080,
        Timeout: 30 * time.Second,
    })
    
    // 读取配置（无锁）
    cfg := config.Load().(*Config)
    fmt.Printf("Server: %s, Port: %d\n", cfg.Server, cfg.Port)
    
    // 更新配置（原子替换）
    config.Store(&Config{
        Server:  "127.0.0.1",
        Port:    9090,
        Timeout: 60 * time.Second,
    })
    
    // 再次读取
    cfg = config.Load().(*Config)
    fmt.Printf("Server: %s, Port: %d\n", cfg.Server, cfg.Port)
}
```

### 4.4 原子计数器实现

```go
package main

import (
    "fmt"
    "sync"
    "sync/atomic"
)

// AtomicCounter 使用原子操作的计数器
type AtomicCounter struct {
    value int64
}

func (c *AtomicCounter) Inc() {
    atomic.AddInt64(&c.value, 1)
}

func (c *AtomicCounter) Dec() {
    atomic.AddInt64(&c.value, -1)
}

func (c *AtomicCounter) Add(delta int64) {
    atomic.AddInt64(&c.value, delta)
}

func (c *AtomicCounter) Value() int64 {
    return atomic.LoadInt64(&c.value)
}

func (c *AtomicCounter) Reset() {
    atomic.StoreInt64(&c.value, 0)
}

func main() {
    counter := &AtomicCounter{}
    
    var wg sync.WaitGroup
    
    // 并发增加
    for i := 0; i < 1000; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            counter.Inc()
        }()
    }
    
    wg.Wait()
    fmt.Println("Counter:", counter.Value())  // 1000
}
```

### 4.5 原子操作 vs 锁

| 特性 | 原子操作 | 锁 |
|------|---------|-----|
| 性能 | 更快 | 较慢 |
| 复杂度 | 简单操作 | 复杂操作 |
| 死锁 | 不会 | 可能 |
| 适用类型 | 基本类型 | 任意类型 |
| 临界区 | 单个操作 | 多个操作 |

### 📝 练习题 4.1

**问题**：使用原子操作实现一个自旋锁。

<details>
<summary>点击查看答案</summary>

```go
package main

import (
    "fmt"
    "runtime"
    "sync"
    "sync/atomic"
)

// SpinLock 自旋锁
type SpinLock struct {
    locked int32
}

func (l *SpinLock) Lock() {
    for !atomic.CompareAndSwapInt32(&l.locked, 0, 1) {
        // 自旋等待
        runtime.Gosched()  // 让出 CPU，避免忙等
    }
}

func (l *SpinLock) Unlock() {
    atomic.StoreInt32(&l.locked, 0)
}

func (l *SpinLock) TryLock() bool {
    return atomic.CompareAndSwapInt32(&l.locked, 0, 1)
}

func main() {
    var lock SpinLock
    counter := 0
    
    var wg sync.WaitGroup
    for i := 0; i < 1000; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            lock.Lock()
            counter++
            lock.Unlock()
        }()
    }
    
    wg.Wait()
    fmt.Println("Counter:", counter)  // 1000
}
```

**注意**：
- 自旋锁适合临界区很短的场景
- 长时间持有会浪费 CPU
- Go 的 sync.Mutex 内部也使用了自旋优化

</details>

## 5. sync.Once ⭐⭐

### 5.1 单例模式

```go
package main

import (
    "fmt"
    "sync"
)

type Database struct {
    Name string
}

var (
    db   *Database
    once sync.Once
)

// GetDB 获取数据库单例
func GetDB() *Database {
    once.Do(func() {
        fmt.Println("初始化数据库连接...")  // 只会执行一次
        db = &Database{Name: "MySQL"}
    })
    return db
}

func main() {
    var wg sync.WaitGroup
    
    // 并发调用 GetDB
    for i := 0; i < 10; i++ {
        wg.Add(1)
        go func(id int) {
            defer wg.Done()
            db := GetDB()
            fmt.Printf("Goroutine %d: %s\n", id, db.Name)
        }(i)
    }
    
    wg.Wait()
    // "初始化数据库连接..." 只会打印一次
}
```

### 5.2 懒加载

```go
package main

import (
    "fmt"
    "sync"
    "time"
)

type ExpensiveResource struct {
    data []byte
}

type LazyLoader struct {
    once     sync.Once
    resource *ExpensiveResource
    err      error
}

func (l *LazyLoader) Get() (*ExpensiveResource, error) {
    l.once.Do(func() {
        fmt.Println("加载资源（耗时操作）...")
        time.Sleep(time.Second)  // 模拟耗时
        l.resource = &ExpensiveResource{
            data: make([]byte, 1024*1024),  // 1MB
        }
        // 如果加载失败，设置 err
        // l.err = errors.New("load failed")
    })
    return l.resource, l.err
}

func main() {
    loader := &LazyLoader{}
    
    fmt.Println("程序启动，资源未加载")
    
    // 第一次调用才会加载
    start := time.Now()
    r, _ := loader.Get()
    fmt.Printf("第一次获取: %d bytes, 耗时: %v\n", len(r.data), time.Since(start))
    
    // 后续调用直接返回
    start = time.Now()
    r, _ = loader.Get()
    fmt.Printf("第二次获取: %d bytes, 耗时: %v\n", len(r.data), time.Since(start))
}
```

### 5.3 Once 的注意事项

```go
// ⚠️ 注意事项

// 1. Do 中的函数 panic 了，Once 仍然认为已执行
var once sync.Once
once.Do(func() {
    panic("oops")  // panic 后，once 不会再执行
})
once.Do(func() {
    fmt.Println("不会执行")
})

// 2. 不要在 Do 中调用 Do（死锁）
var once2 sync.Once
once2.Do(func() {
    once2.Do(func() {  // ❌ 死锁
        fmt.Println("never")
    })
})

// 3. Once 不能重置
// 如果需要重置，使用 atomic.Value 或自定义实现
```

## 6. sync.Cond ⭐⭐

### 6.1 条件变量

```go
package main

import (
    "fmt"
    "sync"
    "time"
)

func main() {
    var mu sync.Mutex
    cond := sync.NewCond(&mu)
    
    ready := false
    
    // 等待者
    go func() {
        mu.Lock()
        for !ready {  // 使用 for 循环检查条件
            fmt.Println("等待条件满足...")
            cond.Wait()  // 释放锁并等待
        }
        fmt.Println("条件满足，继续执行")
        mu.Unlock()
    }()
    
    // 通知者
    time.Sleep(time.Second)
    
    mu.Lock()
    ready = true
    cond.Signal()  // 唤醒一个等待者
    mu.Unlock()
    
    time.Sleep(time.Second)
}
```

### 6.2 生产者-消费者模式

```go
package main

import (
    "fmt"
    "sync"
    "time"
)

type Queue struct {
    mu    sync.Mutex
    cond  *sync.Cond
    items []int
    max   int
}

func NewQueue(max int) *Queue {
    q := &Queue{max: max, items: make([]int, 0)}
    q.cond = sync.NewCond(&q.mu)
    return q
}

// Put 生产者放入数据
func (q *Queue) Put(item int) {
    q.mu.Lock()
    defer q.mu.Unlock()
    
    // 队列满时等待
    for len(q.items) >= q.max {
        q.cond.Wait()
    }
    
    q.items = append(q.items, item)
    q.cond.Broadcast()  // 通知所有等待者
}

// Get 消费者获取数据
func (q *Queue) Get() int {
    q.mu.Lock()
    defer q.mu.Unlock()
    
    // 队列空时等待
    for len(q.items) == 0 {
        q.cond.Wait()
    }
    
    item := q.items[0]
    q.items = q.items[1:]
    q.cond.Broadcast()
    return item
}

func main() {
    queue := NewQueue(5)
    
    // 生产者
    go func() {
        for i := 0; i < 10; i++ {
            queue.Put(i)
            fmt.Println("生产:", i)
            time.Sleep(100 * time.Millisecond)
        }
    }()
    
    // 消费者
    go func() {
        for i := 0; i < 10; i++ {
            item := queue.Get()
            fmt.Println("消费:", item)
            time.Sleep(200 * time.Millisecond)
        }
    }()
    
    time.Sleep(3 * time.Second)
}
```

### 📝 练习题 6.1

**问题**：为什么 cond.Wait() 要在 for 循环中调用，而不是 if？

<details>
<summary>点击查看答案</summary>

**答案**：

必须使用 for 循环的原因：

1. **虚假唤醒（Spurious Wakeup）**：
   - 操作系统可能在没有 Signal/Broadcast 的情况下唤醒等待者
   - 这是底层实现的特性，不是 bug

2. **条件可能已改变**：
   - 被唤醒时，其他 goroutine 可能已经改变了条件
   - 例如：多个消费者被唤醒，但只有一个数据

3. **Broadcast 唤醒所有等待者**：
   - 但可能只有部分等待者的条件满足
   - 其他等待者需要继续等待

**错误示例**：
```go
// ❌ 错误：使用 if
if !ready {
    cond.Wait()
}
// 被唤醒后直接继续，但 ready 可能仍然是 false

// ✅ 正确：使用 for
for !ready {
    cond.Wait()
}
// 被唤醒后重新检查条件
```

</details>

## 7. sync.Map ⭐⭐

### 7.1 并发安全的 Map

```go
package main

import (
    "fmt"
    "sync"
)

func main() {
    var m sync.Map
    
    // 存储
    m.Store("name", "Go")
    m.Store("version", "1.21")
    m.Store("count", 100)
    
    // 读取
    if value, ok := m.Load("name"); ok {
        fmt.Println("name:", value)
    }
    
    // 读取或存储（如果不存在则存储）
    actual, loaded := m.LoadOrStore("name", "Golang")
    fmt.Printf("LoadOrStore: actual=%v, loaded=%v\n", actual, loaded)
    // actual=Go, loaded=true（已存在，返回旧值）
    
    actual, loaded = m.LoadOrStore("new_key", "new_value")
    fmt.Printf("LoadOrStore: actual=%v, loaded=%v\n", actual, loaded)
    // actual=new_value, loaded=false（不存在，存储新值）
    
    // 删除
    m.Delete("version")
    
    // 读取并删除
    value, loaded := m.LoadAndDelete("count")
    fmt.Printf("LoadAndDelete: value=%v, loaded=%v\n", value, loaded)
    
    // 遍历
    fmt.Println("\n遍历:")
    m.Range(func(key, value interface{}) bool {
        fmt.Printf("  %v: %v\n", key, value)
        return true  // 返回 false 停止遍历
    })
}
```

### 7.2 sync.Map vs map + Mutex

```
sync.Map 适合的场景：
┌─────────────────────────────────────────────────────┐
│  1. 读多写少                                        │

map + Mutex 适合的场景：

│  2. 需要频繁遍历                                    │
```

### 7.3 实际应用：缓存

```go
package main

import (
    "fmt"
    "sync"
    "time"
)

type CacheItem struct {
    Value     interface{}
    ExpireAt  time.Time
}

type Cache struct {
    data sync.Map
}

func NewCache() *Cache {
    c := &Cache{}
    // 启动清理过期项的 goroutine
    go c.cleanup()
    return c
}

func (c *Cache) Set(key string, value interface{}, ttl time.Duration) {
    c.data.Store(key, &CacheItem{
        Value:    value,
        ExpireAt: time.Now().Add(ttl),
    })
}

func (c *Cache) Get(key string) (interface{}, bool) {
    if v, ok := c.data.Load(key); ok {
        item := v.(*CacheItem)
        if time.Now().Before(item.ExpireAt) {
            return item.Value, true
        }
        // 已过期，删除
        c.data.Delete(key)
    }
    return nil, false
}

func (c *Cache) Delete(key string) {
    c.data.Delete(key)
}

func (c *Cache) cleanup() {
    ticker := time.NewTicker(time.Minute)
    for range ticker.C {
        c.data.Range(func(key, value interface{}) bool {
            item := value.(*CacheItem)
            if time.Now().After(item.ExpireAt) {
                c.data.Delete(key)
            }
            return true
        })
    }
}

func main() {
    cache := NewCache()
    
    cache.Set("user:1", "Alice", 5*time.Second)
    
    if v, ok := cache.Get("user:1"); ok {
        fmt.Println("Found:", v)
    }
    
    time.Sleep(6 * time.Second)
    
    if _, ok := cache.Get("user:1"); !ok {
        fmt.Println("Expired")
    }
}
```

## 8. 实战：线程安全的数据结构 ⭐⭐⭐

### 8.1 线程安全的队列

```go
package main

import (
    "fmt"
    "sync"
)

type SafeQueue struct {
    mu    sync.Mutex
    items []interface{}
}

func NewSafeQueue() *SafeQueue {
    return &SafeQueue{
        items: make([]interface{}, 0),
    }
}

func (q *SafeQueue) Enqueue(item interface{}) {
    q.mu.Lock()
    defer q.mu.Unlock()
    q.items = append(q.items, item)
}

func (q *SafeQueue) Dequeue() (interface{}, bool) {
    q.mu.Lock()
    defer q.mu.Unlock()
    
    if len(q.items) == 0 {
        return nil, false
    }
    
    item := q.items[0]
    q.items = q.items[1:]
    return item, true
}

func (q *SafeQueue) Len() int {
    q.mu.Lock()
    defer q.mu.Unlock()
    return len(q.items)
}
```

### 8.2 线程安全的 LRU 缓存

```go
package main

import (
    "container/list"
    "sync"
)

type LRUCache struct {
    mu       sync.Mutex
    capacity int
    cache    map[interface{}]*list.Element
    list     *list.List
}

type entry struct {
    key   interface{}
    value interface{}
}

func NewLRUCache(capacity int) *LRUCache {
    return &LRUCache{
        capacity: capacity,
        cache:    make(map[interface{}]*list.Element),
        list:     list.New(),
    }
}

func (c *LRUCache) Get(key interface{}) (interface{}, bool) {
    c.mu.Lock()
    defer c.mu.Unlock()
    
    if elem, ok := c.cache[key]; ok {
        c.list.MoveToFront(elem)
        return elem.Value.(*entry).value, true
    }
    return nil, false
}

func (c *LRUCache) Put(key, value interface{}) {
    c.mu.Lock()
    defer c.mu.Unlock()
    
    if elem, ok := c.cache[key]; ok {
        c.list.MoveToFront(elem)
        elem.Value.(*entry).value = value
        return
    }
    
    if c.list.Len() >= c.capacity {
        oldest := c.list.Back()
        if oldest != nil {
            c.list.Remove(oldest)
            delete(c.cache, oldest.Value.(*entry).key)
        }
    }
    
    elem := c.list.PushFront(&entry{key, value})
    c.cache[key] = elem
}
```

## 9. 面试常见问题 ⭐⭐⭐

**Q1: Mutex 和 RWMutex 的区别？什么时候用哪个？**

<details>
<summary>点击查看答案</summary>

**Mutex**：
- 互斥锁，同一时刻只有一个 goroutine 能持有
- 适合读写均衡的场景

**RWMutex**：
- 读写锁，允许多个读或一个写
- 适合读多写少的场景（读写比 > 10:1）

**选择建议**：
- 默认使用 Mutex（简单）
- 读写比很高时考虑 RWMutex
- 通过 benchmark 验证

</details>

**Q2: 原子操作和锁有什么区别？**

<details>
<summary>点击查看答案</summary>

| 特性 | 原子操作 | 锁 |
|------|---------|-----|
| 性能 | 更快（无锁） | 较慢 |
| 适用范围 | 简单类型 | 任意类型 |
| 临界区 | 单个操作 | 多个操作 |
| 死锁 | 不会 | 可能 |

**使用建议**：
- 简单计数器 → 原子操作
- 复杂数据结构 → 锁

</details>

**Q3: sync.Map 和 map + Mutex 怎么选？**

<details>
<summary>点击查看答案</summary>

**sync.Map 适合**：
- 读多写少
- 不同 goroutine 操作不同 key
- 不需要遍历

**map + Mutex 适合**：
- 读写均衡
- 需要遍历
- 需要获取长度
- 需要批量操作

</details>

## 10. 本章小结

### 同步原语

| 类型 | 用途 | 特点 |
|------|------|------|
| sync.Mutex | 互斥锁 | 独占访问 |
| sync.RWMutex | 读写锁 | 读共享，写独占 |
| sync/atomic | 原子操作 | 无锁，高性能 |
| sync.Once | 单次执行 | 初始化 |
| sync.Cond | 条件变量 | 等待/通知 |
| sync.Map | 并发 Map | 读多写少 |

### 最佳实践

1. 优先使用 channel 通信
2. 简单计数用原子操作
3. 读多写少用 RWMutex
4. 复杂操作用 Mutex
5. 使用 defer 确保解锁
6. 使用 -race 检测竞态
