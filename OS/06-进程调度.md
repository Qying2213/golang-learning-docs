# 进程调度

> **重要程度：⭐⭐ 重要概念**  
> 理解调度算法，有助于理解 Go 调度器和性能优化

## 1. 什么是进程调度？

### 1.1 为什么需要调度？

```
问题：CPU 只有 4 核，但有 100 个进程要运行

┌─────────────────────────────────────────────────────┐
│                    CPU（4核）                        │
│  ┌─────┐  ┌─────┐  ┌─────┐  ┌─────┐               │
│  │ 核1 │  │ 核2 │  │ 核3 │  │ 核4 │               │
│  └─────┘  └─────┘  └─────┘  └─────┘               │
└─────────────────────────────────────────────────────┘
                      ↑
                   调度器
                      ↑
┌─────────────────────────────────────────────────────┐
│              就绪队列（100个进程）                    │
│  P1  P2  P3  P4  P5  P6  ...  P100                  │
└─────────────────────────────────────────────────────┘

调度器的任务：决定哪个进程获得 CPU
```

### 1.2 调度的目标

| 目标 | 说明 |
|------|------|
| 公平性 | 每个进程都能获得 CPU 时间 |
| 效率 | CPU 利用率高 |
| 响应时间 | 交互式任务快速响应 |
| 吞吐量 | 单位时间完成更多任务 |
| 周转时间 | 任务从提交到完成的时间短 |

## 2. 调度算法 ⭐⭐⭐

### 2.1 先来先服务（FCFS）

```
按到达顺序执行，不可抢占

时间线：
P1 ████████████████████ (20ms)
P2                     ████████ (8ms)
P3                             ████ (4ms)

特点：
- 简单
- 对短任务不友好（等待时间长）
- 不适合交互式系统
```

### 2.2 短作业优先（SJF）

```
优先执行运行时间短的任务

时间线：
P3 ████ (4ms)
P2     ████████ (8ms)
P1             ████████████████████ (20ms)

特点：
- 平均等待时间最短
- 需要预知运行时间
- 可能导致长任务饥饿
```

### 2.3 时间片轮转（RR）

```
每个进程执行一个时间片，然后切换

时间片 = 4ms

时间线：
P1 ████
P2     ████
P3         ████
P1             ████
P2                 ████
P1                     ████
...

特点：
- 公平
- 响应时间好
- 时间片太小：切换开销大
- 时间片太大：退化为 FCFS
```

### 2.4 优先级调度

```
按优先级执行，高优先级先运行

优先级：P1=3, P2=1, P3=2（数字小优先级高）

执行顺序：P2 → P3 → P1

特点：
- 重要任务优先
- 可能导致低优先级饥饿
- 解决方案：老化（随时间提升优先级）
```

### 2.5 多级反馈队列（MLFQ）

```
┌─────────────────────────────────────────────────────┐
│  队列0（最高优先级，时间片=2ms）                     │
│  ┌───┐ ┌───┐                                       │
│  │ P1│ │ P2│  ← 新进程进入这里                     │
│  └───┘ └───┘                                       │
├─────────────────────────────────────────────────────┤
│  队列1（中优先级，时间片=4ms）                       │
│  ┌───┐ ┌───┐                                       │
│  │ P3│ │ P4│  ← 用完时间片降级到这里               │
│  └───┘ └───┘                                       │
├─────────────────────────────────────────────────────┤
│  队列2（低优先级，时间片=8ms）                       │
│  ┌───┐ ┌───┐                                       │
│  │ P5│ │ P6│  ← CPU 密集型任务最终在这里           │
│  └───┘ └───┘                                       │
└─────────────────────────────────────────────────────┘

规则：
1. 新进程进入最高优先级队列
2. 用完时间片降级
3. 高优先级队列为空才执行低优先级
4. I/O 密集型任务保持高优先级
```

## 3. Linux 调度器 ⭐⭐

### 3.1 CFS（完全公平调度器）

```
CFS = Completely Fair Scheduler

核心思想：
- 每个进程应该获得相等的 CPU 时间
- 使用虚拟运行时间（vruntime）跟踪
- 总是运行 vruntime 最小的进程

┌─────────────────────────────────────────────────────┐
│                   红黑树                            │
│                                                      │
│                    ┌───┐                            │
│                    │P3 │ vruntime=50                │
│                   /     \                           │
│              ┌───┐       ┌───┐                      │
│              │P1 │       │P5 │                      │
│              │30 │       │70 │                      │
│             /    \      /    \                      │
│        ┌───┐  ┌───┐ ┌───┐  ┌───┐                   │
│        │P2 │  │P4 │ │P6 │  │P7 │                   │
│        │20 │  │40 │ │60 │  │80 │                   │
│        └───┘  └───┘ └───┘  └───┘                   │
│          ↑                                          │
│       最左节点 = 下一个运行的进程                    │
└─────────────────────────────────────────────────────┘
```

### 3.2 查看进程优先级

```bash
# 查看进程优先级
ps -el

# 输出：
# F S   UID   PID  PPID  C PRI  NI ADDR SZ WCHAN  TTY          TIME CMD
# 4 S     0     1     0  0  80   0 -  2900 -      ?        00:00:01 systemd

# PRI = 优先级（越小越高）
# NI = nice 值（-20 到 19）

# 修改 nice 值
nice -n 10 ./myapp      # 启动时设置
renice -n 5 -p 1234     # 修改运行中的进程
```

### 3.3 实时调度

```bash
# 查看调度策略
chrt -p 1234

# 设置实时调度（需要 root）
chrt -f -p 50 1234    # FIFO 策略，优先级 50
chrt -r -p 50 1234    # Round-Robin 策略

# 调度策略：
# SCHED_OTHER  - 普通调度（CFS）
# SCHED_FIFO   - 实时 FIFO
# SCHED_RR     - 实时轮转
```

### 4.2 调度时机

```go
package main

import (
    "fmt"
    "runtime"
    "time"
)

func main() {
    // 1. 主动让出
    go func() {
        for i := 0; i < 10; i++ {
            fmt.Println("goroutine:", i)
            runtime.Gosched() // 主动让出 CPU
        }
    }()
    
    // 2. 系统调用
    go func() {
        // 文件 I/O、网络 I/O 会触发调度
        time.Sleep(time.Second)
    }()
    
    // 3. channel 操作
    go func() {
        ch := make(chan int)
        <-ch // 阻塞，触发调度
    }()
    
    // 4. 函数调用（栈检查）
    go func() {
        recursion(100)
    }()
    
    time.Sleep(2 * time.Second)
}

func recursion(n int) {
    if n == 0 {
        return
    }
    recursion(n - 1)
}
```

### 4.3 Work Stealing

```
当 P 的本地队列为空时，会从其他 P 偷取 G

P0 队列空了：
┌─────────┐  ┌─────────┐  ┌─────────┐
│    P0   │  │    P1   │  │    P2   │
│ ┌─────┐ │  │ ┌─────┐ │  │ ┌─────┐ │
│ │     │ │  │ │G G G│ │  │ │G G G│ │
│ │ 空  │ │  │ │G G G│ │  │ │G G G│ │
│ └─────┘ │  │ └─────┘ │  │ └─────┘ │
└─────────┘  └─────────┘  └─────────┘
     │              ↑
     └──── 偷取 ────┘

偷取策略：
1. 先从全局队列获取
2. 再从其他 P 的本地队列偷取一半
```

### 4.4 调度器参数

```go

func main() {
```

## 6. 调度优化 ⭐⭐

### 6.1 避免过多 goroutine

```go
package main

import (
    "fmt"
    "runtime"
    "sync"
)

// 不好：创建过多 goroutine
func bad() {
    for i := 0; i < 1000000; i++ {
        go func() {
            // 做一些工作
        }()
    }
}

// 好：使用 worker pool
func good() {
    jobs := make(chan int, 100)
    var wg sync.WaitGroup
    
    // 固定数量的 worker
    workers := runtime.NumCPU()
    for i := 0; i < workers; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            for job := range jobs {
                _ = job // 处理任务
            }
        }()
    }
    
    // 发送任务
    for i := 0; i < 1000000; i++ {
        jobs <- i
    }
    close(jobs)
    
    wg.Wait()
}

func main() {
    good()
    fmt.Println("完成")
}
```

### 6.2 减少锁竞争

```go
package main

import (
    "sync"
    "sync/atomic"
)

// 不好：全局锁
type BadCounter struct {
    mu    sync.Mutex
    count int
}

func (c *BadCounter) Inc() {
    c.mu.Lock()
    c.count++
    c.mu.Unlock()
}

// 好：原子操作
type GoodCounter struct {
    count int64
}

func (c *GoodCounter) Inc() {
    atomic.AddInt64(&c.count, 1)
}

// 更好：分片计数器
type ShardedCounter struct {
    shards [16]struct {
        count int64
        _     [56]byte // 避免 false sharing
    }
}

func (c *ShardedCounter) Inc(id int) {
    atomic.AddInt64(&c.shards[id%16].count, 1)
}

func (c *ShardedCounter) Total() int64 {
    var total int64
    for i := range c.shards {
        total += atomic.LoadInt64(&c.shards[i].count)
    }
    return total
}
```

2. GOMAXPROCS 设置为 1 时，goroutine 还能并发吗？

<summary>点击查看答案</summary>

### 练习3：Worker Pool

实现一个可配置的 Worker Pool，支持：
- 动态调整 worker 数量
- 任务超时
- 优雅关闭

<details>
<summary>点击查看答案</summary>

```go
package main

import (
    "context"
    "fmt"
    "sync"
    "time"
)

type Task struct {
    ID      int
    Timeout time.Duration
    Work    func() error
}

type Pool struct {
    tasks    chan Task
    workers  int
    wg       sync.WaitGroup
    ctx      context.Context
    cancel   context.CancelFunc
}

func NewPool(workers int) *Pool {
    ctx, cancel := context.WithCancel(context.Background())
    p := &Pool{
        tasks:   make(chan Task, 100),
        workers: workers,
        ctx:     ctx,
        cancel:  cancel,
    }
    
    for i := 0; i < workers; i++ {
        p.wg.Add(1)
        go p.worker(i)
    }
    
    return p
}

func (p *Pool) worker(id int) {
    defer p.wg.Done()
    
    for {
        select {
        case <-p.ctx.Done():
            fmt.Printf("Worker %d 退出\n", id)
            return
        case task, ok := <-p.tasks:
            if !ok {
                return
            }
            p.executeTask(id, task)
        }
    }
}

func (p *Pool) executeTask(workerID int, task Task) {
    ctx, cancel := context.WithTimeout(p.ctx, task.Timeout)
    defer cancel()
    
    done := make(chan error, 1)
    go func() {
        done <- task.Work()
    }()
    
    select {
    case err := <-done:
        if err != nil {
            fmt.Printf("Worker %d: 任务 %d 失败: %v\n", workerID, task.ID, err)
        } else {
            fmt.Printf("Worker %d: 任务 %d 完成\n", workerID, task.ID)
        }
    case <-ctx.Done():
        fmt.Printf("Worker %d: 任务 %d 超时\n", workerID, task.ID)
    }
}

func (p *Pool) Submit(task Task) {
    p.tasks <- task
}

func (p *Pool) Shutdown() {
    close(p.tasks)
    p.wg.Wait()
}

func (p *Pool) ForceShutdown() {
    p.cancel()
    p.wg.Wait()
}

func main() {
    pool := NewPool(3)
    
    for i := 1; i <= 10; i++ {
        id := i
        pool.Submit(Task{
            ID:      id,
            Timeout: 2 * time.Second,
            Work: func() error {
                time.Sleep(time.Duration(id*100) * time.Millisecond)
                return nil
            },
        })
    }
    
    pool.Shutdown()
    fmt.Println("所有任务完成")
}
```
</details>

## 8. 本章小结

**调度算法：**

| 算法 | 特点 | 适用场景 |
|------|------|---------|
| FCFS | 简单，不公平 | 批处理 |
| SJF | 平均等待最短 | 已知运行时间 |
| RR | 公平，响应好 | 交互式系统 |
| 优先级 | 重要任务优先 | 实时系统 |
| MLFQ | 综合优化 | 通用系统 |

**Go 调度器：**

| 概念 | 说明 |
|------|------|
| G | goroutine |
| M | 系统线程 |
| P | 逻辑处理器 |
| Work Stealing | 负载均衡 |
| GOMAXPROCS | P 的数量 |

**优化建议：**
- 使用 Worker Pool 限制 goroutine 数量
- 减少锁竞争，使用原子操作
- 合理设置 GOMAXPROCS
- 使用 trace 工具分析调度
