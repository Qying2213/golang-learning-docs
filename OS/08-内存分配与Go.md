## 1. 为什么需要内存分配器？⭐⭐⭐

### 1.1 系统调用的问题

```
直接使用系统调用分配内存的问题：

┌─────────────────────────────────────────────────────┐
│  应用程序                                           │
│     │                                               │
│     │ malloc(100)                                   │
│     ↓                                               │
│  ┌─────────────────────────────────────────────┐   │
│  │              系统调用                        │   │
│  │  mmap() / brk()                             │   │
│  │  - 需要切换到内核态（约 1000 个时钟周期）   │   │
│  │  - 每次分配都要系统调用                     │   │
│  │  - 小对象分配效率极低                       │   │
│  └─────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────┘

问题：
1. 系统调用开销大（用户态 ↔ 内核态切换）
2. 小对象频繁分配效率低
3. 内存碎片严重
4. 无法针对应用特点优化
```

### 1.2 用户态内存分配器

```
解决方案：用户态内存分配器

┌─────────────────────────────────────────────────────┐
│  应用程序                                           │
│     │                                               │
│     │ new(Object)                                   │
│     ↓                                               │
│  ┌─────────────────────────────────────────────┐   │
│  │           Go 内存分配器                      │   │
│  │  - 预先从系统申请大块内存                   │   │
│  │  - 自己管理内存分配和回收                   │   │
│  │  - 针对不同大小对象优化                     │   │
│  │  - 减少系统调用                             │   │
│  │  - 减少锁竞争（每个 P 有本地缓存）          │   │
│  └─────────────────────────────────────────────┘   │
│                        │                            │
│                        │ 偶尔                       │
│                        ↓                            │
│  ┌─────────────────────────────────────────────┐   │
│  │              系统调用                        │   │
│  │  mmap() - 申请大块内存                      │   │
│  └─────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────┘
```

### 📝 练习题 1.1

**问题**：为什么 Go 不直接使用 C 的 malloc/free？

<details>
<summary>点击查看答案</summary>

**答案**：

1. **GC 集成**：Go 有垃圾回收，需要追踪所有分配的对象，自定义分配器可以更好地与 GC 配合

2. **goroutine 优化**：Go 需要为大量 goroutine 分配栈，自定义分配器可以优化栈的分配和增长

3. **减少锁竞争**：Go 的分配器为每个 P 提供本地缓存（mcache），大部分分配无需加锁

4. **内存布局**：Go 需要特定的内存布局来支持 GC 的标记和清扫

5. **性能优化**：针对 Go 程序的特点（大量小对象、goroutine）进行优化

</details>

## 2. Go 内存分配器架构 ⭐⭐⭐

### 2.1 三层架构

```
Go 内存分配器的三层架构：

┌─────────────────────────────────────────────────────┐
│                   Go 程序                            │
│                                                      │
│  ┌─────────────────────────────────────────────┐    │
│  │     mcache（每个 P 一个，无锁分配）          │    │
│  │  ┌─────┐ ┌─────┐ ┌─────┐ ┌─────┐           │    │
│  │  │tiny │ │ 8B  │ │ 16B │ │ 32B │ ... (67种) │    │
│  │  │alloc│ │span │ │span │ │span │           │    │
│  │  └─────┘ └─────┘ └─────┘ └─────┘           │    │
│  └─────────────────────────────────────────────┘    │
│                        ↓ 本地缓存用完                │
│  ┌─────────────────────────────────────────────┐    │
│  │     mcentral（全局，每种大小一个，需要锁）   │    │
│  │  管理各种大小的 span 列表                    │    │
│  │  - nonempty: 有空闲对象的 span              │    │
│  │  - empty: 没有空闲对象的 span               │    │
│  └─────────────────────────────────────────────┘    │
│                        ↓ mcentral 也没有            │
│  ┌─────────────────────────────────────────────┐    │
│  │     mheap（全局唯一，管理所有内存）          │    │
│  │  - 管理所有的 arena（64MB 块）              │    │
│  │  - 向操作系统申请/归还内存                  │    │
│  └─────────────────────────────────────────────┘    │
│                        ↓                             │
│  ┌─────────────────────────────────────────────┐    │
│  │              操作系统                        │    │
│  │  mmap / munmap                              │    │
│  └─────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────┘
```

### 2.2 核心概念

| 概念 | 说明 | 大小 |
|------|------|------|
| mcache | 每个 P 的本地缓存，无锁 | - |
| mcentral | 全局缓存，按大小等级管理 | - |
| mheap | 堆管理器，管理所有内存 | - |
| mspan | 内存管理的基本单位 | N 个页 |
| page | 内存页 | 8KB |
| arena | 从系统申请的大块内存 | 64MB |
| object | 分配给用户的对象 | 变化 |

### 2.3 mspan 详解

```
mspan：一组连续的页，用于分配特定大小的对象

┌─────────────────────────────────────────────────────┐
│                     mspan                            │
├─────────────────────────────────────────────────────┤
│  startAddr:  起始地址                               │
│  npages:     页数                                   │
│  spanclass:  大小等级（0-67）                       │
│  elemsize:   对象大小                               │
│  nelems:     对象数量                               │
│  allocBits:  分配位图（哪些对象已分配）             │
│  gcmarkBits: GC 标记位图                            │
│  freeindex:  下一个空闲对象的索引                   │
├─────────────────────────────────────────────────────┤
│  ┌─────┬─────┬─────┬─────┬─────┬─────┬─────┐       │
│  │obj0 │obj1 │obj2 │obj3 │obj4 │obj5 │ ... │       │
│  │used │used │free │used │free │free │     │       │
│  └─────┴─────┴─────┴─────┴─────┴─────┴─────┘       │
│                                                      │
│  allocBits: 1 1 0 1 0 0 ...                         │
└─────────────────────────────────────────────────────┘
```

### 📝 练习题 2.1

**问题**：为什么 mcache 是每个 P 一个，而不是每个 M 或每个 goroutine 一个？

<details>
<summary>点击查看答案</summary>

**答案**：

1. **每个 goroutine 一个**：
   - goroutine 数量可能很大（百万级）
   - 每个都有 mcache 会浪费大量内存
   - goroutine 创建/销毁频繁，管理开销大

2. **每个 M 一个**：
   - M 的数量不固定，可能很多
   - M 可能被阻塞，mcache 无法被其他 M 使用
   - M 和 P 解绑时需要处理 mcache

3. **每个 P 一个（Go 的选择）**：
   - P 的数量固定（GOMAXPROCS）
   - 同一时刻只有一个 M 绑定到 P，无需加锁
   - P 切换 M 时 mcache 不变
   - 内存使用可控

</details>

## 3. 对象大小分类 ⭐⭐⭐

### 3.1 三种对象类型

```
Go 将对象分为三类，采用不同的分配策略：

┌─────────────────────────────────────────────────────┐
│  微小对象（Tiny）：< 16B，无指针                     │
│  ─────────────────────────────────────────────────  │
│  特点：                                             │
│  - 多个微小对象合并到一个 16B 块                    │
│  - 减少内存浪费                                     │
│  - 典型：小字符串、小整数                           │
│                                                      │
│  分配方式：                                         │
│  ┌────────────────────────────────────┐            │
│  │  16B 块                            │            │
│  │  ┌────┬────┬────┬────┬────────┐   │            │
│  │  │ 3B │ 5B │ 4B │ 2B │ 空闲2B │   │            │
│  │  └────┴────┴────┴────┴────────┘   │            │
│  └────────────────────────────────────┘            │
├─────────────────────────────────────────────────────┤
│  小对象（Small）：16B - 32KB                        │
│  ─────────────────────────────────────────────────  │
│  特点：                                             │
│  - 使用 mcache 分配（无锁）                         │
│  - 按大小分成 67 个等级                             │
│  - 最常见的分配类型                                 │
│                                                      │
│  分配方式：                                         │
│  1. 根据大小找到对应的 span                        │
│  2. 从 span 中分配一个对象                         │
├─────────────────────────────────────────────────────┤
│  大对象（Large）：> 32KB                            │
│  ─────────────────────────────────────────────────  │
│  特点：                                             │
│  - 直接从 mheap 分配                                │
│  - 不使用 mcache                                    │
│  - 分配的是整数个页（8KB）                          │
│                                                      │
│  分配方式：                                         │
│  直接从 mheap 获取足够的页                         │
└─────────────────────────────────────────────────────┘
```

### 3.2 大小等级（Size Class）

```go
// Go 的 67 种大小等级（部分）
// 等级  对象大小  span页数  每span对象数  浪费率
//  1      8B        1         1024        0%
//  2     16B        1          512        0%
//  3     24B        1          341        0%
//  4     32B        1          256        0%
//  5     48B        1          170        0%
//  6     64B        1          128        0%
//  7     80B        1          102        0%
//  8     96B        1           85        0%
//  9    112B        1           73        0%
// 10    128B        1           64        0%
// ...
// 66   32768B       4            1        0%

// 分配示例：
// 分配 20B → 向上取整到 24B（等级 3）
// 分配 100B → 向上取整到 112B（等级 9）
// 分配 50000B → 大对象，直接从 mheap 分配
```

### 3.3 查看分配情况

```go
package main

import (
    "fmt"
    "runtime"
    "unsafe"
)

func main() {
    // 查看初始状态
    var m1 runtime.MemStats
    runtime.ReadMemStats(&m1)
    
    fmt.Println("=== 对象大小分类演示 ===")
    
    // 微小对象（< 16B，无指针）
    var a int8 = 1
    var b int8 = 2
    var c int16 = 3
    fmt.Printf("\n微小对象:\n")
    fmt.Printf("  int8 大小: %d bytes, 地址: %p\n", unsafe.Sizeof(a), &a)
    fmt.Printf("  int8 大小: %d bytes, 地址: %p\n", unsafe.Sizeof(b), &b)
    fmt.Printf("  int16 大小: %d bytes, 地址: %p\n", unsafe.Sizeof(c), &c)
    
    // 小对象（16B - 32KB）
    small1 := make([]byte, 100)   // 112B 等级
    small2 := make([]byte, 1000)  // 1024B 等级
    small3 := make([]byte, 10000) // 10240B 等级
    fmt.Printf("\n小对象:\n")
    fmt.Printf("  []byte(100) 地址: %p\n", &small1[0])
    fmt.Printf("  []byte(1000) 地址: %p\n", &small2[0])
    fmt.Printf("  []byte(10000) 地址: %p\n", &small3[0])
    
    // 大对象（> 32KB）
    large := make([]byte, 100000)
    fmt.Printf("\n大对象:\n")
    fmt.Printf("  []byte(100000) 地址: %p\n", &large[0])
    
    // 查看分配后的状态
    var m2 runtime.MemStats
    runtime.ReadMemStats(&m2)
    
    fmt.Printf("\n=== 内存统计 ===\n")
    fmt.Printf("堆分配增加: %d bytes\n", m2.HeapAlloc-m1.HeapAlloc)
    fmt.Printf("分配次数增加: %d\n", m2.Mallocs-m1.Mallocs)
    fmt.Printf("堆对象数增加: %d\n", m2.HeapObjects-m1.HeapObjects)
    
    // 防止变量被优化掉
    _ = a + b + int8(c)
    _ = small1
    _ = small2
    _ = small3
    _ = large
}
```

### 📝 练习题 3.1

**问题**：分配一个 20 字节的对象，实际会占用多少内存？为什么？

<details>
<summary>点击查看答案</summary>

**答案**：实际占用 24 字节。

**原因**：
1. Go 的内存分配器使用大小等级
2. 20 字节向上取整到最近的等级：24 字节（等级 3）
3. 这种设计的好处：
   - 减少内存碎片
   - 简化内存管理
   - 提高分配效率

**内部碎片**：
- 浪费了 4 字节（24 - 20 = 4）
- 浪费率：4/24 ≈ 16.7%

**工作中的应用**：
- 设计结构体时考虑大小等级
- 避免刚好超过等级边界（如 17 字节会分配 24 字节）

</details>

## 4. 分配流程详解 ⭐⭐⭐

### 4.1 小对象分配流程

```
分配 100 字节对象的完整流程：

步骤1：计算大小等级
┌─────────────────────────────────────────────────────┐
│  100 字节 → 查表 → 等级 9（112 字节）               │
└─────────────────────────────────────────────────────┘
                        │
                        ↓
步骤2：从 mcache 分配
┌─────────────────────────────────────────────────────┐
│  mcache.alloc[9] 有空闲对象？                       │
│  ├─ 是：从 span 的 freeindex 位置分配，返回        │
│  └─ 否：继续步骤 3                                  │
└─────────────────────────────────────────────────────┘
                        │
                        ↓
步骤3：从 mcentral 获取 span
┌─────────────────────────────────────────────────────┐
│  mcentral[9].nonempty 有 span？                     │
│  ├─ 是：取一个 span 给 mcache，返回步骤 2          │
│  └─ 否：继续步骤 4                                  │
│                                                      │
│  注意：这一步需要加锁！                             │
└─────────────────────────────────────────────────────┘
                        │
                        ↓
步骤4：从 mheap 分配新 span
┌─────────────────────────────────────────────────────┐
│  mheap.free 有合适大小的 span？                     │
│  ├─ 是：分配给 mcentral                            │
│  └─ 否：向操作系统申请新的 arena（64MB）           │
│                                                      │
│  注意：这一步也需要加锁！                           │
└─────────────────────────────────────────────────────┘
```

### 4.2 大对象分配流程

```
分配 > 32KB 对象的流程（简单得多）：

┌─────────────────────────────────────────────────────┐
│  1. 计算需要的页数                                  │
│     例：50KB → 需要 7 页（向上取整，每页 8KB）      │
│                                                      │
│  2. 直接从 mheap 分配                               │
│     - 不经过 mcache 和 mcentral                     │
│     - 需要加锁                                      │
│                                                      │
│  3. 如果 mheap 没有足够的页                         │
│     - 向操作系统申请新的 arena                      │
│                                                      │
│  4. 返回分配的内存地址                              │
└─────────────────────────────────────────────────────┘
```

### 4.3 分配流程代码演示

```go
package main

import (
    "fmt"
    "runtime"
)

func main() {
    // 记录初始状态
    var m1, m2 runtime.MemStats
    
    fmt.Println("=== 小对象分配 ===")
    runtime.GC()
    runtime.ReadMemStats(&m1)
    
    // 分配 1000 个小对象
    objects := make([]*[100]byte, 1000)
    for i := range objects {
        objects[i] = new([100]byte)
    }
    
    runtime.ReadMemStats(&m2)
    fmt.Printf("分配 1000 个 100B 对象:\n")
    fmt.Printf("  HeapAlloc 增加: %d bytes (%.2f KB)\n", 
        m2.HeapAlloc-m1.HeapAlloc, float64(m2.HeapAlloc-m1.HeapAlloc)/1024)
    fmt.Printf("  理论最小: %d bytes (%.2f KB)\n", 
        1000*100, float64(1000*100)/1024)
    fmt.Printf("  实际/理论: %.2f%%\n", 
        float64(m2.HeapAlloc-m1.HeapAlloc)/float64(1000*100)*100)
    
    fmt.Println("\n=== 大对象分配 ===")
    runtime.GC()
    runtime.ReadMemStats(&m1)
    
    // 分配大对象
    largeObj := make([]byte, 100*1024) // 100KB
    _ = largeObj
    
    runtime.ReadMemStats(&m2)
    fmt.Printf("分配 100KB 大对象:\n")
    fmt.Printf("  HeapAlloc 增加: %d bytes (%.2f KB)\n", 
        m2.HeapAlloc-m1.HeapAlloc, float64(m2.HeapAlloc-m1.HeapAlloc)/1024)
    
    // 防止被优化
    _ = objects
}
```

### 📝 练习题 4.1

**问题**：为什么小对象分配通常比大对象分配快？

<details>
<summary>点击查看答案</summary>

**答案**：

1. **无锁分配**：
   - 小对象从 mcache 分配，无需加锁
   - 大对象直接从 mheap 分配，需要加锁

2. **预分配**：
   - mcache 预先持有各种大小的 span
   - 分配时只需移动 freeindex
   - 大对象每次都要从 mheap 查找

3. **缓存友好**：
   - mcache 是 P 本地的，缓存命中率高
   - mheap 是全局的，可能有缓存竞争

4. **分配复杂度**：
   - 小对象：O(1) 位图操作
   - 大对象：需要查找合适大小的连续页

**性能差距**：小对象分配可能快 10-100 倍

</details>

## 5. 逃逸分析 ⭐⭐⭐

### 5.1 什么是逃逸分析？

```
逃逸分析：编译器决定变量分配在栈还是堆

┌─────────────────────────────────────────────────────┐
│                    栈分配                            │
│  ─────────────────────────────────────────────────  │
│  优点：                                             │
│  - 分配快（只需移动栈指针）                         │
│  - 释放快（函数返回自动释放）                       │
│  - 不需要 GC                                        │
│  - 缓存友好                                         │
│                                                      │
│  限制：                                             │
│  - 生命周期必须在函数内                             │
│  - 大小必须编译时确定                               │
├─────────────────────────────────────────────────────┤
│                    堆分配                            │
│  ─────────────────────────────────────────────────  │
│  优点：                                             │
│  - 生命周期灵活                                     │
│  - 大小可以动态确定                                 │
│                                                      │
│  缺点：                                             │
│  - 分配慢（需要查找空闲内存）                       │
│  - 需要 GC 回收                                     │
│  - 可能导致内存碎片                                 │
└─────────────────────────────────────────────────────┘

逃逸 = 变量的生命周期超出函数范围 → 必须分配在堆上
```

### 5.2 常见逃逸场景

```go
package main

import "fmt"

// 场景1：返回局部变量的指针 ⭐⭐⭐
func escape1() *int {
    x := 100      // x 逃逸到堆
    return &x     // 返回指针，x 必须在函数返回后仍然有效
}

// 场景2：闭包引用 ⭐⭐⭐
func escape2() func() int {
    x := 100      // x 逃逸到堆
    return func() int {
        return x  // 闭包引用了 x
    }
}

// 场景3：interface{} 参数 ⭐⭐⭐
func escape3(x interface{}) {
    _ = x
}

func callEscape3() {
    y := 200
    escape3(y)    // y 逃逸到堆（interface{} 需要堆分配）
}

// 场景4：slice 扩容
func escape4() {
    s := make([]int, 0)
    for i := 0; i < 10000; i++ {
        s = append(s, i)  // 扩容时可能逃逸
    }
}

// 场景5：发送到 channel
func escape5(ch chan *int) {
    x := 100
    ch <- &x      // x 逃逸到堆
}

// 场景6：slice/map 的元素是指针
func escape6() {
    m := make(map[int]*int)
    x := 100
    m[1] = &x     // x 逃逸到堆
}

// 场景7：大对象
func escape7() {
    // 大于一定大小的对象会逃逸
    // 具体阈值取决于 Go 版本
    large := make([]byte, 64*1024)
    _ = large
}

// 不逃逸的例子
func noEscape() int {
    x := 100      // x 在栈上
```

## 6. sync.Pool 对象池 ⭐⭐⭐

### 6.1 为什么需要对象池？

```
问题：频繁创建和销毁对象

┌─────────────────────────────────────────────────────┐
│  HTTP 请求处理                                      │
│                                                      │
│  for 每个请求 {                                     │
│      buf := make([]byte, 4096)  // 分配            │
│      // 处理请求...                                 │
│      // 函数返回，buf 等待 GC                       │
│  }                                                  │
│                                                      │
│  问题：                                             │
│  - 大量内存分配                                     │
│  - GC 压力大                                        │
│  - 性能下降                                         │
└─────────────────────────────────────────────────────┘

解决：对象池

┌─────────────────────────────────────────────────────┐
│  使用 sync.Pool                                     │
│                                                      │
│  pool := sync.Pool{New: func() interface{} {       │
│      return make([]byte, 4096)                     │
│  }}                                                 │
│                                                      │
│  for 每个请求 {                                     │
│      buf := pool.Get().([]byte)  // 从池中获取     │
│      // 处理请求...                                 │
│      pool.Put(buf)               // 归还到池中     │
│  }                                                  │
│                                                      │
│  优点：                                             │
│  - 复用对象，减少分配                               │
│  - 减少 GC 压力                                     │
│  - 提高性能                                         │
└─────────────────────────────────────────────────────┘
```

### 6.2 sync.Pool 基本使用

```go
package main

import (
    "bytes"
    "fmt"
    "sync"
)

// 创建对象池
var bufferPool = sync.Pool{
    New: func() interface{} {
        fmt.Println("创建新的 buffer")
        return new(bytes.Buffer)
    },
}

func getBuffer() *bytes.Buffer {
    return bufferPool.Get().(*bytes.Buffer)
}

func putBuffer(buf *bytes.Buffer) {
    buf.Reset()  // 重要：清理后再放回
    bufferPool.Put(buf)
}

func main() {
    fmt.Println("=== sync.Pool 演示 ===")
    
    // 第一次获取，会调用 New
    buf1 := getBuffer()
    buf1.WriteString("Hello")
    fmt.Printf("buf1: %s, addr: %p\n", buf1.String(), buf1)
    
    // 归还
    putBuffer(buf1)
    
    // 再次获取，复用之前的对象
    buf2 := getBuffer()
    fmt.Printf("buf2 addr: %p (应该和 buf1 相同)\n", buf2)
    
    // 同时获取多个
    buf3 := getBuffer()
    buf4 := getBuffer()  // 这个会调用 New
    fmt.Printf("buf3 addr: %p\n", buf3)
    fmt.Printf("buf4 addr: %p\n", buf4)
    
    putBuffer(buf2)
    putBuffer(buf3)
    putBuffer(buf4)
}
```

### 6.3 实际应用：HTTP 请求处理

```go
package main

import (
    "bytes"
    "encoding/json"
    "net/http"
    "sync"
)

// JSON 编码器池
var encoderPool = sync.Pool{
    New: func() interface{} {
        return new(bytes.Buffer)
    },
}

// 响应结构
type Response struct {
    Code    int         `json:"code"`
    Message string      `json:"message"`
    Data    interface{} `json:"data"`
}

func handler(w http.ResponseWriter, r *http.Request) {
    // 从池中获取 buffer
    buf := encoderPool.Get().(*bytes.Buffer)
    defer func() {
        buf.Reset()
        encoderPool.Put(buf)
    }()
    
    // 构造响应
    resp := Response{
        Code:    200,
        Message: "success",
        Data:    map[string]string{"hello": "world"},
    }
    
    // 编码到 buffer
    if err := json.NewEncoder(buf).Encode(resp); err != nil {
        http.Error(w, err.Error(), 500)
        return
    }
    
    // 写入响应
    w.Header().Set("Content-Type", "application/json")
    w.Write(buf.Bytes())
}

func main() {
    http.HandleFunc("/api", handler)
    http.ListenAndServe(":8080", nil)
}
```

### 6.5 sync.Pool 注意事项

```go
// ⚠️ 注意事项

// 1. Pool 中的对象可能随时被 GC 回收
// 不要依赖 Pool 中一定有对象

// 2. 放回前要清理对象状态

    pool.Put(buf)

// 如果需要缓存，使用 map 或专门的缓存库
```

### 📝 练习题 6.1

**问题**：实现一个带大小限制的对象池。

<details>
<summary>点击查看答案</summary>

```go
package main

import (
    "fmt"
    "sync"
    "sync/atomic"
)

type LimitedPool struct {
    pool     sync.Pool
    maxSize  int64
    currSize int64
}

func NewLimitedPool(maxSize int64, newFunc func() interface{}) *LimitedPool {
    return &LimitedPool{
        pool: sync.Pool{
            New: newFunc,
        },
        maxSize: maxSize,
    }
}

func (p *LimitedPool) Get() interface{} {
    obj := p.pool.Get()
    if obj != nil {
        atomic.AddInt64(&p.currSize, -1)
    }
    return obj
}

func (p *LimitedPool) Put(obj interface{}) bool {
    if atomic.LoadInt64(&p.currSize) >= p.maxSize {
        // 池已满，丢弃对象
        return false
    }
    atomic.AddInt64(&p.currSize, 1)
    p.pool.Put(obj)
    return true
}

func (p *LimitedPool) Size() int64 {
    return atomic.LoadInt64(&p.currSize)
}

func main() {
    pool := NewLimitedPool(3, func() interface{} {
        return make([]byte, 1024)
    })
    
    // 获取并放回
    obj1 := pool.Get()
    obj2 := pool.Get()
    obj3 := pool.Get()
    
    pool.Put(obj1)
    pool.Put(obj2)
    pool.Put(obj3)
    
    fmt.Println("池大小:", pool.Size())
    
    // 尝试放入第4个
    obj4 := pool.Get()
    if !pool.Put(obj4) {
        fmt.Println("池已满，对象被丢弃")
    }
}
```

```go
    for _, s := range strs {

    total := 0
    for _, s := range strs {

    var builder strings.Builder
    builder.Grow(total)  // 预分配
    for _, s := range strs {
        builder.WriteString(s)
    }
    return builder.String()
}

func main() {
```

| 内存碎片 | 内存使用高但对象少 | 使用对象池、预分配 |
| GC 压力大 | GC 频繁、STW 时间长 | 减少分配、使用 sync.Pool |
| 大对象分配 | 分配慢、内存峰值高 | 复用大对象、分批处理 |

## 9. 面试常见问题 ⭐⭐⭐

**Q1: Go 的内存分配器有几层？各自的作用是什么？**

<details>
<summary>点击查看答案</summary>

三层架构：

1. **mcache**（本地缓存）
   - 每个 P 一个
   - 无锁分配
   - 存储各种大小的 span

2. **mcentral**（中心缓存）
   - 全局，每种大小一个
   - 需要加锁
   - 管理 span 列表

3. **mheap**（堆）
   - 全局唯一
   - 管理所有内存
   - 向操作系统申请/归还内存

</details>

**Q2: 什么是逃逸分析？如何避免逃逸？**

<details>
<summary>点击查看答案</summary>

**逃逸分析**：编译器分析变量的生命周期，决定分配在栈还是堆。

**避免逃逸的方法**：
1. 不返回局部变量的指针
2. 避免使用 interface{}
3. 预分配 slice 容量
4. 使用值传递而非指针（小对象）
5. 避免闭包捕获局部变量

**查看逃逸**：`go build -gcflags="-m"`

</details>

**Q3: sync.Pool 的原理是什么？什么时候使用？**

<details>
<summary>点击查看答案</summary>

**原理**：
- 每个 P 有本地池（无锁）
- 有全局共享池（需要锁）
- GC 时会清空池中对象

**使用场景**：
- 频繁创建和销毁的临时对象
- 对象创建成本高
- 对象可以安全复用

**不适合**：
- 需要持久化的对象
- 有状态的对象
- 作为缓存使用

</details>

## 10. 本章小结

### 核心概念

| 概念 | 说明 | 重要程度 |
|------|------|---------|
| mcache | P 本地缓存，无锁 | ⭐⭐⭐ |
| mcentral | 全局缓存，按大小管理 | ⭐⭐⭐ |
| mheap | 堆管理器 | ⭐⭐⭐ |
| 逃逸分析 | 决定栈/堆分配 | ⭐⭐⭐ |
| sync.Pool | 对象池 | ⭐⭐⭐ |

### 对象分类

| 类型 | 大小 | 分配方式 |
|------|------|---------|
| Tiny | < 16B | mcache tiny 分配器 |
| Small | 16B - 32KB | mcache → mcentral |
| Large | > 32KB | 直接从 mheap |

### 优化技巧

1. 避免逃逸，尽量栈分配
2. 使用 sync.Pool 复用对象
3. 预分配 slice/map 容量
4. 优化结构体字段顺序
5. 使用 strings.Builder 拼接字符串
6. 避免不必要的指针
