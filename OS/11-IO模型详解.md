## 1. I/O 基础概念 ⭐⭐⭐

### 1.1 什么是 I/O？

```
I/O = Input/Output（输入/输出）

常见的 I/O 操作：
┌─────────────────────────────────────────────────────┐
│  文件 I/O：读写文件                                 │
│  网络 I/O：发送接收网络数据                         │
│  数据库 I/O：查询数据库                             │
│  终端 I/O：键盘输入、屏幕输出                       │
└─────────────────────────────────────────────────────┘

I/O 的特点：
- 比 CPU 计算慢得多（差距可达 100000 倍）
- 需要等待外部设备
- 是程序性能的主要瓶颈

速度对比：
┌─────────────────────────────────────────────────────┐
│  操作              │  延迟                          │
├─────────────────────────────────────────────────────┤
│  CPU 寄存器        │  < 1 ns                        │
│  L1 缓存           │  1 ns                          │
│  L2 缓存           │  4 ns                          │
│  L3 缓存           │  12 ns                         │
│  内存              │  100 ns                        │
│  SSD               │  100 μs (100000 ns)            │
│  HDD               │  10 ms (10000000 ns)           │
│  网络（同机房）    │  500 μs                        │
│  网络（跨城市）    │  50 ms                         │
└─────────────────────────────────────────────────────┘
```

### 1.2 I/O 操作的两个阶段

```
以网络读取为例，I/O 操作分为两个阶段：

阶段1：等待数据准备好（数据从网卡到内核缓冲区）
┌─────────────────────────────────────────────────────┐
│  应用程序                     内核                   │
│     │                          │                    │
│     │ ──── read() ────────→   │                    │
│     │                          │ 等待网络数据到达   │
│     │                          │ ...               │
│     │                          │ 数据到达内核缓冲区 │
└─────────────────────────────────────────────────────┘

阶段2：将数据从内核复制到用户空间
┌─────────────────────────────────────────────────────┐
│  应用程序                     内核                   │
│     │                          │                    │
│     │                          │ 复制数据到用户空间 │
│     │ ←─── 返回数据 ─────────  │                    │
│     │                          │                    │
└─────────────────────────────────────────────────────┘

不同 I/O 模型的区别就在于这两个阶段的处理方式不同！
```

### 📝 练习题 1.1

**问题**：为什么需要两个阶段？为什么不能直接把数据写到用户空间？

<details>
<summary>点击查看答案</summary>

**答案**：

1. **安全隔离**：
   - 用户空间和内核空间是隔离的
   - 网卡等硬件只能访问内核空间
   - 防止用户程序直接操作硬件

2. **缓冲管理**：
   - 内核可以统一管理缓冲区
   - 支持多个进程共享数据
   - 实现零拷贝等优化

3. **权限控制**：
   - 内核可以检查权限
   - 防止越权访问

**零拷贝技术**（如 sendfile）可以减少拷贝次数，但仍需要内核参与。

</details>

## 2. 五种 I/O 模型 ⭐⭐⭐

### 2.1 阻塞 I/O（Blocking I/O）

```
最简单的模型，也是默认模型

应用程序                     内核
    │                          │
    │ ──── read() ────────→   │
    │         阻塞等待         │ 等待数据（阶段1）
    │         ...              │ ...
    │         ...              │ 数据到达
    │         ...              │ 复制数据（阶段2）
    │ ←─── 返回数据 ─────────  │
    │                          │

特点：
✅ 简单易用
✅ 编程模型直观
❌ 一个线程只能处理一个连接
❌ 大量连接需要大量线程
❌ 线程切换开销大
```

```go
package main

import (
    "fmt"
    "net"
)

func main() {
    conn, _ := net.Dial("tcp", "example.com:80")
    defer conn.Close()
    
    // 发送请求
    conn.Write([]byte("GET / HTTP/1.1\r\nHost: example.com\r\n\r\n"))
    
    // 阻塞 I/O：一直等待直到有数据
    buf := make([]byte, 1024)
    n, _ := conn.Read(buf)  // 阻塞在这里
    fmt.Println(string(buf[:n]))
}
```

### 2.2 非阻塞 I/O（Non-blocking I/O）

```
调用立即返回，需要轮询

应用程序                     内核
    │                          │
    │ ──── read() ────────→   │
    │ ←─── EAGAIN ──────────  │ 数据未准备好
    │                          │
    │ ──── read() ────────→   │
    │ ←─── EAGAIN ──────────  │ 数据未准备好
    │                          │
    │ ──── read() ────────→   │
    │         ...              │ 数据到达，复制
    │ ←─── 返回数据 ─────────  │

特点：
✅ 不会阻塞
❌ 需要不断轮询（polling）
❌ CPU 利用率低（忙等待）
❌ 实际很少单独使用
```

### 2.3 I/O 多路复用（I/O Multiplexing）⭐⭐⭐

```
一个线程监控多个 I/O，最常用的高性能模型

应用程序                     内核
    │                          │
    │ ──── select() ──────→   │ 监控多个 fd
    │         阻塞等待         │ ...
    │         ...              │ fd1 可读！
    │ ←─── 返回就绪 fd ──────  │
    │                          │
    │ ──── read(fd1) ─────→   │
    │ ←─── 返回数据 ─────────  │

特点：
✅ 一个线程处理多个连接
✅ 只在 I/O 就绪时才处理
✅ 高效处理大量连接
❌ 需要两次系统调用（select + read）

实现方式：select、poll、epoll（Linux）、kqueue（BSD/macOS）
```

### 2.4 信号驱动 I/O（Signal-driven I/O）

```
使用信号通知，较少使用

应用程序                     内核
    │                          │
    │ ──── sigaction() ───→   │ 注册信号处理
    │ ←─── 立即返回 ─────────  │
    │                          │
    │   继续其他工作           │ 等待数据
    │   ...                    │ ...
    │                          │ 数据到达
    │ ←─── SIGIO 信号 ───────  │
    │                          │
    │ ──── read() ────────→   │
    │ ←─── 返回数据 ─────────  │

特点：
✅ 数据准备好时收到信号
✅ 不需要轮询
❌ 信号处理复杂
❌ 实际使用较少
```

### 2.5 异步 I/O（Asynchronous I/O）

```
真正的异步，两个阶段都不阻塞

应用程序                     内核
    │                          │
    │ ──── aio_read() ────→   │
    │ ←─── 立即返回 ─────────  │
    │                          │
    │   继续其他工作           │ 等待数据
    │   ...                    │ ...
    │                          │ 数据到达
    │                          │ 复制数据到用户空间
    │ ←─── 完成通知 ─────────  │ 数据已在用户空间

特点：
✅ 真正的异步
✅ 两个阶段都不阻塞
✅ 性能最好
❌ 编程复杂
❌ Linux 支持有限（io_uring 是新方案）
```

### 📝 练习题 2.1

**问题**：五种 I/O 模型中，哪些是同步的，哪些是异步的？

<details>
<summary>点击查看答案</summary>

**答案**：

**同步 I/O**（阶段2会阻塞）：
1. 阻塞 I/O
2. 非阻塞 I/O
3. I/O 多路复用
4. 信号驱动 I/O

**异步 I/O**（两个阶段都不阻塞）：
5. 异步 I/O

**关键区别**：
- 同步 I/O：数据从内核复制到用户空间时，进程会阻塞
- 异步 I/O：内核完成所有工作后通知进程，进程不阻塞

**注意**：非阻塞 I/O 虽然阶段1不阻塞，但阶段2仍然阻塞，所以是同步的。

</details>

## 3. select/poll/epoll ⭐⭐⭐

### 3.1 select

```c
// select 函数原型
int select(int nfds, fd_set *readfds, fd_set *writefds,
           fd_set *exceptfds, struct timeval *timeout);

工作原理：
1. 将 fd 集合从用户空间复制到内核空间
2. 内核遍历所有 fd，检查是否就绪
3. 返回就绪的 fd 数量
4. 用户遍历 fd 集合，找到就绪的 fd

限制：
- 最多监控 1024 个 fd（FD_SETSIZE）
- 每次调用都要复制 fd 集合
- 返回后需要遍历所有 fd
- 时间复杂度 O(n)
```

### 3.2 poll

```c
// poll 函数原型
int poll(struct pollfd *fds, nfds_t nfds, int timeout);

struct pollfd {
    int   fd;         // 文件描述符
    short events;     // 监控的事件
    short revents;    // 返回的事件
};

改进：
- 没有 fd 数量限制
- 使用 pollfd 数组，更灵活

仍然存在的问题：
- 每次调用仍需复制 fd 数组
- 仍需遍历所有 fd
- 时间复杂度仍是 O(n)
```

### 3.3 epoll（Linux 特有）⭐⭐⭐

```c
// epoll 三个函数
int epoll_create(int size);  // 创建 epoll 实例
int epoll_ctl(int epfd, int op, int fd, struct epoll_event *event);  // 添加/修改/删除 fd
int epoll_wait(int epfd, struct epoll_event *events, int maxevents, int timeout);  // 等待事件

工作原理：
1. epoll_create：创建 epoll 实例（红黑树 + 就绪链表）
2. epoll_ctl：添加 fd 到红黑树
3. epoll_wait：返回就绪链表中的 fd

优势：
✅ 没有 fd 数量限制
✅ 只需要一次 epoll_ctl 添加 fd
✅ 只返回就绪的 fd，不需要遍历
✅ 时间复杂度 O(1)
✅ 支持边缘触发（ET）和水平触发（LT）
```

### 3.4 对比总结

| 特性 | select | poll | epoll |
|------|--------|------|-------|
| fd 数量限制 | 1024 | 无限制 | 无限制 |
| fd 传递方式 | 每次复制 | 每次复制 | 只需一次 |
| 返回方式 | 遍历所有 | 遍历所有 | 只返回就绪 |
| 时间复杂度 | O(n) | O(n) | O(1) |
| 触发模式 | LT | LT | LT/ET |
| 适用场景 | 连接少 | 连接少 | 高并发 |

### 3.5 水平触发 vs 边缘触发

```
水平触发（LT - Level Triggered）：
┌─────────────────────────────────────────────────────┐
│  只要 fd 可读/可写，就会通知                        │
│  如果不处理，下次 epoll_wait 还会通知               │
│                                                      │
│  优点：编程简单，不会丢数据                         │
│  缺点：可能重复通知，效率略低                       │
└─────────────────────────────────────────────────────┘

边缘触发（ET - Edge Triggered）：
┌─────────────────────────────────────────────────────┐
│  只在状态变化时通知一次                             │
│  必须一次性读完所有数据                             │
│                                                      │
│  优点：效率更高，减少系统调用                       │
│  缺点：编程复杂，必须使用非阻塞 I/O                 │
└─────────────────────────────────────────────────────┘
```

### 📝 练习题 3.1

**问题**：为什么 epoll 比 select 快？

<details>
<summary>点击查看答案</summary>

**答案**：

1. **fd 传递方式**：
   - select：每次调用都要把 fd 集合从用户空间复制到内核空间
   - epoll：只需要 epoll_ctl 添加一次，后续不需要复制

2. **返回方式**：
   - select：返回后需要遍历所有 fd 找到就绪的
   - epoll：只返回就绪的 fd，不需要遍历

3. **内部实现**：
   - select：线性遍历所有 fd
   - epoll：使用红黑树管理 fd，就绪链表存储就绪 fd

4. **时间复杂度**：
   - select：O(n)，n 是 fd 数量
   - epoll：O(1)，与 fd 数量无关

**实际差距**：
- 100 个连接：差距不大
- 10000 个连接：epoll 快 10-100 倍
- 100000 个连接：select 基本不可用

</details>

## 4. Go 的网络模型 ⭐⭐⭐

### 4.1 netpoller 架构

```
Go 的网络 I/O 模型：同步的写法，异步的性能

┌─────────────────────────────────────────────────────┐
│                    Go 程序                           │
│                                                      │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐             │
│  │Goroutine│  │Goroutine│  │Goroutine│             │
│  │  conn1  │  │  conn2  │  │  conn3  │             │
│  │ (阻塞)  │  │ (阻塞)  │  │ (阻塞)  │             │
│  └────┬────┘  └────┬────┘  └────┬────┘             │
│       │            │            │                   │
│       └────────────┼────────────┘                   │
│                    │                                │
│              ┌─────┴─────┐                          │
│              │ netpoller │ ← 使用 epoll/kqueue      │
│              │  (后台)   │                          │
│              └─────┬─────┘                          │
│                    │                                │
└────────────────────┼────────────────────────────────┘
                     │
              ┌──────┴──────┐
              │   内核      │
              │ epoll/kqueue│
              └─────────────┘

工作流程：
1. goroutine 调用 conn.Read()
2. 如果数据没准备好，goroutine 被挂起（不是阻塞线程！）
3. netpoller 使用 epoll 监控所有 fd
4. 数据就绪时，netpoller 唤醒对应的 goroutine
5. goroutine 继续执行
```

### 4.2 为什么 Go 的模型好？

```
传统模型（如 Java NIO）：
┌─────────────────────────────────────────────────────┐
│  // 需要手动管理 selector                           │
│  Selector selector = Selector.open();               │
│  channel.register(selector, SelectionKey.OP_READ); │
│                                                      │
│  while (true) {                                     │
│      selector.select();  // 阻塞等待               │
│      Set<SelectionKey> keys = selector.selectedKeys();│
│      for (SelectionKey key : keys) {               │
│          if (key.isReadable()) {                   │
│              // 处理读取                            │
│          }                                          │
│      }                                              │
│  }                                                  │
│                                                      │
│  问题：                                             │
│  - 代码复杂，回调地狱                               │
│  - 需要手动管理状态                                 │
│  - 容易出错                                         │
└─────────────────────────────────────────────────────┘

Go 模型：
┌─────────────────────────────────────────────────────┐
│  // 同步的写法，异步的性能                          │
│  for {                                              │
│      conn, _ := listener.Accept()                  │
│      go handleConn(conn)  // 每个连接一个 goroutine │
│  }                                                  │
│                                                      │
│  func handleConn(conn net.Conn) {                  │
│      buf := make([]byte, 1024)                     │
│      for {                                          │
│          n, _ := conn.Read(buf)  // 看起来阻塞     │
│          conn.Write(buf[:n])     // 实际是异步     │
│      }                                              │
│  }                                                  │
│                                                      │
│  优点：                                             │
│  - 代码简单直观                                     │
│  - 每个连接一个 goroutine                          │
│  - goroutine 轻量，可以创建很多                    │
│  - 底层自动使用 epoll                              │
└─────────────────────────────────────────────────────┘
```

### 4.3 Go 网络编程示例

```go
package main

import (
    "bufio"
    "fmt"
    "net"
)

func main() {
    // 监听端口
    listener, err := net.Listen("tcp", ":8080")
    if err != nil {
        panic(err)
    }
    defer listener.Close()
    
    fmt.Println("服务器启动在 :8080")
    
    for {
        // Accept 看起来是阻塞的，实际上底层使用 epoll
        conn, err := listener.Accept()
        if err != nil {
            continue
        }
        
        // 每个连接一个 goroutine
        // goroutine 很轻量（2KB 栈），可以创建很多
        go handleConn(conn)
    }
}

func handleConn(conn net.Conn) {
    defer conn.Close()
    
    reader := bufio.NewReader(conn)
    for {
        // Read 看起来是阻塞的
        // 实际上 goroutine 会让出 M，不阻塞线程
        line, err := reader.ReadString('\n')
        if err != nil {
            return
        }
        
        // Echo
        conn.Write([]byte("Echo: " + line))
    }
}
```

### 📝 练习题 4.1

**问题**：Go 的 goroutine 在等待 I/O 时，底层线程（M）在做什么？

<details>
<summary>点击查看答案</summary>

**答案**：

当 goroutine 等待 I/O 时：

1. **goroutine 被挂起**：
   - 从当前 M 上解绑
   - 状态变为 waiting
   - 被放入 netpoller 的等待队列

2. **M 继续工作**：
   - M 不会阻塞
   - M 会从 P 的本地队列获取其他 goroutine 执行
   - 如果没有 goroutine，M 可能会休眠或被回收

3. **I/O 就绪时**：
   - netpoller 检测到 fd 就绪
   - 将 goroutine 放回可运行队列
   - 某个 M 会执行这个 goroutine

**关键点**：
- goroutine 阻塞 ≠ 线程阻塞
- 一个 M 可以服务多个 goroutine
- 这就是 Go 能高效处理大量连接的原因

</details>

## 5. 文件 I/O ⭐⭐

### 5.1 标准文件操作

```go
package main

import (
    "fmt"
    "io"
    "os"
)

func main() {
    // 打开文件
    file, err := os.Open("test.txt")
    if err != nil {
        panic(err)
    }
    defer file.Close()
    
    // 读取全部
    data, _ := io.ReadAll(file)
    fmt.Println(string(data))
    
    // 写入文件
    file2, _ := os.Create("output.txt")
    defer file2.Close()
    file2.WriteString("Hello, World!")
    
    // 追加写入
    file3, _ := os.OpenFile("output.txt", os.O_APPEND|os.O_WRONLY, 0644)
    defer file3.Close()
    file3.WriteString("\nAppended line")
}
```

### 5.2 带缓冲的 I/O

```go
package main

import (
    "bufio"
    "fmt"
    "os"
)

func main() {
    // 带缓冲的读取（减少系统调用）
    file, _ := os.Open("test.txt")
    defer file.Close()
    
    reader := bufio.NewReader(file)
    
    // 按行读取
    for {
        line, err := reader.ReadString('\n')
        if err != nil {
            break
        }
        fmt.Print(line)
    }
    
    // 带缓冲的写入
    file2, _ := os.Create("output.txt")
    defer file2.Close()
    
    writer := bufio.NewWriter(file2)
    writer.WriteString("Buffered write\n")
    writer.WriteString("Another line\n")
    writer.Flush()  // 必须 Flush，否则数据可能丢失
}
```

## 6. 实战：高性能服务器 ⭐⭐⭐

### 6.1 限制并发连接数

```go
package main

import (
    "fmt"
    "net"
)

func main() {
    listener, _ := net.Listen("tcp", ":8080")
    defer listener.Close()
    
    // 使用信号量限制并发
    maxConns := 100
    semaphore := make(chan struct{}, maxConns)
    
    fmt.Printf("服务器启动，最大连接数: %d\n", maxConns)
    
    for {
        conn, err := listener.Accept()
        if err != nil {
            continue
        }
        
        // 获取信号量（如果满了会阻塞）
        semaphore <- struct{}{}
        
        go func(c net.Conn) {
            defer func() {
                c.Close()
                <-semaphore  // 释放信号量
            }()
            
            handleConn(c)
        }(conn)
    }
}

func handleConn(conn net.Conn) {
    buf := make([]byte, 1024)
    for {
        n, err := conn.Read(buf)
        if err != nil {
            return
        }
        conn.Write(buf[:n])
    }
}
```

### 6.2 带超时的连接

```go
package main

import (
    "context"
    "fmt"
    "net"
    "time"
)

func main() {
    // 带超时的连接
    ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
    defer cancel()
    
    var d net.Dialer
    conn, err := d.DialContext(ctx, "tcp", "example.com:80")
    if err != nil {
        fmt.Println("连接超时:", err)
        return
    }
    defer conn.Close()
    
    // 设置读写超时
    conn.SetDeadline(time.Now().Add(10 * time.Second))
    
    // 发送请求
    conn.Write([]byte("GET / HTTP/1.1\r\nHost: example.com\r\n\r\n"))
    
    // 读取响应
    buf := make([]byte, 1024)
    n, err := conn.Read(buf)
    if err != nil {
        fmt.Println("读取超时:", err)
        return
    }
    
    fmt.Println(string(buf[:n]))
}
```

**Q1: 五种 I/O 模型的区别？**

<details>
<summary>点击查看答案</summary>

| 模型 | 阶段1 | 阶段2 | 特点 |
|------|-------|-------|------|

| 非阻塞 I/O | 轮询 | 阻塞 | 需要轮询 |
| I/O 多路复用 | 阻塞在 select | 阻塞 | 一个线程处理多个连接 |
| 信号驱动 | 信号通知 | 阻塞 | 较少使用 |
| 异步 I/O | 不阻塞 | 不阻塞 | 真正异步 |

</details>

**Q2: epoll 为什么比 select 快？**

<details>
<summary>点击查看答案</summary>

1. **fd 传递**：select 每次复制，epoll 只需一次
2. **返回方式**：select 遍历所有，epoll 只返回就绪
3. **时间复杂度**：select O(n)，epoll O(1)
4. **数据结构**：epoll 使用红黑树和就绪链表

</details>

**Q3: Go 的网络模型是什么？**

<details>
<summary>点击查看答案</summary>

Go 使用 netpoller：
- 底层使用 epoll/kqueue
- goroutine 看起来阻塞，实际非阻塞
- goroutine 阻塞时让出 M
- 数据就绪时唤醒 goroutine
- 同步的写法，异步的性能

</details>

- 每个连接一个 goroutine
