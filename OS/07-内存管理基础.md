## 1. 为什么需要内存管理？⭐⭐⭐

### 1.1 早期的问题

```
早期：直接使用物理内存
┌─────────────────────────────────────────────────────┐
│                    物理内存                          │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐             │
│  │ 程序 A  │  │ 程序 B  │  │ 程序 C  │             │
│  │ 0-100MB │  │100-200MB│  │200-300MB│             │
│  └─────────┘  └─────────┘  └─────────┘             │
└─────────────────────────────────────────────────────┘

存在的问题：
1. 安全问题：程序 A 可以访问程序 B 的内存
2. 碎片问题：程序 B 结束后，留下 100MB 空洞
3. 连续性问题：程序 D 需要 150MB，但没有连续空间
4. 重定位问题：程序必须知道自己被加载到哪个地址
```

### 1.2 解决方案：虚拟内存

```
现代：每个进程有独立的虚拟地址空间

进程 A 的视角：              进程 B 的视角：
┌─────────────────┐         ┌─────────────────┐
│ 0x0000 - 代码   │         │ 0x0000 - 代码   │
│ 0x1000 - 数据   │         │ 0x1000 - 数据   │
│ 0x2000 - 堆     │         │ 0x2000 - 堆     │
│ ...             │         │ ...             │
│ 0xFFFF - 栈     │         │ 0xFFFF - 栈     │
└─────────────────┘         └─────────────────┘
        │                           │
        │ 页表映射                  │ 页表映射
        ↓                           ↓
┌─────────────────────────────────────────────────────┐
│                    物理内存                          │
│  实际存储位置可以不连续，由 MMU 管理                 │
└─────────────────────────────────────────────────────┘

虚拟内存的优点：
1. 隔离性：每个进程有独立地址空间，互不干扰
2. 安全性：进程无法访问其他进程的内存
3. 灵活性：物理内存可以不连续
4. 简化编程：程序不需要关心实际物理地址
```

### 📝 练习题 1.1

**问题**：为什么每个进程都能使用从 0x0000 开始的地址？这不会冲突吗？

<details>
<summary>点击查看答案</summary>

**答案**：不会冲突。因为每个进程有独立的虚拟地址空间，虚拟地址通过各自的页表映射到不同的物理地址。

举例说明：
- 进程 A 的虚拟地址 0x1000 → 物理地址 0x5000
- 进程 B 的虚拟地址 0x1000 → 物理地址 0x8000

虽然虚拟地址相同，但实际访问的物理内存完全不同。这就像两个城市都有"人民路1号"，但实际位置完全不同。

</details>

## 2. 虚拟内存详解 ⭐⭐⭐

### 2.1 进程地址空间布局

```
64 位 Linux 系统的虚拟地址空间：

高地址 0xFFFFFFFFFFFFFFFF
┌─────────────────────────────────────────────────────┐
│                    内核空间                          │
│                  （用户不可访问）                    │
│                 约 128TB                            │
├─────────────────────────────────────────────────────┤ 0x7FFFFFFFFFFF (128TB)
│                      栈 (Stack)                     │
│                    （向下增长 ↓）                   │
│                 存放局部变量、函数参数               │
│                                                      │
│                    ↓ 增长方向                       │
│                                                      │
│                    空闲区域                          │
│                （可用于 mmap）                       │
│                                                      │
│                    ↑ 增长方向                       │
│                                                      │
│                      堆 (Heap)                       │
│                    （向上增长 ↑）                   │
│                 动态分配的内存                       │
├─────────────────────────────────────────────────────┤
│                    BSS 段                            │
│                （未初始化的全局变量）                │
│                 初始化为 0                          │
├─────────────────────────────────────────────────────┤
│                    数据段 (Data)                     │
│                （已初始化的全局变量）                │
├─────────────────────────────────────────────────────┤
│                    代码段 (Text)                     │
│                  （只读，可执行）                    │
│                 存放程序指令                        │
└─────────────────────────────────────────────────────┘
低地址 0x0000000000000000
```

### 2.2 各段详解

| 段名 | 内容 | 权限 | 特点 |
|------|------|------|------|
| 代码段 | 程序指令 | 只读、可执行 | 多进程可共享 |
| 数据段 | 已初始化全局变量 | 读写 | 程序启动时分配 |
| BSS段 | 未初始化全局变量 | 读写 | 自动初始化为0 |
| 堆 | 动态分配内存 | 读写 | 向上增长 |
| 栈 | 局部变量、函数调用 | 读写 | 向下增长 |

### 2.3 Go 程序的内存布局实战

```go
package main

import (
    "fmt"
    "runtime"
    "unsafe"
)

// 全局变量 - 数据段
var globalInitVar = 100

// 未初始化全局变量 - BSS 段
var globalUninitVar int

func main() {
    // 局部变量 - 栈
    localVar := 200
    
    // 动态分配 - 堆
    heapVar := new(int)
    *heapVar = 300
    
    // 切片底层数组 - 堆
    slice := make([]int, 10)
    
    fmt.Println("=== 内存地址分析 ===")
    fmt.Printf("全局已初始化变量 (数据段): %p\n", &globalInitVar)
    fmt.Printf("全局未初始化变量 (BSS段):  %p\n", &globalUninitVar)
    fmt.Printf("局部变量 (栈):             %p\n", &localVar)
    fmt.Printf("堆分配变量:                %p\n", heapVar)
    fmt.Printf("切片底层数组:              %p\n", &slice[0])
    
    // 查看内存统计
    var m runtime.MemStats
    runtime.ReadMemStats(&m)
    
    fmt.Println("\n=== 内存统计 ===")
    fmt.Printf("堆分配内存: %.2f KB\n", float64(m.HeapAlloc)/1024)
    fmt.Printf("栈使用内存: %.2f KB\n", float64(m.StackInuse)/1024)
    fmt.Printf("系统总内存: %.2f MB\n", float64(m.Sys)/1024/1024)
    
    // 查看变量大小
    fmt.Println("\n=== 变量大小 ===")
    fmt.Printf("int 大小: %d bytes\n", unsafe.Sizeof(localVar))
    fmt.Printf("指针大小: %d bytes\n", unsafe.Sizeof(heapVar))
    fmt.Printf("slice header 大小: %d bytes\n", unsafe.Sizeof(slice))
}
```

**运行结果示例**：
```
=== 内存地址分析 ===
全局已初始化变量 (数据段): 0x5b3a20
全局未初始化变量 (BSS段):  0x5b5a60
局部变量 (栈):             0xc000018090
堆分配变量:                0xc0000180a0
切片底层数组:              0xc0000180b0

=== 内存统计 ===
堆分配内存: 0.12 KB
栈使用内存: 32.00 KB
系统总内存: 6.23 MB
```

### 📝 练习题 2.1

**问题**：观察上面的地址，为什么栈和堆的地址差距很大？

<details>
<summary>点击查看答案</summary>

**答案**：
1. 栈从高地址向低地址增长（地址以 0xc000... 开头，接近用户空间顶部）
2. 堆从低地址向高地址增长
3. 它们之间留有大量空闲空间，用于各自的增长
4. 这种设计可以最大化利用地址空间，避免栈和堆相互覆盖

在 Go 中，goroutine 的栈是在堆上分配的，所以地址可能看起来相近，但原理相同。

</details>

### 📝 练习题 2.2

**问题**：32 位系统最大能寻址多少内存？为什么实际可用内存更少？

<details>
<summary>点击查看答案</summary>

**答案**：
1. 32 位系统最大寻址：2^32 = 4GB
2. 实际可用更少的原因：
   - 内核空间占用约 1GB（Linux 默认 3:1 分配）
   - 用户空间只有约 3GB
   - 还要减去代码段、数据段等固定占用
   - 实际可用堆内存通常只有 2GB 左右

这就是为什么大内存应用需要 64 位系统。

</details>

## 3. 分页机制 ⭐⭐⭐

### 3.1 什么是分页？

```
分页：将内存分成固定大小的块（页）

为什么要分页？
- 解决外部碎片问题
- 简化内存管理
- 支持虚拟内存

术语：
- 页（Page）：虚拟内存的基本单位，通常 4KB
- 帧（Frame）：物理内存的基本单位，与页大小相同
- 页表（Page Table）：虚拟页到物理帧的映射表

虚拟内存                    物理内存
┌─────────┐                ┌─────────┐
│  页 0   │ ──────────────→│  帧 3   │
├─────────┤                ├─────────┤
│  页 1   │ ──────────────→│  帧 7   │
├─────────┤                ├─────────┤
│  页 2   │ ──────────────→│  帧 1   │
├─────────┤                ├─────────┤
│  页 3   │ ──────────────→│  帧 5   │
└─────────┘                └─────────┘

注意：物理帧不需要连续！
```

### 3.2 页表结构

```
页表项（PTE - Page Table Entry）的结构：

┌─────────────────────────────────────────────────────┐
│                    页表项 (64位)                     │
├──────────┬──────────┬──────────┬───────────────────┤
│ 物理帧号 │ 有效位   │ 权限位   │ 其他标志位        │
│ (PFN)    │ (Valid)  │ (R/W/X)  │ (Dirty/Access等)  │
├──────────┼──────────┼──────────┼───────────────────┤
│ 52 bits  │ 1 bit    │ 3 bits   │ 8 bits            │
└──────────┴──────────┴──────────┴───────────────────┘

各字段含义：
- 物理帧号：映射到的物理内存帧
- 有效位：1=已映射，0=未映射（访问会触发缺页）
- 权限位：R=可读，W=可写，X=可执行
- Dirty位：页面是否被修改过
- Access位：页面是否被访问过（用于页面置换）
```

### 3.3 地址转换过程

```
虚拟地址 → 物理地址 的转换

假设：页大小 = 4KB = 4096 = 2^12 字节

虚拟地址: 0x12345

步骤1：分解虚拟地址
┌────────────────────────────────────────┐
│         虚拟地址: 0x12345              │
│         二进制: 0001 0010 0011 0100 0101│
├────────────────┬───────────────────────┤
│    页号        │      页内偏移         │
│    0x12        │      0x345            │
│   (高位)       │     (低12位)          │
└────────────────┴───────────────────────┘

步骤2：查页表
页号 0x12 → 查页表 → 物理帧号 0x7

步骤3：计算物理地址
物理地址 = 帧号 × 页大小 + 偏移
        = 0x7 × 0x1000 + 0x345
        = 0x7000 + 0x345
        = 0x7345
```

### 3.4 多级页表

```
问题：单级页表太大！

计算：64位地址空间，4KB页
- 虚拟地址空间：2^64 字节
- 页数：2^64 / 2^12 = 2^52 页
- 页表大小：2^52 × 8字节 = 32PB！

解决方案：多级页表（按需分配）

Linux x86-64 使用 4 级页表：

虚拟地址（48位有效）：
┌─────────┬─────────┬─────────┬─────────┬──────────┐
│  PML4   │  PDPT   │   PD    │   PT    │  Offset  │
│  9位    │  9位    │  9位    │  9位    │  12位    │
│ (索引)  │ (索引)  │ (索引)  │ (索引)  │ (偏移)   │
└─────────┴─────────┴─────────┴─────────┴──────────┘

地址转换流程：
┌─────────┐
│  CR3    │ ← CPU 寄存器，存放 PML4 基地址
└────┬────┘
     │
     ↓
┌─────────┐
│  PML4   │ ← 页全局目录（512项）
│ [PML4索引]
└────┬────┘
     │
     ↓
┌─────────┐
│  PDPT   │ ← 页目录指针表（512项）
│ [PDPT索引]
└────┬────┘
     │
     ↓
┌─────────┐
│   PD    │ ← 页目录（512项）
│ [PD索引]
└────┬────┘
     │
     ↓
┌─────────┐
│   PT    │ ← 页表（512项）
│ [PT索引]
└────┬────┘
     │
     ↓
┌─────────┐
│  物理帧  │ + Offset = 物理地址
└─────────┘

优点：
- 只分配使用到的页表
- 大大减少内存占用
- 支持稀疏地址空间
```

### 📝 练习题 3.1

**问题**：一个进程只使用了 1GB 内存，使用 4 级页表需要多少页表内存？

<details>
<summary>点击查看答案</summary>

**答案**：

计算过程：
1. 1GB = 2^30 字节
2. 页数 = 2^30 / 4KB = 2^30 / 2^12 = 2^18 = 262144 页
3. 每个页表有 512 项（2^9）

需要的页表：
- PT（页表）：262144 / 512 = 512 个
- PD（页目录）：512 / 512 = 1 个
- PDPT：1 个
- PML4：1 个

总计：512 + 1 + 1 + 1 = 515 个页表
每个页表 4KB，总共：515 × 4KB ≈ 2MB

相比单级页表的 32PB，节省了巨大的空间！

</details>

### 📝 练习题 3.2

**问题**：为什么页大小通常是 4KB？太大或太小有什么问题？

<details>
<summary>点击查看答案</summary>

**答案**：

4KB 是一个平衡点：

**页太小（如 1KB）的问题**：
- 页表项太多，页表占用内存大
- 地址转换开销增加
- TLB 命中率降低

**页太大（如 1MB）的问题**：
- 内部碎片严重（分配 1 字节也要 1MB）
- 内存利用率低
- 不适合小程序

**4KB 的优点**：
- 与磁盘扇区大小（512B/4KB）匹配
- 页表大小适中
- 内部碎片可接受
- 历史兼容性好

现代系统也支持大页（2MB/1GB），用于大内存应用。

</details>

## 4. TLB 缓存 ⭐⭐⭐

### 4.1 TLB 是什么？

```
TLB = Translation Lookaside Buffer（地址转换后备缓冲器）

问题：每次内存访问都要查 4 级页表，太慢了！
- 访问一次数据需要 5 次内存访问（4次页表 + 1次数据）

解决：TLB 缓存最近的页表项

┌─────────────────────────────────────────────────────┐
│                      CPU                             │
│  ┌─────────────────────────────────────────────┐    │
│  │                   TLB                        │    │
│  │  ┌──────────┬──────────┬──────────┐         │    │
│  │  │ 虚拟页号 │ 物理帧号 │ 权限     │         │    │
│  │  ├──────────┼──────────┼──────────┤         │    │
│  │  │   0x12   │   0x7    │  R W -   │         │    │
│  │  │   0x34   │   0x9    │  R - X   │         │    │
│  │  │   0x56   │   0x3    │  R W -   │         │    │
│  │  │   ...    │   ...    │  ...     │         │    │
│  │  └──────────┴──────────┴──────────┘         │    │
│  │  通常 64-1024 项                             │    │
│  └─────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────┘

访问流程：
1. CPU 发出虚拟地址
2. 先查 TLB
   - 命中：直接得到物理地址（1个时钟周期）
   - 未命中：查页表，更新 TLB（几十到几百个时钟周期）
3. 访问物理内存
```

### 4.2 TLB 命中率的重要性

```
假设：
- TLB 命中时间：1ns
- 内存访问时间：100ns
- TLB 命中率：99%

平均访问时间 = 0.99 × 1ns + 0.01 × (100ns × 5)
            = 0.99ns + 5ns
            = 5.99ns

如果 TLB 命中率降到 90%：
平均访问时间 = 0.90 × 1ns + 0.10 × 500ns
            = 0.9ns + 50ns
            = 50.9ns

性能差了 8 倍！
```

### 4.3 TLB 友好的代码

```go
package main

import (
    "fmt"
    "time"
)

const SIZE = 1024 * 1024 // 1M 个元素

func main() {
    arr := make([]int, SIZE)
    
    // 测试1：顺序访问（TLB 友好）
    start := time.Now()
    for i := 0; i < SIZE; i++ {
        arr[i] = i
    }
    fmt.Println("顺序访问:", time.Since(start))
    
    // 测试2：跨页访问（TLB 不友好）
    start = time.Now()
    pageSize := 4096 / 8 // 每页 512 个 int64
    for i := 0; i < SIZE; i++ {
        idx := (i * pageSize) % SIZE // 每次跳到不同的页
        arr[idx] = i
    }
    fmt.Println("跨页访问:", time.Since(start))
    
    // 测试3：随机访问（最差情况）
    start = time.Now()
    for i := 0; i < SIZE; i++ {
        idx := (i * 997) % SIZE // 伪随机访问
        arr[idx] = i
    }
    fmt.Println("随机访问:", time.Since(start))
}
```

**运行结果示例**：
```
顺序访问: 2.1ms
跨页访问: 15.3ms
随机访问: 12.8ms
```

### 📝 练习题 4.1

**问题**：为什么顺序访问比随机访问快这么多？

<details>
<summary>点击查看答案</summary>

**答案**：

1. **TLB 命中率**：
   - 顺序访问：同一页内的多次访问只需一次 TLB 查询
   - 随机访问：每次可能访问不同的页，TLB 频繁未命中

2. **CPU 缓存**：
   - 顺序访问：利用空间局部性，缓存行预取有效
   - 随机访问：缓存行频繁失效

3. **内存预取**：
   - 顺序访问：CPU 可以预测下一次访问，提前加载
   - 随机访问：无法预测，每次都要等待

**工作中的应用**：
- 遍历数组时尽量顺序访问
- 避免使用链表（随机访问）
- 数据结构设计考虑内存布局

</details>

## 5. 页面置换算法 ⭐⭐⭐

### 5.1 缺页中断

```
当访问的页不在物理内存中时，触发缺页中断（Page Fault）

缺页处理流程：
┌─────────────────────────────────────────────────────┐
│ 1. CPU 访问虚拟地址                                 │
│ 2. MMU 查页表，发现有效位=0                         │
│ 3. 触发缺页中断，CPU 切换到内核态                   │
│ 4. 操作系统处理：                                   │
│    a. 找到页面在磁盘上的位置                        │
│    b. 如果物理内存满，选择一个页面换出              │
│    c. 从磁盘加载页面到物理内存                      │
│    d. 更新页表                                      │
│ 5. 返回用户态，重新执行指令                         │
└─────────────────────────────────────────────────────┘

缺页的代价：
- 磁盘 I/O：约 10ms
- 内存访问：约 100ns
- 差距：100000 倍！

所以要尽量减少缺页！
```

### 5.2 FIFO 算法（先进先出）

```
原理：最先进入内存的页面最先被换出

页面访问序列：1, 2, 3, 4, 1, 2, 5, 1, 2, 3, 4, 5
内存容量：3 页

时间  访问  内存状态        缺页?  说明
─────────────────────────────────────────────────
1     1    [1, -, -]        ✓     页1进入
2     2    [1, 2, -]        ✓     页2进入
3     3    [1, 2, 3]        ✓     页3进入，内存满
4     4    [4, 2, 3]        ✓     页1最老，换出，页4进入
5     1    [4, 1, 3]        ✓     页2最老，换出，页1进入
6     2    [4, 1, 2]        ✓     页3最老，换出，页2进入

9     2    [5, 1, 2]        ✗     页2在内存中
10    3    [5, 3, 2]        ✓     页1最老，换出，页3进入

12    5    [5, 3, 4]        ✗     页5在内存中

缺页次数：9 次
缺页率：9/12 = 75%

FIFO 的问题：Belady 异常
- 增加内存反而可能增加缺页！
```

### 5.3 LRU 算法（最近最少使用）⭐⭐⭐

```
原理：最长时间未被访问的页面被换出

页面访问序列：1, 2, 3, 4, 1, 2, 5, 1, 2, 3, 4, 5
内存容量：3 页

时间  访问  内存状态        缺页?  说明
─────────────────────────────────────────────────
1     1    [1, -, -]        ✓     页1进入
2     2    [1, 2, -]        ✓     页2进入
3     3    [1, 2, 3]        ✓     页3进入
4     4    [4, 2, 3]        ✓     页1最久未用，换出
5     1    [4, 1, 3]        ✓     页2最久未用，换出
6     2    [4, 1, 2]        ✓     页3最久未用，换出
7     5    [5, 1, 2]        ✓     页4最久未用，换出
8     1    [5, 1, 2]        ✗     命中，更新访问时间
9     2    [5, 1, 2]        ✗     命中，更新访问时间
10    3    [3, 1, 2]        ✓     页5最久未用，换出
11    4    [3, 4, 2]        ✓     页1最久未用，换出
12    5    [3, 4, 5]        ✓     页2最久未用，换出

缺页次数：10 次
缺页率：10/12 = 83%

注意：这个例子中 LRU 不如 FIFO，但通常 LRU 更好
```

### 5.4 Go 实现 LRU 缓存

```go
package main

import (
    "container/list"
    "fmt"
)

// LRUCache LRU 缓存实现
type LRUCache struct {
    capacity int
    cache    map[int]*list.Element
    list     *list.List
}

type entry struct {
    key   int
    value int
}

// NewLRUCache 创建 LRU 缓存
func NewLRUCache(capacity int) *LRUCache {
    return &LRUCache{
        capacity: capacity,
        cache:    make(map[int]*list.Element),
        list:     list.New(),
    }
}

// Get 获取值
func (c *LRUCache) Get(key int) (int, bool) {
    if elem, ok := c.cache[key]; ok {
        // 移到链表头部（最近使用）
        c.list.MoveToFront(elem)
        return elem.Value.(*entry).value, true
    }
    return 0, false
}

// Put 设置值
func (c *LRUCache) Put(key, value int) {
    // 如果 key 已存在，更新值并移到头部
    if elem, ok := c.cache[key]; ok {
        c.list.MoveToFront(elem)
        elem.Value.(*entry).value = value
        return
    }
    
    // 如果容量满了，删除最久未使用的（链表尾部）
    if c.list.Len() >= c.capacity {
        oldest := c.list.Back()
        if oldest != nil {
            c.list.Remove(oldest)
            delete(c.cache, oldest.Value.(*entry).key)
        }
    }
    
    // 添加新元素到头部
    elem := c.list.PushFront(&entry{key, value})
    c.cache[key] = elem
}

// 打印缓存状态
func (c *LRUCache) Print() {
    fmt.Print("缓存状态: [")
    for e := c.list.Front(); e != nil; e = e.Next() {
        ent := e.Value.(*entry)
        fmt.Printf("%d:%d ", ent.key, ent.value)
    }
    fmt.Println("]")
}

func main() {
    cache := NewLRUCache(3)
    
    fmt.Println("=== LRU 缓存演示 ===")
    
    cache.Put(1, 100)
    cache.Print() // [1:100]
    
    cache.Put(2, 200)
    cache.Print() // [2:200 1:100]
    
    cache.Put(3, 300)
    cache.Print() // [3:300 2:200 1:100]
    
    // 访问 key=1，移到头部
    val, _ := cache.Get(1)
    fmt.Printf("Get(1) = %d\n", val)
    cache.Print() // [1:100 3:300 2:200]
    
    // 添加 key=4，淘汰最久未用的 key=2
    cache.Put(4, 400)
    fmt.Println("Put(4, 400) - 淘汰 key=2")
    cache.Print() // [4:400 1:100 3:300]
    
    // 验证 key=2 已被淘汰
    _, ok := cache.Get(2)
    fmt.Printf("Get(2) exists: %v\n", ok) // false
}
```

| LRU | 最近最少使用 | 性能好 | 实现复杂 |

package main

import (
    "container/list"
    "fmt"
    "sync"
    "time"
)

type LRUCacheWithTTL struct {
    capacity int
    cache    map[int]*list.Element
    list     *list.List
    mu       sync.RWMutex
}

type entryWithTTL struct {
    key       int
    value     int
    expireAt  time.Time
}

func NewLRUCacheWithTTL(capacity int) *LRUCacheWithTTL {
    return &LRUCacheWithTTL{
        capacity: capacity,
        cache:    make(map[int]*list.Element),
        list:     list.New(),
    }
}

func (c *LRUCacheWithTTL) Get(key int) (int, bool) {
    c.mu.Lock()
    defer c.mu.Unlock()
    
    if elem, ok := c.cache[key]; ok {
        ent := elem.Value.(*entryWithTTL)
        
        // 检查是否过期
        if time.Now().After(ent.expireAt) {
            c.list.Remove(elem)
            delete(c.cache, key)
            return 0, false
        }
        
        c.list.MoveToFront(elem)
        return ent.value, true
    }
    return 0, false
}

func (c *LRUCacheWithTTL) Put(key, value int, ttl time.Duration) {
    c.mu.Lock()
    defer c.mu.Unlock()
    
    expireAt := time.Now().Add(ttl)
    
    if elem, ok := c.cache[key]; ok {
        c.list.MoveToFront(elem)
        ent := elem.Value.(*entryWithTTL)
        ent.value = value
        ent.expireAt = expireAt
        return
    }
    
    if c.list.Len() >= c.capacity {
        oldest := c.list.Back()
        if oldest != nil {
            c.list.Remove(oldest)
            delete(c.cache, oldest.Value.(*entryWithTTL).key)
        }
    }
    
    elem := c.list.PushFront(&entryWithTTL{key, value, expireAt})
    c.cache[key] = elem
}

func main() {
    cache := NewLRUCacheWithTTL(3)
    
    cache.Put(1, 100, 2*time.Second)
    cache.Put(2, 200, 5*time.Second)
    
    val, ok := cache.Get(1)
    fmt.Printf("Get(1): %d, %v\n", val, ok) // 100, true
    
    time.Sleep(3 * time.Second)
    
    val, ok = cache.Get(1)
    fmt.Printf("Get(1) after 3s: %d, %v\n", val, ok) // 0, false (已过期)
    
    val, ok = cache.Get(2)
    fmt.Printf("Get(2) after 3s: %d, %v\n", val, ok) // 200, true (未过期)
}

## 6. 内存映射（mmap）⭐⭐

### 6.1 什么是 mmap？

```
mmap = 将文件映射到进程的虚拟地址空间

传统文件 I/O：
┌─────────┐    read()    ┌─────────┐    copy    ┌─────────┐
│  磁盘   │ ──────────→ │ 内核缓冲 │ ────────→ │ 用户缓冲 │
└─────────┘             └─────────┘            └─────────┘
                        （Page Cache）
需要两次数据拷贝！

mmap 方式：
┌─────────┐    映射    ┌─────────────────────────────────┐
│  磁盘   │ ────────→ │        共享内存区域              │
└─────────┘           │  （用户空间直接访问 Page Cache） │
                      └─────────────────────────────────┘
只需要一次拷贝（磁盘到 Page Cache）

mmap 的优点：
1. 减少数据拷贝
2. 可以像访问内存一样访问文件
3. 多进程可以共享同一映射
4. 延迟加载（按需加载页面）

mmap 的缺点：
1. 小文件效率不高（页对齐开销）
2. 不适合频繁修改的文件
3. 需要处理信号（SIGBUS）
```

### 6.2 Go 中使用 mmap

```go
package main

import (
    "fmt"
    "os"
    "syscall"
)

func main() {
    // 创建测试文件
    filename := "mmap_test.dat"
    file, err := os.OpenFile(filename, os.O_RDWR|os.O_CREATE, 0666)
    if err != nil {
        panic(err)
    }
    defer file.Close()
    defer os.Remove(filename)
    
    // 设置文件大小（必须先设置）
    size := 4096
    if err := file.Truncate(int64(size)); err != nil {
        panic(err)
    }
    
    // 内存映射
    data, err := syscall.Mmap(
        int(file.Fd()),           // 文件描述符
        0,                        // 偏移量
        size,                     // 映射大小
        syscall.PROT_READ|syscall.PROT_WRITE, // 权限
        syscall.MAP_SHARED,       // 共享映射（修改会写回文件）
    )
    if err != nil {
        panic(err)
    }
    defer syscall.Munmap(data)
    
    // 像操作内存一样操作文件
    message := "Hello, mmap! 这是内存映射测试。"
    copy(data, message)
    
    fmt.Println("写入内容:", message)
    
    // 强制同步到磁盘
    if err := syscall.Msync(data, syscall.MS_SYNC); err != nil {
        panic(err)
    }
    
    // 读取验证
    fmt.Println("读取内容:", string(data[:len(message)]))
    
    // 修改内容
    data[0] = 'h' // 小写
    fmt.Println("修改后:", string(data[:len(message)]))
}
```

### 6.3 mmap 的应用场景

```
1. 大文件处理
   - 数据库文件（SQLite、LevelDB）
   - 日志文件分析
   - 不需要一次性加载整个文件

2. 进程间通信
   - 共享内存
   - 比管道、Socket 更快

3. 动态链接库加载
   - .so/.dll 文件映射到内存
   - 多进程共享代码段

4. 零拷贝传输
   - sendfile() 配合 mmap
   - 网络文件传输
```

<details>

1. 大文件随机访问（如数据库）

### 📝 练习题 7.1

**问题**：VSZ 和 RSS 有什么区别？为什么 VSZ 通常比 RSS 大很多？

<details>
<summary>点击查看答案</summary>

**答案**：

**VSZ（Virtual Size）**：
- 进程的虚拟地址空间大小
- 包括所有映射的内存（代码、数据、共享库、mmap等）
- 不一定都在物理内存中

**RSS（Resident Set Size）**：
- 实际占用的物理内存
- 只包括当前在物理内存中的页面
- 不包括换出到 swap 的页面

**VSZ > RSS 的原因**：
1. 延迟分配：malloc 后不立即分配物理内存
2. 共享库：多进程共享，但 VSZ 各自计算
3. mmap 文件：映射但未访问的部分
4. 换出的页面：在 swap 中，不计入 RSS

**工作中的应用**：
- 监控 RSS 来判断实际内存使用
- VSZ 过大可能表示内存泄漏（虚拟地址空间耗尽）
- 容器限制通常基于 RSS

</details>

## 8. 内存相关面试题 ⭐⭐⭐

### 8.1 常见面试题

**Q1: 什么是虚拟内存？有什么好处？**

<details>
<summary>点击查看答案</summary>

虚拟内存是操作系统提供的一种内存管理技术，为每个进程提供独立的虚拟地址空间。

**好处**：
1. **隔离性**：进程间内存隔离，提高安全性
2. **简化编程**：程序不需要关心物理地址
3. **内存扩展**：可以使用比物理内存更大的地址空间（通过 swap）
4. **共享内存**：多进程可以共享同一物理页面
5. **按需分配**：延迟分配，提高内存利用率

</details>

**Q2: 页表是什么？为什么需要多级页表？**

<details>
<summary>点击查看答案</summary>

**页表**：存储虚拟页号到物理帧号映射的数据结构。

**为什么需要多级页表**：
- 单级页表太大：64位地址空间需要 32PB 页表
- 多级页表按需分配：只分配使用到的页表
- 节省内存：1GB 内存只需约 2MB 页表

**多级页表的缺点**：
- 地址转换需要多次内存访问
- 通过 TLB 缓存来弥补

</details>

**Q3: 什么是 TLB？TLB 未命中会怎样？**

<details>
<summary>点击查看答案</summary>

**TLB（Translation Lookaside Buffer）**：
- 页表的高速缓存
- 存储最近使用的虚拟页到物理帧的映射
- 通常有 64-1024 项

**TLB 未命中的处理**：
1. 硬件遍历页表（x86）或触发异常（MIPS）
2. 找到映射后更新 TLB
3. 重新执行指令

**TLB 未命中的代价**：
- 需要访问多级页表（4次内存访问）
- 比 TLB 命中慢 100 倍以上

</details>

**Q4: 什么是缺页中断？如何减少缺页？**

<details>
<summary>点击查看答案</summary>

**缺页中断**：访问的页面不在物理内存中时触发的中断。

**处理流程**：
1. 保存当前状态
2. 从磁盘加载页面
3. 更新页表
4. 恢复执行

**减少缺页的方法**：
1. 增加物理内存
2. 优化程序的内存访问模式（提高局部性）
3. 使用大页（减少页表项）
4. 预取（prefetch）
5. 避免内存泄漏

</details>
