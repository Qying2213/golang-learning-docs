# CI/CD 基础概念 ⭐⭐⭐

> 本章目标：理解 CI/CD 是什么、为什么需要、核心流程是怎样的

---

## 1. 什么是 CI/CD？

### 1.1 CI - 持续集成（Continuous Integration）

**定义：** 开发人员频繁地将代码合并到主分支，每次合并都会自动触发构建和测试。

```
开发者A 写代码 → push → 自动测试 → 合并到主分支
                         ↓
开发者B 写代码 → push → 自动测试 → 合并到主分支
                         ↓
                    代码始终保持可用状态
```

**CI 解决的问题：**
- ❌ 没有CI：代码合并时才发现冲突，一堆bug
- ✅ 有CI：每次提交都测试，问题早发现早解决

### 1.2 CD - 持续交付/持续部署

**持续交付（Continuous Delivery）：**
- 代码随时可以部署到生产环境
- 但需要**手动点击**才会真正部署

**持续部署（Continuous Deployment）：**
- 代码通过测试后**自动部署**到生产环境
- 完全自动化，无需人工干预

```
CI（持续集成）
    ↓
CD（持续交付）→ 手动点击部署
    ↓
CD（持续部署）→ 全自动部署
```

## 2. 为什么需要 CI/CD？

### 2.1 传统开发流程的痛点

```
场景：小明开发一个功能

1. 写代码（2小时）
2. 本地测试（30分钟）
3. 打包项目（10分钟）
4. 登录服务器（5分钟）
5. 上传文件（10分钟）
6. 停止旧服务（2分钟）
7. 启动新服务（2分钟）
8. 验证是否正常（10分钟）

总计：约3小时，其中1小时在做重复的部署工作
```

**问题：**
- 🔴 浪费时间：每次部署都要重复操作
- 🔴 容易出错：手动操作难免失误
- 🔴 难以回滚：出问题了不知道怎么快速恢复
- 🔴 团队协作难：多人开发时代码冲突频繁

### 2.2 CI/CD 解决方案

```
场景：小明用 CI/CD 开发

1. 写代码（2小时）
2. git push（10秒）
3. 喝杯咖啡，等通知（自动完成测试、打包、部署）
4. 收到成功通知，验证功能（10分钟）

总计：约2小时10分钟，节省50分钟
```

**好处：**
- ✅ 节省时间：自动化重复工作
- ✅ 减少错误：机器比人可靠
- ✅ 快速回滚：一键恢复到上个版本
- ✅ 团队协作：代码冲突早发现

## 3. CI/CD 核心流程

### 3.1 完整流水线

```
┌─────────────────────────────────────────────────────────────────┐
│                        CI/CD 流水线                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  ┌──────┐    ┌──────┐    ┌──────┐    ┌──────┐    ┌──────┐      │
│  │ 代码 │ →  │ 构建 │ →  │ 测试 │ →  │ 打包 │ →  │ 部署 │      │
│  │ 提交 │    │ Build│    │ Test │    │ Pack │    │Deploy│      │
│  └──────┘    └──────┘    └──────┘    └──────┘    └──────┘      │
│      ↓           ↓           ↓           ↓           ↓          │
│   git push   编译代码    运行测试   Docker镜像  发布上线        │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 3.2 每个阶段做什么？

#### 阶段1：代码提交（Source）
```bash
git add .
git commit -m "添加用户登录功能"
git push origin main
```
- 触发条件：push 到指定分支
- 做的事情：拉取最新代码

#### 阶段2：构建（Build）
```bash
# Go 项目
go build -o app main.go

# 前端项目
npm install
npm run build
```
- 做的事情：编译代码，安装依赖
- 失败原因：语法错误、依赖缺失

#### 阶段3：测试（Test）
```bash
# Go 项目
go test ./...

# 前端项目
npm run test
```
- 做的事情：运行单元测试、集成测试
- 失败原因：测试用例不通过

#### 阶段4：打包（Package）
```bash
# 构建 Docker 镜像
docker build -t myapp:v1.0 .

# 推送到镜像仓库
docker push myapp:v1.0
```
- 做的事情：打包成可部署的产物
- 产物：Docker 镜像、二进制文件、压缩包

#### 阶段5：部署（Deploy）
```bash
# 拉取新镜像
docker pull myapp:v1.0

# 停止旧容器，启动新容器
docker stop myapp
docker run -d --name myapp myapp:v1.0
```
- 做的事情：把新版本发布到服务器
- 环境：测试环境、预发环境、生产环境

## 4. 常见 CI/CD 工具

### 4.1 工具对比

| 工具 | 特点 | 适用场景 |
|------|------|----------|
| **GitHub Actions** | 免费、与GitHub深度集成 | GitHub项目首选 ⭐推荐 |
| **GitLab CI** | 功能强大、自带仓库 | 企业私有部署 |
| **Jenkins** | 老牌、插件丰富 | 复杂定制需求 |
| **Travis CI** | 简单易用 | 开源项目 |
| **CircleCI** | 速度快 | 中小项目 |

### 4.2 为什么选 GitHub Actions？

对于你来说，GitHub Actions 是最佳选择：

1. **免费**：公开仓库完全免费，私有仓库每月2000分钟免费额度
2. **简单**：配置文件就是 YAML，学习成本低
3. **集成好**：和 GitHub 无缝配合
4. **生态丰富**：大量现成的 Action 可以直接用

## 5. 实际案例理解

### 5.1 场景：Go Web 项目的 CI/CD

```
你的 Go 项目结构：
├── main.go
├── handler/
├── model/
├── go.mod
├── Dockerfile
└── .github/
    └── workflows/
        └── ci.yml      ← CI/CD 配置文件
```

**当你 push 代码后会发生什么：**

```
1. GitHub 检测到 push 事件
        ↓
2. 读取 .github/workflows/ci.yml 配置
        ↓
3. 启动一台云服务器（GitHub提供）
        ↓
4. 拉取你的代码
        ↓
5. 执行 go build 编译
        ↓
6. 执行 go test 测试
        ↓
7. 执行 docker build 构建镜像
        ↓
8. 推送镜像到 Docker Hub
        ↓
9. SSH 登录你的服务器，拉取新镜像，重启服务
        ↓
10. 发送通知（成功/失败）
```

### 5.2 配置文件预览

```yaml
# .github/workflows/ci.yml
name: CI/CD Pipeline

on:
  push:
    branches: [ main ]

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    
    steps:
    - name: 拉取代码
      uses: actions/checkout@v3
    
    - name: 设置 Go 环境
      uses: actions/setup-go@v4
      with:
        go-version: '1.21'
    
    - name: 编译
      run: go build -o app
    
    - name: 测试
      run: go test ./...
    
    - name: 构建 Docker 镜像
      run: docker build -t myapp .
    
    - name: 部署
      run: echo "部署到服务器..."
```

## 6. 练习题

### 练习1：概念理解（选择题）

**Q1：CI 的主要目的是什么？**
- A. 自动部署到生产环境
- B. 频繁集成代码并自动测试
- C. 构建 Docker 镜像
- D. 管理服务器

<details>
<summary>查看答案</summary>

**答案：B**

CI（持续集成）的核心是让开发者频繁地将代码合并到主分支，每次合并都自动触发构建和测试，尽早发现问题。

</details>

**Q2：持续交付和持续部署的区别是什么？**
- A. 没有区别
- B. 持续交付需要手动触发部署，持续部署完全自动
- C. 持续部署需要手动触发，持续交付完全自动
- D. 持续交付只做测试，持续部署只做部署

<details>
<summary>查看答案</summary>

**答案：B**

- 持续交付（Delivery）：代码随时可部署，但需要手动点击确认
- 持续部署（Deployment）：测试通过后自动部署，无需人工干预

</details>

**Q3：以下哪个不是 CI/CD 的好处？**
- A. 减少手动操作错误
- B. 加快发布速度
- C. 代码写得更好
- D. 问题早发现早解决

<details>
<summary>查看答案</summary>

**答案：C**

CI/CD 不能让你的代码写得更好，它只是自动化了构建、测试、部署的流程。代码质量还是取决于开发者本身。

</details>

### 练习3：场景分析

**场景：你的团队有3个开发者，每天都要部署2-3次。目前是手动部署，每次需要30分钟。**

问题：
1. 一周（5天）花在部署上的时间是多少？
2. 如果用 CI/CD，假设配置好后每次部署只需要5分钟人工确认，能节省多少时间？

<details>
<summary>查看答案</summary>

1. 手动部署时间：
   - 每天：2.5次 × 30分钟 = 75分钟
   - 每周：75分钟 × 5天 = 375分钟 ≈ 6.25小时

2. CI/CD 后：
   - 每天：2.5次 × 5分钟 = 12.5分钟
   - 每周：12.5分钟 × 5天 = 62.5分钟 ≈ 1小时
   - 节省：6.25 - 1 = 5.25小时/周

一年节省：5.25 × 52 = 273小时 ≈ 34个工作日！

</details>

## 7. 本章小结

```
CI/CD 核心概念：
├── CI（持续集成）
│   └── 频繁合并代码 + 自动测试
├── CD（持续交付）
│   └── 随时可部署 + 手动确认
└── CD（持续部署）
    └── 全自动部署

CI/CD 流水线：
代码提交 → 构建 → 测试 → 打包 → 部署

推荐工具：GitHub Actions（免费、简单、生态好）
```
