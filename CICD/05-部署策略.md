# 部署策略 ⭐⭐

> 本章目标：了解常见的部署策略，知道什么场景用什么策略

---

## 1. 为什么需要部署策略？

**场景：** 你的服务有 1000 个用户在用，现在要发布新版本

```
最简单的做法：
停止旧版本 → 启动新版本
     ↓
问题：停机期间用户无法访问（可能几秒到几分钟）

更好的做法：
使用部署策略，实现零停机或最小影响
```

---

## 2. 常见部署策略对比

| 策略 | 停机时间 | 回滚速度 | 资源消耗 | 复杂度 | 适用场景 |
|------|----------|----------|----------|--------|----------|
| 停机部署 | 有 | 慢 | 低 | 简单 | 开发环境 |
| 滚动更新 | 无 | 中 | 中 | 中 | 大多数场景 |
| 蓝绿部署 | 无 | 快 | 高 | 中 | 需要快速回滚 |
| 金丝雀发布 | 无 | 快 | 中 | 高 | 大规模服务 |

---

## 3. 停机部署（Recreate）

最简单的方式：停止旧版本，启动新版本。

```bash
docker stop myapp
docker rm myapp
docker pull myapp:v2
docker run -d --name myapp myapp:v2
```

**适用：** 开发环境、用户量很少的服务

---

## 4. 滚动更新（Rolling Update）

逐步替换旧实例：

```
初始：[v1] [v1] [v1] [v1]
  ↓
步骤1：[v2] [v1] [v1] [v1]
  ↓
步骤2：[v2] [v2] [v1] [v1]
  ↓
最终：[v2] [v2] [v2] [v2]
```

**优点：** 零停机，资源消耗适中
**适用：** 大多数 Web 服务

---

## 5. 蓝绿部署（Blue-Green）

同时运行两套环境，通过负载均衡切换：

```
负载均衡器
    │
    ├── 蓝环境 (v1) ← 当前流量
    │
    └── 绿环境 (v2) ← 待命

部署：在绿环境部署 v2 → 测试 → 切换流量
回滚：把流量切回蓝环境（秒级）
```

**优点：** 回滚极快
**缺点：** 需要双倍资源

---

## 6. 金丝雀发布（Canary）

先让一小部分用户使用新版本：

```
阶段1：5% 流量到 v2，95% 到 v1
阶段2：25% 流量到 v2
阶段3：100% 流量到 v2
```

**优点：** 风险最小，可以收集真实反馈
**适用：** 大规模用户的服务

---

## 7. 选择建议

| 阶段 | 推荐策略 |
|------|----------|
| 学习/开发 | 停机部署 |
| 小项目上线 | 滚动更新 |
| 正式生产 | 蓝绿部署 |
| 大规模服务 | 金丝雀发布 |
