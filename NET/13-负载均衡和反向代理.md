## 1. 负载均衡基础 ⭐⭐⭐

### 1.1 什么是负载均衡

```
负载均衡 = 将请求分发到多个服务器

为什么需要：
1. 单台服务器性能有限
2. 单点故障风险
3. 需要水平扩展

架构演进：
单机 → 负载均衡 + 多台服务器

         ┌─────────────┐
         │   客户端     │
         └──────┬──────┘
                │
         ┌──────▼──────┐
         │  负载均衡器   │
         └──────┬──────┘
                │
    ┌───────────┼───────────┐
    │           │           │
┌───▼───┐  ┌───▼───┐  ┌───▼───┐
│服务器1 │  │服务器2 │  │服务器3 │
└───────┘  └───────┘  └───────┘
```

### 1.2 负载均衡的层次

```
四层负载均衡（传输层）：
- 基于 IP + 端口
- 不解析应用层协议
- 性能高
- 例：LVS、F5

七层负载均衡（应用层）：
- 基于 HTTP 协议
- 可以根据 URL、Header 等分发
- 功能丰富
- 例：Nginx、HAProxy
```

## 2. 负载均衡算法 ⭐⭐⭐

### 2.1 轮询（Round Robin）

```
原理：依次分发请求

请求1 → 服务器1
请求2 → 服务器2
请求3 → 服务器3
请求4 → 服务器1
...

优点：简单公平
缺点：不考虑服务器性能差异

Nginx 配置：
upstream backend {
    server 192.168.1.1;
    server 192.168.1.2;
    server 192.168.1.3;
}
```

### 2.2 加权轮询（Weighted Round Robin）

```
原理：按权重分发请求

服务器1（权重3）：处理 3/6 = 50% 请求
服务器2（权重2）：处理 2/6 = 33% 请求
服务器3（权重1）：处理 1/6 = 17% 请求

适用：服务器性能不同

Nginx 配置：
upstream backend {
    server 192.168.1.1 weight=3;
    server 192.168.1.2 weight=2;
    server 192.168.1.3 weight=1;
}
```

### 2.3 IP 哈希（IP Hash）

```
原理：根据客户端 IP 计算哈希值，固定分发到某台服务器

hash(client_ip) % server_count = server_index

优点：同一客户端总是访问同一服务器
缺点：服务器增减时，大量请求会重新分配

Nginx 配置：
upstream backend {
    ip_hash;
    server 192.168.1.1;
    server 192.168.1.2;
    server 192.168.1.3;
}
```

### 2.4 最少连接（Least Connections）

```
原理：将请求分发到当前连接数最少的服务器

服务器1：10 个连接
服务器2：5 个连接  ← 新请求分发到这里
服务器3：8 个连接

适用：请求处理时间差异大的场景

Nginx 配置：
upstream backend {
    least_conn;
    server 192.168.1.1;
    server 192.168.1.2;
    server 192.168.1.3;
}
```

### 2.5 一致性哈希

```
原理：将服务器和请求都映射到哈希环上

优点：
- 服务器增减时，只影响相邻节点
- 减少缓存失效

适用：分布式缓存

Nginx 配置（需要模块）：
upstream backend {
    hash $request_uri consistent;
    server 192.168.1.1;
    server 192.168.1.2;
    server 192.168.1.3;
}
```

## 3. 反向代理 ⭐⭐⭐

### 3.1 正向代理 vs 反向代理

```
正向代理：
- 代理客户端
- 客户端知道代理的存在
- 例：VPN、科学上网

客户端 → 正向代理 → 目标服务器

反向代理：
- 代理服务器
- 客户端不知道真实服务器
- 例：Nginx、CDN

客户端 → 反向代理 → 后端服务器
```

### 3.2 反向代理的作用

```
1. 负载均衡
   - 分发请求到多台服务器

2. 安全防护
   - 隐藏真实服务器 IP
   - 防御 DDoS 攻击

3. SSL 终止
   - 在代理层处理 HTTPS
   - 后端用 HTTP

4. 缓存加速
   - 缓存静态资源
   - 减轻后端压力

5. 压缩
   - 压缩响应数据
   - 减少传输量
```

## 4. Nginx 配置详解 ⭐⭐⭐

### 4.1 基本反向代理

```nginx
# /etc/nginx/nginx.conf

http {
    # 定义后端服务器组
    upstream backend {
        server 127.0.0.1:8080;
        server 127.0.0.1:8081;
        server 127.0.0.1:8082;
    }

    server {
        listen 80;
        server_name example.com;

        location / {
            proxy_pass http://backend;
            
            # 传递真实客户端信息
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;
        }
    }
}
```

### 4.2 按路径分发

```nginx
server {
    listen 80;
    server_name example.com;

    # API 请求转发到后端
    location /api/ {
        proxy_pass http://api_backend/;
    }

    # 静态文件直接返回
    location /static/ {
        root /var/www;
        expires 30d;
    }

    # 前端页面
    location / {
        root /var/www/html;
        try_files $uri $uri/ /index.html;
    }
}

upstream api_backend {
    server 127.0.0.1:8080;
    server 127.0.0.1:8081;
}
```

### 4.3 HTTPS 配置

```nginx
server {
    listen 443 ssl http2;
    server_name example.com;

    # SSL 证书
    ssl_certificate /etc/nginx/ssl/cert.pem;
    ssl_certificate_key /etc/nginx/ssl/key.pem;

    # SSL 优化
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256;
    ssl_prefer_server_ciphers on;
    ssl_session_cache shared:SSL:10m;
    ssl_session_timeout 10m;

    location / {
        proxy_pass http://backend;
    }
}

# HTTP 重定向到 HTTPS
server {
    listen 80;
    server_name example.com;
    return 301 https://$server_name$request_uri;
}
```

### 4.4 WebSocket 代理

```nginx
location /ws/ {
    proxy_pass http://websocket_backend;
    
    # WebSocket 必需的头
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
    
    # 超时设置
    proxy_read_timeout 86400s;
    proxy_send_timeout 86400s;
}
```

## 5. 健康检查 ⭐⭐⭐

### 5.1 被动健康检查

```nginx
upstream backend {
    server 192.168.1.1 max_fails=3 fail_timeout=30s;
    server 192.168.1.2 max_fails=3 fail_timeout=30s;
    server 192.168.1.3 backup;  # 备用服务器
}

# max_fails=3：失败 3 次后标记为不可用
# fail_timeout=30s：30 秒后重新尝试
# backup：只有其他服务器都不可用时才使用
```

### 5.2 主动健康检查（Nginx Plus）

```nginx
# Nginx Plus 商业版功能
upstream backend {
    zone backend 64k;
    server 192.168.1.1;
    server 192.168.1.2;
    
    health_check interval=5s fails=3 passes=2;
}

# 开源替代：使用 nginx_upstream_check_module
upstream backend {
    server 192.168.1.1;
    server 192.168.1.2;
    
    check interval=3000 rise=2 fall=3 timeout=1000 type=http;
    check_http_send "GET /health HTTP/1.0\r\n\r\n";
    check_http_expect_alive http_2xx http_3xx;
}
```

### 5.3 Go 健康检查接口

```go
// 健康检查接口
func HealthHandler(c *gin.Context) {
    // 检查数据库连接
    if err := db.Ping(); err != nil {
        c.JSON(503, gin.H{"status": "unhealthy", "error": "database"})
        return
    }
    
    // 检查 Redis 连接
    if err := rdb.Ping(c).Err(); err != nil {
        c.JSON(503, gin.H{"status": "unhealthy", "error": "redis"})
        return
    }
    
    c.JSON(200, gin.H{"status": "healthy"})
}

// 注册路由
r.GET("/health", HealthHandler)
```

## 6. 会话保持 ⭐⭐

### 6.1 为什么需要会话保持

```
问题：
- 用户登录后，Session 存在服务器 A
- 下次请求被分发到服务器 B
- 服务器 B 没有 Session，用户需要重新登录

解决方案：
1. 会话保持（Sticky Session）
2. Session 共享（Redis）
3. 无状态设计（JWT）
```

### 6.2 IP Hash 会话保持

```nginx
upstream backend {
    ip_hash;
    server 192.168.1.1;
    server 192.168.1.2;
}

# 同一 IP 总是访问同一服务器
# 缺点：NAT 后多用户共享 IP
```

### 6.3 Cookie 会话保持

```nginx
upstream backend {
    server 192.168.1.1;
    server 192.168.1.2;
    
    sticky cookie srv_id expires=1h domain=.example.com path=/;
}

# 第一次访问时设置 Cookie
# 后续请求根据 Cookie 分发
```

### 6.4 Session 共享（推荐）

```go
// 使用 Redis 存储 Session
import "github.com/gin-contrib/sessions"
import "github.com/gin-contrib/sessions/redis"

func main() {
    r := gin.Default()
    
    // Redis Session 存储
    store, _ := redis.NewStore(10, "tcp", "localhost:6379", "", []byte("secret"))
    r.Use(sessions.Sessions("session", store))
    
    r.GET("/login", func(c *gin.Context) {
        session := sessions.Default(c)
        session.Set("user_id", 123)
        session.Save()
    })
    
    r.GET("/profile", func(c *gin.Context) {
        session := sessions.Default(c)
        userID := session.Get("user_id")
        // ...
    })
}
```

## 7. 高可用架构 ⭐⭐

### 7.1 单点故障问题

```
问题：负载均衡器本身是单点

         ┌─────────────┐
         │   客户端     │
         └──────┬──────┘
                │
         ┌──────▼──────┐
         │  负载均衡器   │ ← 单点故障！
         └──────┬──────┘
                │
    ┌───────────┼───────────┐
    │           │           │
┌───▼───┐  ┌───▼───┐  ┌───▼───┐
│服务器1 │  │服务器2 │  │服务器3 │
└───────┘  └───────┘  └───────┘
```

### 7.2 主备模式

```
解决：使用 Keepalived 实现主备

         ┌─────────────┐
         │   客户端     │
         └──────┬──────┘
                │
           VIP: 192.168.1.100
                │
    ┌───────────┴───────────┐
    │                       │
┌───▼───┐              ┌───▼───┐
│ Nginx │ ←─心跳检测─→ │ Nginx │
│ 主节点 │              │ 备节点 │
└───┬───┘              └───┬───┘
    │                       │
    └───────────┬───────────┘
                │
         ┌──────▼──────┐
         │  后端服务器   │
         └─────────────┘

主节点故障时，VIP 自动漂移到备节点
```

### 7.3 Keepalived 配置

```bash
# /etc/keepalived/keepalived.conf

# 主节点配置
vrrp_instance VI_1 {
    state MASTER
    interface eth0
    virtual_router_id 51
    priority 100
    advert_int 1
    
    authentication {
        auth_type PASS
        auth_pass 1234
    }
    
    virtual_ipaddress {
        192.168.1.100
    }
    
    track_script {
        check_nginx
    }
}

vrrp_script check_nginx {
    script "/usr/local/bin/check_nginx.sh"
    interval 2
    weight -20
}

# check_nginx.sh
#!/bin/bash
if ! pidof nginx > /dev/null; then
    exit 1
fi
exit 0
```

## 8. 常见问题排查 ⭐⭐

### 8.1 502 Bad Gateway

```
原因：
1. 后端服务未启动
2. 后端服务崩溃
3. 连接超时

排查：
# 检查后端服务
curl http://127.0.0.1:8080/health

# 检查 Nginx 错误日志
tail -f /var/log/nginx/error.log

# 检查连接数
ss -s
```

### 8.2 504 Gateway Timeout

```
原因：
1. 后端处理太慢
2. 超时设置太短

解决：
# 增加超时时间
location / {
    proxy_pass http://backend;
    proxy_connect_timeout 60s;
    proxy_send_timeout 60s;
    proxy_read_timeout 60s;
}
```

### 8.3 负载不均衡

```
原因：
1. 权重设置不当
2. 长连接导致连接数不均
3. 某些请求处理时间长

解决：
# 使用最少连接算法
upstream backend {
    least_conn;
    server 192.168.1.1;
    server 192.168.1.2;
}

# 限制单个后端的连接数
upstream backend {
    server 192.168.1.1 max_conns=100;
    server 192.168.1.2 max_conns=100;
}
```

## 9. 本章总结

### 必须记住的内容

```
负载均衡算法：
- 轮询：简单公平
- 加权轮询：考虑服务器性能
- IP Hash：会话保持
- 最少连接：动态均衡

Nginx 关键配置：
- upstream：定义后端服务器组
- proxy_pass：转发请求
- proxy_set_header：传递客户端信息

高可用：
- 健康检查：检测后端状态
- 主备模式：避免单点故障
- Session 共享：使用 Redis
```

### 面试常见问题

```
Q: 常见的负载均衡算法有哪些？
A: 轮询、加权轮询、IP Hash、最少连接、一致性哈希。

Q: 如何实现会话保持？
A: IP Hash、Cookie 粘滞、Session 共享（Redis）、JWT 无状态。

Q: 如何保证负载均衡器的高可用？
A: 使用 Keepalived 实现主备切换，VIP 漂移。

Q: 502 和 504 错误的区别？
A: 502 是后端服务不可用，504 是后端响应超时。
```
