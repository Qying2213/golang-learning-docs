## 1. 网络性能指标 ⭐⭐⭐

### 1.1 关键指标

```
1. 延迟（Latency）
   - 请求发出到收到响应的时间
   - 单位：毫秒（ms）
   - 目标：< 100ms（优秀），< 300ms（可接受）

2. 带宽（Bandwidth）
   - 单位时间内传输的数据量
   - 单位：Mbps、Gbps
   - 影响大文件传输速度

3. 吞吐量（Throughput）
   - 单位时间内处理的请求数
   - 单位：QPS（每秒查询数）、TPS（每秒事务数）
   - 衡量服务器处理能力

4. 连接数
   - 并发连接数量
   - 影响服务器资源消耗
```

### 1.2 延迟的组成

```
总延迟 = DNS 解析 + TCP 握手 + TLS 握手 + 请求传输 + 服务器处理 + 响应传输

DNS 解析：    ~50ms（首次）
TCP 握手：    1 RTT（~50ms）
TLS 握手：    2 RTT（~100ms）
请求传输：    取决于数据大小和带宽
服务器处理：  取决于业务逻辑
响应传输：    取决于数据大小和带宽

RTT = Round Trip Time（往返时间）
```

### 1.3 测量延迟

```bash
# 使用 curl 测量各阶段时间
curl -w "\
DNS 解析:     %{time_namelookup}s\n\
TCP 连接:     %{time_connect}s\n\
TLS 握手:     %{time_appconnect}s\n\
首字节时间:   %{time_starttransfer}s\n\
总时间:       %{time_total}s\n" \
-o /dev/null -s https://api.example.com
```

## 2. 连接池 ⭐⭐⭐

### 2.1 为什么需要连接池

```
问题：每次请求都建立新连接

HTTP 请求流程：
1. DNS 解析        ~50ms
2. TCP 三次握手    ~50ms
3. TLS 握手        ~100ms
4. 发送请求        ~10ms
5. 等待响应        ~50ms
6. 接收响应        ~10ms
7. TCP 四次挥手    ~50ms

总计：~320ms，其中连接建立占 200ms！

解决：连接池
- 复用已建立的连接
- 避免重复握手
- 大幅降低延迟
```

### 2.2 HTTP 连接池（Go）

```go
package main

import (
    "io"
    "net/http"
    "time"
)

func main() {
    // 创建自定义 Transport（连接池）
    transport := &http.Transport{
        MaxIdleConns:        100,              // 最大空闲连接数
        MaxIdleConnsPerHost: 10,               // 每个主机最大空闲连接
        MaxConnsPerHost:     100,              // 每个主机最大连接数
        IdleConnTimeout:     90 * time.Second, // 空闲连接超时
    }

    // 创建 Client
    client := &http.Client{
        Transport: transport,
        Timeout:   10 * time.Second,
    }

    // 复用 client 发送请求
    for i := 0; i < 100; i++ {
        resp, err := client.Get("https://api.example.com/users")
        if err != nil {
            continue
        }
        // 必须读取并关闭 Body，否则连接无法复用
        io.Copy(io.Discard, resp.Body)
        resp.Body.Close()
    }
}
```

### 2.3 数据库连接池

```go
package main

import (
    "database/sql"
    _ "github.com/go-sql-driver/mysql"
)

func main() {
    db, _ := sql.Open("mysql", "user:password@tcp(localhost:3306)/dbname")

    // 连接池配置
    db.SetMaxOpenConns(100)    // 最大打开连接数
    db.SetMaxIdleConns(10)     // 最大空闲连接数
    db.SetConnMaxLifetime(time.Hour)  // 连接最大生命周期
    db.SetConnMaxIdleTime(10 * time.Minute)  // 空闲连接最大生命周期

    // 复用 db 执行查询
    // db.Query(), db.Exec() 会自动从连接池获取连接
}
```

### 2.4 Redis 连接池

```go
package main

import (
    "github.com/go-redis/redis/v8"

func main() {
    rdb := redis.NewClient(&redis.Options{

        IdleTimeout:  5 * time.Minute,  // 空闲超时
```

## 3. Keep-Alive ⭐⭐⭐

### 3.1 HTTP Keep-Alive

```
HTTP/1.0：默认短连接，每次请求都建立新连接
HTTP/1.1：默认长连接（Keep-Alive）

Keep-Alive 的作用：
- 复用 TCP 连接
- 减少握手次数
- 降低延迟

请求头：
Connection: keep-alive
Keep-Alive: timeout=5, max=100

timeout：连接保持时间（秒）
max：最多复用次数
```

### 3.2 TCP Keep-Alive

```
TCP Keep-Alive：检测连接是否存活

作用：
- 检测死连接
- 防止 NAT 超时断开
- 释放无效连接资源

Linux 配置：
# 空闲多久后发送探测包（默认 7200 秒）
net.ipv4.tcp_keepalive_time = 600

# 探测包发送间隔（默认 75 秒）
net.ipv4.tcp_keepalive_intvl = 60

# 探测失败次数后断开（默认 9 次）
net.ipv4.tcp_keepalive_probes = 3
```

### 3.3 Go 中设置 Keep-Alive

```go
// HTTP Keep-Alive（默认开启）
transport := &http.Transport{
    DisableKeepAlives: false,  // 不禁用 Keep-Alive
    IdleConnTimeout:   90 * time.Second,
}

// TCP Keep-Alive
conn, _ := net.Dial("tcp", "example.com:80")
tcpConn := conn.(*net.TCPConn)
tcpConn.SetKeepAlive(true)
tcpConn.SetKeepAlivePeriod(60 * time.Second)
```

## 4. HTTP/2 优化 ⭐⭐⭐

### 4.1 HTTP/2 特性

```
HTTP/1.1 的问题：
1. 队头阻塞：一个请求阻塞，后面的都等待
2. 头部冗余：每次请求都发送完整头部
3. 并发限制：浏览器限制每个域名 6 个连接

HTTP/2 的解决方案：
1. 多路复用：一个连接并发多个请求
2. 头部压缩：HPACK 算法压缩头部
3. 服务器推送：主动推送资源
4. 二进制分帧：更高效的解析
```

### 4.2 多路复用

```
HTTP/1.1：
连接1: 请求A → 响应A
连接2: 请求B → 响应B
连接3: 请求C → 响应C
（需要多个连接）

HTTP/2：
连接1: 请求A ─┐
       请求B ─┼─→ 响应A、响应B、响应C（交错传输）
       请求C ─┘
（一个连接搞定）
```

### 4.3 Go HTTP/2 服务器

```go
package main

import (
    "net/http"
    "golang.org/x/net/http2"
)

func main() {
    mux := http.NewServeMux()
    mux.HandleFunc("/", func(w http.ResponseWriter, r *http.Request) {
        w.Write([]byte("Hello HTTP/2"))
    })

    server := &http.Server{
        Addr:    ":443",
        Handler: mux,
    }

    // 启用 HTTP/2
    http2.ConfigureServer(server, &http2.Server{})

    // 需要 TLS
    server.ListenAndServeTLS("cert.pem", "key.pem")
}
```

### 4.4 HTTP/2 客户端

```go
package main

import (
    "crypto/tls"
    "net/http"
    "golang.org/x/net/http2"
)

func main() {
    // 创建支持 HTTP/2 的 Transport
    transport := &http2.Transport{
        TLSClientConfig: &tls.Config{
            InsecureSkipVerify: true,
        },
    }

    client := &http.Client{
        Transport: transport,
    }

    resp, _ := client.Get("https://example.com")
    // resp.Proto 会显示 "HTTP/2.0"
}
```

## 5. HTTP/3 和 QUIC ⭐⭐

### 5.1 HTTP/3 特性

```
HTTP/2 的问题：
- 基于 TCP，仍有队头阻塞（TCP 层面）
- TCP 握手 + TLS 握手延迟

HTTP/3 的解决方案：
- 基于 QUIC（UDP）
- 0-RTT 连接建立
- 独立的流，无队头阻塞
- 连接迁移（切换网络不断开）

QUIC = Quick UDP Internet Connections

Go 支持情况：
- 标准库暂无原生支持
- 可使用第三方库：github.com/quic-go/quic-go
- Go 1.24+ 实验性支持 QUIC
```

### 5.2 HTTP/3 优势

```
连接建立时间对比：

TCP + TLS 1.2：
1. TCP 握手：1 RTT
2. TLS 握手：2 RTT
总计：3 RTT

TCP + TLS 1.3：
1. TCP 握手：1 RTT
2. TLS 握手：1 RTT
总计：2 RTT

QUIC（首次）：
1. QUIC + TLS：1 RTT
总计：1 RTT

QUIC（恢复）：
0-RTT（直接发送数据）
```

## 6. CDN 加速 ⭐⭐⭐

### 6.1 什么是 CDN

```
CDN = Content Delivery Network
      内容分发网络

原理：
- 在全球部署边缘节点
- 用户访问最近的节点
- 缓存静态资源

好处：
1. 降低延迟（就近访问）
2. 减轻源站压力
3. 提高可用性
4. 防御 DDoS 攻击
```

### 6.2 CDN 工作流程

```
没有 CDN：
用户（北京）→ 源站（美国）
延迟：200ms

有 CDN：
用户（北京）→ CDN 节点（北京）→ 源站（美国）
                    ↓
              缓存命中，直接返回
延迟：20ms
```

### 6.3 CDN 缓存策略

```
缓存控制头：
Cache-Control: max-age=86400    # 缓存 1 天
Cache-Control: no-cache         # 每次验证
Cache-Control: no-store         # 不缓存

ETag 验证：
ETag: "abc123"
If-None-Match: "abc123"
→ 304 Not Modified（未修改，使用缓存）

Last-Modified 验证：
Last-Modified: Wed, 01 Jan 2025 00:00:00 GMT
If-Modified-Since: Wed, 01 Jan 2025 00:00:00 GMT
→ 304 Not Modified
```

## 7. 其他优化技巧 ⭐⭐

### 7.1 压缩传输

```go
// 服务端启用 Gzip
import "github.com/gin-contrib/gzip"

r := gin.Default()
r.Use(gzip.Gzip(gzip.DefaultCompression))

// 客户端请求压缩
req.Header.Set("Accept-Encoding", "gzip, deflate")
```

### 7.2 减少请求数

```
1. 合并文件
   - CSS/JS 打包
   - 雪碧图（CSS Sprites）

2. 内联小资源
   - Base64 编码小图片
   - 内联关键 CSS

3. 使用缓存
   - 浏览器缓存
   - CDN 缓存
   - 应用层缓存
```

### 7.3 DNS 预解析

```html
<!-- 预解析 DNS -->
<link rel="dns-prefetch" href="//api.example.com" />

<!-- 预连接（DNS + TCP + TLS）-->
<link rel="preconnect" href="https://api.example.com" />
```

### 7.4 TCP 优化

```bash
# Linux TCP 优化参数

# 增大 TCP 缓冲区
net.core.rmem_max = 16777216
net.core.wmem_max = 16777216
net.ipv4.tcp_rmem = 4096 87380 16777216
net.ipv4.tcp_wmem = 4096 65536 16777216

# 开启 TCP Fast Open
net.ipv4.tcp_fastopen = 3

# 开启 TCP 窗口缩放
net.ipv4.tcp_window_scaling = 1

# 复用 TIME_WAIT 连接（仅用于客户端发起的连接）
# 注意：需谨慎使用，确保理解其影响
net.ipv4.tcp_tw_reuse = 1
```

## 9. 本章总结

### 必须记住的内容

```
性能指标：
- 延迟：请求到响应的时间
- 吞吐量：每秒处理的请求数
- 连接数：并发连接数量

优化手段：
1. 连接池：复用连接，减少握手
2. Keep-Alive：保持长连接
3. HTTP/2：多路复用，头部压缩
4. CDN：就近访问，缓存加速
5. 压缩：减少传输数据量
6. 缓存：减少重复请求
```

### 面试常见问题

```
Q: 如何优化 HTTP 请求性能？
A: 使用连接池、Keep-Alive、HTTP/2、CDN、压缩、缓存等。

Q: HTTP/2 相比 HTTP/1.1 有什么优势？
A: 多路复用、头部压缩、服务器推送、二进制分帧。

Q: 什么是连接池？为什么需要？
A: 预先建立并复用连接，避免每次请求都建立新连接，减少握手开销。

Q: CDN 是如何加速的？
A: 在全球部署边缘节点，用户访问最近的节点，缓存静态资源。
```
