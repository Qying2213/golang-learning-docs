## 1. 为什么需要 HTTPS？⭐⭐⭐

### 1.1 HTTP 的问题

```
HTTP 是明文传输！

你在咖啡店连 WiFi，登录银行网站：
用户名：zhangsan
密码：123456

如果用 HTTP：
┌─────────┐                    ┌─────────┐
│  你的电脑 │ ──── 明文 ────→  │  服务器  │
└─────────┘                    └─────────┘
      │
      │  黑客在中间偷听
      ↓
┌─────────┐
│  黑客    │ 看到了：用户名=zhangsan, 密码=123456
└─────────┘

三大风险：
1. 窃听 - 数据被偷看
2. 篡改 - 数据被修改
3. 冒充 - 假装是银行网站
```

### 1.2 HTTPS 如何解决

```
HTTPS = HTTP + SSL/TLS（加密层）

┌─────────┐                    ┌─────────┐
│  你的电脑 │ ──── 密文 ────→  │  服务器  │
└─────────┘                    └─────────┘
      │
      │  黑客在中间偷听
      ↓
┌─────────┐
│  黑客    │ 看到的是：x7#kL9@mN... （看不懂）
└─────────┘

HTTPS 提供：
1. 加密 - 数据变成密文，偷看也看不懂
2. 完整性 - 数据被篡改能发现
3. 身份验证 - 确认对方是真的银行网站
```

### 1.3 一眼区分 HTTP 和 HTTPS

```
浏览器地址栏：

HTTP：
http://www.example.com
⚠️ 不安全

HTTPS：
https://www.example.com
🔒 安全（有小锁图标）

端口号：
HTTP  → 80
HTTPS → 443
```

## 2. 加密基础知识 ⭐⭐⭐

### 2.1 什么是加密？

```
加密 = 把明文变成密文
解密 = 把密文变回明文

明文：Hello World
  ↓ 加密（用密钥）
密文：x7#kL9@mN...
  ↓ 解密（用密钥）
明文：Hello World

密钥 = 加密解密的"钥匙"
没有密钥，就无法解密
```

### 2.2 对称加密

```
对称加密 = 加密和解密用同一把钥匙

┌─────────┐                         ┌─────────┐
│  小明    │                         │  小红    │
│         │                         │         │
│ 密钥：ABC │                         │ 密钥：ABC │
└─────────┘                         └─────────┘
     │                                   │
     │  1. 用 ABC 加密                    │
     │  "你好" → "x7k"                   │
     │                                   │
     │  ─────── 发送 "x7k" ───────→      │
     │                                   │
     │                    2. 用 ABC 解密  │
     │                    "x7k" → "你好"  │

常见算法：
- AES（推荐，最常用）
- DES（已淘汰，不安全）
- 3DES（DES 的加强版）

特点：
✅ 速度快
✅ 适合加密大量数据
❌ 密钥怎么安全传给对方？（密钥分发问题）
```

### 2.3 非对称加密

```
非对称加密 = 用两把不同的钥匙

公钥：可以公开给任何人
私钥：只有自己知道，绝对保密

规则：
- 公钥加密 → 只有私钥能解密
- 私钥加密 → 只有公钥能解密

┌─────────┐                         ┌─────────┐
│  小明    │                         │  小红    │
│         │                         │         │
│ 小红的公钥 │                         │ 公钥+私钥 │
└─────────┘                         └─────────┘
     │                                   │
     │  1. 小红把公钥发给小明              │
     │  ←────────────────────────────    │
     │                                   │
     │  2. 小明用公钥加密                  │
     │  "你好" → "x7k"                   │
     │                                   │
     │  ─────── 发送 "x7k" ───────→      │
     │                                   │
     │                    3. 小红用私钥解密 │
     │                    "x7k" → "你好"  │

即使黑客拿到公钥，也无法解密！
因为只有私钥才能解密公钥加密的内容。

常见算法：
- RSA（最常用）
- ECC（椭圆曲线，更安全更快）

特点：
✅ 解决了密钥分发问题
✅ 可以用于数字签名
❌ 速度慢（比对称加密慢 100-1000 倍）
❌ 不适合加密大量数据
```

### 2.4 对称 vs 非对称

| 对比项   | 对称加密       | 非对称加密             |
| -------- | -------------- | ---------------------- |
| 密钥数量 | 1 把           | 2 把（公钥+私钥）      |
| 速度     | 快             | 慢                     |
| 安全性   | 密钥泄露就完蛋 | 私钥泄露才完蛋         |
| 密钥分发 | 困难           | 简单（公钥可公开）     |
| 适用场景 | 加密大量数据   | 加密少量数据、密钥交换 |

### 2.5 实际应用：混合加密

```
HTTPS 使用混合加密：

1. 用非对称加密交换密钥（安全）
2. 用对称加密传输数据（快速）

┌─────────┐                         ┌─────────┐
│  客户端   │                         │  服务器  │
└─────────┘                         └─────────┘
     │                                   │
     │  1. 服务器发送公钥                  │
     │  ←────────────────────────────    │
     │                                   │
     │  2. 客户端生成随机密钥（会话密钥）    │
     │     用公钥加密会话密钥              │
     │  ─────── 发送加密的会话密钥 ──────→ │
     │                                   │
     │  3. 服务器用私钥解密，得到会话密钥   │
     │                                   │
     │  4. 双方用会话密钥（对称加密）通信   │
     │  ←─────── 加密的数据 ───────→     │

这样既安全又快速！
```

### 📝 练习题 2.1

**问题**：为什么 HTTPS 不全程使用非对称加密？

<details>
<summary>点击查看答案</summary>

**答案**：

因为非对称加密太慢了！

```
性能对比：
- 对称加密（AES）：加密 1GB 数据约 1 秒
- 非对称加密（RSA）：加密 1GB 数据约 100-1000 秒

如果全程用非对称加密：
- 网页加载会非常慢
- 服务器 CPU 会爆炸
- 用户体验极差

所以 HTTPS 的做法是：
1. 用非对称加密交换一个"会话密钥"（只需要加密几十字节）
2. 后续通信用对称加密（快速）

这就是"混合加密"的智慧！
```

</details>

## 3. 数字签名 ⭐⭐⭐

### 3.1 为什么需要数字签名？

```
问题：你怎么知道收到的消息没被篡改？

小明给小红发消息："转账 100 元给小红"

黑客截获后改成："转账 10000 元给黑客"

小红收到后，怎么知道这是不是小明发的？
怎么知道内容有没有被改过？

答案：数字签名！
```

### 3.2 数字签名的原理

```
数字签名 = 用私钥加密消息的摘要

步骤：
1. 对消息计算哈希值（摘要）
2. 用私钥加密哈希值 → 这就是签名
3. 把消息和签名一起发送

验证：
1. 用公钥解密签名，得到哈希值 A
2. 对收到的消息计算哈希值 B
3. 比较 A 和 B，相同则验证通过

┌─────────┐                              ┌─────────┐
│  小明    │                              │  小红    │
│ 私钥签名  │                              │ 公钥验证  │
└─────────┘                              └─────────┘
     │                                        │
     │  消息："转账100元"                      │
     │  ↓                                     │
     │  哈希：abc123                           │
     │  ↓                                     │
     │  签名：xyz789（私钥加密哈希）            │
     │                                        │
     │  ─── 发送消息 + 签名 ───→              │
     │                                        │
     │                    收到消息 + 签名       │
     │                    ↓                   │
     │                    用公钥解密签名 → abc123
     │                    计算消息哈希 → abc123
     │                    ↓                   │
     │                    相同！验证通过 ✓      │
```

### 3.3 哈希函数

```
哈希函数 = 把任意长度的数据变成固定长度的"指纹"

特点：
1. 相同输入 → 相同输出
2. 不同输入 → 不同输出（极大概率）
3. 不可逆（无法从哈希值反推原文）
4. 雪崩效应（改一个字，哈希值完全不同）

例子：
"Hello" → SHA256 → 185f8db32271fe25f561a6fc938b2e264306ec304eda518007d1764826381969
"hello" → SHA256 → 2cf24dba5fb0a30e26e83b2ac5b9e29e1b161e5c1fa7425e73043362938b9824
                   ↑ 只改了一个字母大小写，整个哈希值完全不同！

常见哈希算法：
- MD5（已不安全，不推荐）
- SHA-1（已不安全）
- SHA-256（推荐）
- SHA-512（更安全）
```

### 3.4 Go 中使用哈希和签名

```go
package main

import (
    "crypto/sha256"
    "encoding/hex"
    "fmt"
)

func main() {
    // 计算 SHA256 哈希
    message := "Hello World"
    hash := sha256.Sum256([]byte(message))
    hashHex := hex.EncodeToString(hash[:])

    fmt.Printf("消息: %s\n", message)
    fmt.Printf("哈希: %s\n", hashHex)

    // 输出：
    // 消息: Hello World
    // 哈希: a591a6d40bf420404a011733cfb7b190d62c65bf0bcda32b57b277d9ad9f146e
}
```

## 4. 数字证书 ⭐⭐⭐

### 4.1 为什么需要证书？

```
问题：你怎么知道公钥是真的？

你想访问银行网站，银行发给你公钥。
但是，黑客可以冒充银行，发给你假的公钥！

┌─────────┐                    ┌─────────┐
│   你     │                    │  黑客    │
└─────────┘                    └─────────┘
     │                              │
     │  我要访问银行                  │
     │  ─────────────────────→     │
     │                              │
     │  这是银行的公钥（假的）         │
     │  ←─────────────────────     │
     │                              │
     │  你用假公钥加密数据             │
     │  黑客用自己的私钥解密           │
     │  你的密码被盗了！              │

解决方案：数字证书
- 由权威机构（CA）签发
- 证明"这个公钥确实属于这个网站"
```

### 4.2 证书的内容

```
数字证书包含：

┌─────────────────────────────────────┐
│           数字证书                   │
├─────────────────────────────────────┤
│ 持有者：www.bank.com                 │
│ 公钥：MIIBIjANBgkqhkiG9w0BAQEFAA... │
│ 颁发者：DigiCert CA                  │
│ 有效期：2024-01-01 至 2025-01-01    │
│ 签名算法：SHA256withRSA              │
├─────────────────────────────────────┤
│ CA 的数字签名：x7#kL9@mN...          │
│ （CA 用自己的私钥签名）               │
└─────────────────────────────────────┘

CA（Certificate Authority）= 证书颁发机构
- 权威的第三方
- 验证网站身份后才颁发证书
- 常见 CA：DigiCert、Let's Encrypt、GlobalSign
```

### 4.3 证书验证过程

```
浏览器如何验证证书？

1. 浏览器收到网站的证书
2. 检查证书是否过期
3. 检查颁发者（CA）是否可信
4. 用 CA 的公钥验证签名
5. 检查域名是否匹配

┌─────────┐                    ┌─────────┐
│  浏览器  │                    │  服务器  │
└─────────┘                    └─────────┘
     │                              │
     │  请求 https://bank.com       │
     │  ─────────────────────→     │
     │                              │
     │  返回证书                     │
     │  ←─────────────────────     │
     │                              │
     │  验证证书：                   │
     │  1. 没过期 ✓                  │
     │  2. CA 可信 ✓                 │
     │  3. 签名正确 ✓                │
     │  4. 域名匹配 ✓                │
     │                              │
     │  验证通过，显示 🔒            │
```

### 4.4 证书链

```
证书是分层的：

根证书（Root CA）
    │
    ├── 中间证书（Intermediate CA）
    │       │
    │       └── 网站证书（End Entity）
    │
    └── 中间证书
            │
            └── 网站证书

为什么要分层？
- 根证书非常重要，要离线保存
- 中间证书用于日常签发
- 即使中间证书泄露，也可以吊销

证书透明度（Certificate Transparency, CT）：
- 现代浏览器要求证书必须记录到公开的 CT 日志
- 防止 CA 偷偷签发恶意证书
- 任何人都可以监控 CT 日志发现可疑证书

浏览器内置了根证书：
- 操作系统和浏览器预装了可信的根证书
- 所以浏览器能验证证书链
```

### 📝 练习题 4.1

**问题**：如果你看到浏览器提示"证书不受信任"，可能是什么原因？

<details>
<summary>点击查看答案</summary>

**答案**：

```
可能的原因：

1. 证书过期
   - 证书有有效期
   - 过期后浏览器不信任

2. 自签名证书
   - 不是 CA 签发的
   - 开发环境常见

3. 域名不匹配
   - 证书是给 www.example.com 的
   - 但你访问的是 api.example.com

4. CA 不受信任
   - 颁发证书的 CA 不在浏览器信任列表中

5. 证书被吊销
   - CA 发现问题后吊销了证书

6. 中间人攻击
   - 有人在中间替换了证书
   - 这是最危险的情况！

处理建议：
- 开发环境：可以忽略（但要知道风险）
- 生产环境：绝对不能忽略！
```

</details>

## 5. SSL/TLS 握手过程 ⭐⭐⭐

### 5.1 TLS 握手概述

```
TLS 握手 = 建立安全连接的过程

目的：
1. 验证服务器身份（通过证书）
2. 协商加密算法
3. 交换密钥

握手完成后，双方就可以用对称加密安全通信了。
```

### 5.2 TLS 1.2 握手过程（简化版）

```
┌─────────┐                              ┌─────────┐
│  客户端  │                              │  服务器  │
└─────────┘                              └─────────┘
     │                                        │
     │  1. Client Hello                       │
     │  - 支持的 TLS 版本                      │
     │  - 支持的加密算法列表                   │
     │  - 客户端随机数                         │
     │  ─────────────────────────────────→   │
     │                                        │
     │  2. Server Hello                       │
     │  - 选择的 TLS 版本                      │
     │  - 选择的加密算法                       │
     │  - 服务器随机数                         │
     │  - 服务器证书                          │
     │  ←─────────────────────────────────   │
     │                                        │
     │  3. 客户端验证证书                      │
     │     生成预主密钥（Pre-Master Secret）   │
     │     用服务器公钥加密预主密钥            │
     │  ─────────────────────────────────→   │
     │                                        │
     │  4. 双方根据三个随机数生成会话密钥       │
     │     - 客户端随机数                      │
     │     - 服务器随机数                      │
     │     - 预主密钥                         │
     │                                        │
     │  5. Finished（用会话密钥加密）          │
     │  ←────────────────────────────────→   │
     │                                        │
     │  握手完成！开始加密通信                  │
     │  ←═══════ 加密的数据 ═══════→         │
```

### 5.3 为什么需要三个随机数？

```
会话密钥 = f(客户端随机数, 服务器随机数, 预主密钥)

为什么不只用预主密钥？

1. 增加随机性
   - 每次连接的随机数都不同
   - 即使预主密钥相同，会话密钥也不同

2. 防止重放攻击
   - 黑客录制了你的加密通信
   - 下次重放这些数据
   - 但随机数不同，密钥不同，重放无效

3. 前向安全性
   - 即使私钥泄露
   - 之前的通信也无法解密
   - 因为每次的会话密钥都不同
```

### 5.4 TLS 1.3 的改进

```
TLS 1.3（2018年发布）更快更安全：

TLS 1.2：需要 2 个 RTT（往返时间）
TLS 1.3：只需要 1 个 RTT

┌─────────┐                    ┌─────────┐
│  客户端  │                    │  服务器  │
└─────────┘                    └─────────┘
     │                              │
     │  Client Hello + Key Share    │
     │  ─────────────────────→     │
     │                              │
     │  Server Hello + Key Share    │
     │  + 证书 + Finished           │
     │  ←─────────────────────     │
     │                              │
     │  Finished                    │
     │  ─────────────────────→     │
     │                              │
     │  开始加密通信                 │

TLS 1.3 的改进：
1. 握手更快（1-RTT，甚至 0-RTT）
2. 移除了不安全的加密算法
3. 加密了更多握手内容
4. 默认前向安全
```

## 6. Go 中使用 HTTPS ⭐⭐⭐

### 6.1 创建 HTTPS 服务器

```go
package main

import (
    "fmt"
    "log"
    "net/http"
)

func main() {
    http.HandleFunc("/", func(w http.ResponseWriter, r *http.Request) {
        fmt.Fprintf(w, "Hello, HTTPS!")
    })

    // 需要证书文件和私钥文件
    // cert.pem - 证书文件
    // key.pem  - 私钥文件
    log.Println("Starting HTTPS server on :443")
    err := http.ListenAndServeTLS(":443", "cert.pem", "key.pem", nil)
    if err != nil {
        log.Fatal(err)
    }
}
```

### 6.2 生成自签名证书（开发用）

```bash
# 生成私钥和自签名证书
openssl req -x509 -newkey rsa:4096 -keyout key.pem -out cert.pem -days 365 -nodes

# 参数说明：
# -x509      生成自签名证书
# -newkey    生成新的私钥
# rsa:4096   RSA 算法，4096 位
# -keyout    私钥输出文件
# -out       证书输出文件
# -days 365  有效期 365 天
# -nodes     私钥不加密
```

### 6.3 Gin 框架使用 HTTPS

```go
package main

import (
    "github.com/gin-gonic/gin"
)

func main() {
    r := gin.Default()

    r.GET("/", func(c *gin.Context) {
        c.JSON(200, gin.H{
            "message": "Hello, HTTPS!",
        })
    })

    // 使用 HTTPS
    r.RunTLS(":443", "cert.pem", "key.pem")
}
```

### 6.4 HTTP 自动跳转 HTTPS

```go
package main

import (
    "net/http"
    "github.com/gin-gonic/gin"
)

func main() {
    // HTTP 服务器，重定向到 HTTPS
    go func() {
        http.ListenAndServe(":80", http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
            target := "https://" + r.Host + r.URL.Path
            http.Redirect(w, r, target, http.StatusMovedPermanently)
        }))
    }()

    // HTTPS 服务器
    r := gin.Default()
    r.GET("/", func(c *gin.Context) {
        c.JSON(200, gin.H{"message": "Secure!"})
    })
    r.RunTLS(":443", "cert.pem", "key.pem")
}
```

## 7. 本章总结

### 必须记住的内容

```
HTTPS = HTTP + TLS（加密）

加密方式：
- 对称加密：一把钥匙，快，用于数据传输
- 非对称加密：两把钥匙，慢，用于密钥交换
- 混合加密：HTTPS 的实际做法

数字签名：
- 用私钥签名
- 用公钥验证
- 保证数据完整性和身份认证

数字证书：
- CA 签发
- 证明公钥的身份
- 浏览器验证证书链

TLS 握手：
- 验证身份
- 协商算法
- 交换密钥
```

### 面试常见问题

```
Q: HTTPS 比 HTTP 慢吗？
A: 握手阶段会慢一点（多 1-2 个 RTT），但数据传输速度差不多。
   现代硬件加密很快，TLS 1.3 握手也更快了。

Q: HTTPS 能防止所有攻击吗？
A: 不能。HTTPS 防的是传输层的攻击（窃听、篡改）。
   不能防 XSS、SQL 注入、CSRF 等应用层攻击。

Q: 自签名证书和 CA 证书有什么区别？
A: 自签名证书：自己签发，浏览器不信任，适合开发测试。
   CA 证书：权威机构签发，浏览器信任，适合生产环境。
```
