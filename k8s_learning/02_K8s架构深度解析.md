# 02. Kubernetes 架构深度解析

> 本章目标：理解 K8s 的核心组件及其协作机制

---

## 1. 为什么要学架构？

很多人跳过架构直接学命令，结果遇到问题就懵了：

- "为什么 Pod 一直 Pending？"
- "为什么 Service 访问不通？"
- "kubectl 到底在和谁通信？"

**理解架构 = 拥有排错能力**。花 30 分钟学架构，能省你未来无数小时的调试时间。

## 3. Control Plane 组件详解

Control Plane 是集群的"大脑"，通常运行在专门的 Master 节点上。

### 3.1 API Server (`kube-apiserver`)

**角色**：集群的唯一入口，所有人都必须通过它

**职责**：

- 接收所有 REST 请求（来自 kubectl、Dashboard、其他组件）
- 认证和授权（你有没有权限做这件事？）
- 数据校验（你提交的 YAML 格式对不对？）
- 将数据写入 etcd

**类比**：公司的前台 + 安检

**实战验证**：

```bash
# kubectl 其实就是在调用 API Server
kubectl get pods -v=9
# 加上 -v=9 可以看到底层 HTTP 请求
```

### 3.2 Etcd

**角色**：集群的数据库，存储所有状态

**职责**：

- 保存集群的完整状态（有哪些 Pod、Service、ConfigMap...）
- 提供强一致性保证
- 支持 Watch 机制（状态变化时主动通知）

**类比**：公司的档案室，所有合同、员工信息都在这里

**注意**：只有 API Server 可以直接读写 etcd，其他组件都要通过 API Server

**实战验证**：

```bash
# 进入 Minikube 查看 etcd

minikube ssh
docker ps | grep etcd
# 你会看到 etcd 容器正在运行
exit
```

### 3.3 Scheduler (`kube-scheduler`)

**角色**：调度决策者

**职责**：

- 监听新创建的、尚未分配节点的 Pod
- 根据资源需求、亲和性规则选择最合适的节点
- 将决策写回 API Server（Pod.spec.nodeName）

**类比**：人力资源经理，决定新员工去哪个部门

**调度考虑因素**：

- 节点剩余 CPU/内存够不够？
- 节点是否满足 nodeSelector 要求？
- 是否有污点（Taint）阻止调度？

**实战验证**：

```bash
# 创建一个 Pod 后查看调度结果
kubectl describe pod <pod-name> | grep -A5 "Events"
# 你会看到 "Successfully assigned xxx to minikube"
```

### 3.4 Controller Manager (`kube-controller-manager`)

**角色**：状态维持者

**职责**：

- 运行多个控制器（Controller）
- 每个控制器负责一种资源的状态维护
- 核心逻辑：**期望状态 vs 实际状态，不一致就修复**

**包含的主要控制器**：

| 控制器                    | 职责                        |
| :------------------------ | :-------------------------- |
| Node Controller           | 监控节点健康状态            |
| Replication Controller    | 维持 Pod 副本数量           |
| Endpoints Controller      | 维护 Service 的 Endpoints   |
| ServiceAccount Controller | 为新 Namespace 创建默认账户 |

**类比**：自动化巡检系统，发现问题自动修复

## 4. Worker Node 组件详解

Worker Node 是实际运行 Pod 的机器。

### 4.1 Kubelet

**角色**：节点上的代理人

**职责**：

- 接收 API Server 的 Pod 调度指令
- 调用 Container Runtime 创建/销毁容器
- 定期上报节点和 Pod 状态
- 执行健康检查（Liveness/Readiness Probe）

**类比**：工地上的工段长/监工

**实战验证**：

```bash
# 查看 kubelet 状态
minikube ssh
systemctl status kubelet
exit
```

### 4.2 Kube-proxy

**角色**：网络代理

**职责**：

- 维护节点上的网络规则（iptables / IPVS）
- 实现 Service 的负载均衡
- 将访问 Service IP 的流量转发到正确的 Pod

**类比**：交通指挥员/路由器

**工作模式**：

- **iptables 模式**：通过 Linux iptables 规则转发（默认）
- **IPVS 模式**：性能更好，适合大规模集群

### 4.3 Container Runtime

**角色**：容器运行时

**职责**：

- 实际执行容器的生命周期管理
- 拉取镜像、创建容器、启动/停止容器

**常见的运行时**：

- Docker（传统，Minikube 默认）
- containerd（更轻量，生产主流）
- CRI-O（Red Hat 主推）

## 5. 组件协作流程

让我们通过一个具体例子来看组件如何协作：

### 场景：创建一个 Nginx Pod

```bash
kubectl run nginx --image=nginx
```

**完整流程**：

```
1. kubectl 发送请求到 API Server
   "我要创建一个 nginx Pod"

2. API Server 验证请求
   - 认证：你是谁？
   - 授权：你有权限吗？
   - 校验：请求格式对吗？

3. API Server 将 Pod 信息写入 etcd
   "Pod nginx 创建了，但还没有分配节点"

4. Scheduler 监听到新 Pod
   "有新 Pod 需要调度！"
   - 检查各节点资源
   - 选择 minikube 节点
   - 写回 API Server：Pod.spec.nodeName = "minikube"

5. Kubelet (on minikube) 监听到分配给自己的 Pod
   "有个 Pod 要我来创建！"
   - 调用 Docker 拉取 nginx 镜像
   - 创建并启动容器
   - 上报状态：Pod Running

6. 用户查询状态
   kubectl get pods
   → Pod nginx: Running
```

## 6. 在 Minikube 中验证架构

### 6.1 查看所有系统组件

```bash
kubectl get pods -n kube-system
```

**你会看到**：

```
NAME                               READY   STATUS
coredns-xxx                        1/1     Running   # DNS 服务
etcd-minikube                      1/1     Running   # Etcd
kube-apiserver-minikube            1/1     Running   # API Server
kube-controller-manager-minikube   1/1     Running   # Controller Manager
kube-proxy-xxx                     1/1     Running   # Kube-proxy
kube-scheduler-minikube            1/1     Running   # Scheduler
storage-provisioner                1/1     Running   # 存储供应者
```

### 6.2 SSH 进去看容器

```bash
minikube ssh
docker ps --format "table {{.Names}}\t{{.Status}}"
```

你会发现：**K8s 的组件本身也是以容器运行的！**

### 6.3 查看组件日志

```bash
# 查看 API Server 日志
kubectl logs -n kube-system kube-apiserver-minikube

# 查看 Scheduler 日志
kubectl logs -n kube-system kube-scheduler-minikube
```

## 7. 声明式 vs 命令式

这是 K8s 最核心的设计理念。

### 命令式（Imperative）

```bash
# 一步一步告诉系统怎么做
kubectl run nginx --image=nginx
kubectl scale deployment nginx --replicas=3
kubectl set image deployment/nginx nginx=nginx:1.19
```

### 声明式（Declarative）

```yaml
# 告诉系统你想要的最终状态
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx
spec:
  replicas: 3 # 我要 3 个副本
  template:
    spec:
      containers:
        - name: nginx
          image: nginx:1.19 # 用这个镜像
```

**K8s 的核心工作**：让实际状态不断逼近期望状态

```
期望状态: 3 个 nginx Pod
实际状态: 2 个 nginx Pod
   ↓
Controller 发现差异
   ↓
创建 1 个新 Pod
   ↓
实际状态: 3 个 nginx Pod ✓
```

## 8. 本章重点回顾

| 组件               | 一句话总结                         |
| :----------------- | :--------------------------------- |
| API Server         | 集群的唯一入口，所有人都要和它说话 |
| Etcd               | 存储一切状态的数据库               |
| Scheduler          | 决定 Pod 跑在哪个节点              |
| Controller Manager | 保证实际状态符合期望状态           |
| Kubelet            | 节点上的代理人，负责管理容器       |
| Kube-proxy         | 负责 Service 的网络转发            |
